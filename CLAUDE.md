# リポジトリ運用ルール

本リポジトリ(system-document-template)を編集・運用する際のルールを定義する。個々のドキュメント作成ルールではなく、**リポジトリ全体の構成と、テンプレート/サンプル間の同期の規約**を扱う。

> 各プロジェクト(docs/・template/)内の文書作成ルール(役割・ID規則・参照方向・状態・レビュー・禁止事項)は、それぞれの CLAUDE.md を正本とする(本ファイルでは扱わない)。

## 1. リポジトリの構成

| ディレクトリ | 役割 |
|---|---|
| template/ | 新規プロジェクトへコピーして使うテンプレート集(文書種別ごとの雛形・運用ルール・採番ルール・プロンプト)。**テンプレートの正本** |
| docs/ | template/ に従って記入した完成サンプル(会議室予約プロダクト MeetRoom)。**記入例・実例** |
| .github/ | CI(markdown-link-check) |

## 2. README / CLAUDE の書き分け

README.md・CLAUDE.md は階層ごとに責務を分ける。

| 階層 | README.md | CLAUDE.md |
|---|---|---|
| ルート(本リポジトリ) | リポジトリの概要 | リポジトリの運用ルール(本ファイル) |
| docs/ | サンプルプロジェクト(MeetRoom)の概要 | プロジェクトの運用ルール ※ |
| template/ | プロジェクト概要の雛形(docs/README.md のテンプレート) | プロジェクトの運用ルール ※ |

※ **docs/CLAUDE.md と template/CLAUDE.md は常に同一内容に保つ**(プロジェクト運用ルールの正本)。

## 3. テンプレートとサンプルの同期ルール

| No | ルール |
|---|---|
| 1 | 文書の**フォーマット・記法・セクション構成・運用ルール**を変える場合は、まず template/ 側(正本)を変更し、その後 docs/ のサンプルへ反映する |
| 2 | 文書種別テンプレート(テンプレート_XXX.md)・一覧/レビューの共通雛形・採番ルール・プロンプトの正本は template/ に置く。docs/ はサンプル記入済みのため、文書種別テンプレート・共通雛形を保持しない(参照は template/ を指す) |
| 3 | docs/CLAUDE.md と template/CLAUDE.md のどちらかを変更したら、必ずもう一方も同一内容に更新する |
| 4 | template/ 一式を新規プロジェクトの docs/ としてコピーして使う前提を崩さない(template/ 内の参照・手順が単体で成立するように保つ) |
| 5 | 設計カテゴリの追加・移動・再採番では、template/ と docs/ のディレクトリ、README、CLAUDE、index、テンプレート、プロンプト、トレーサビリティ、本文中パスを同一作業で更新し、旧パスを残さない |
| 6 | **docs/ の全文書種別に対応するテンプレートを template/ に用意する(テンプレート化を徹底する)**。新しい文書種別・セクションを docs/ に持たせる前に、まず template/ にテンプレート(または既存テンプレートへのセクション追加)を作り、docs/ へ反映する。文書種別↔テンプレートの対応を突合し、欠落(テンプレートのない文書種別)を残さない。対象は概要設計のCMP/IFC、09_運用設計のOPS、要件定義層トレース(RQ→FR・FR→NFR)、データ所有/状態遷移、API共通設計、ER図、共通カラム、命名規則を含む |

## 4. CI

[.github/workflows/markdown-link-check.yml](.github/workflows/markdown-link-check.yml) が push / Pull Request 時にマークダウンのリンク切れを検査する。マークダウンリンク(`[表示テキスト](パス)` 形式)のリンク切れを残さない。
