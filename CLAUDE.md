# リポジトリ運用ルール

本リポジトリは、要求定義から詳細設計までを1冊で管理する統合設計書テンプレートと、社員名簿管理システムの記入例を提供する。

## 1. リポジトリの構成

| パス | 役割 |
|---|---|
| `テンプレート_統合設計書.md` | 0〜13章を1ファイルにまとめた**テンプレートの正本** |
| `テンプレート/` | テンプレートを設計種別ごとに分割した版 |
| `社員名簿管理システム_統合設計書.md` | テンプレートに従って作成した**記入例の統合版** |
| `社員名簿管理システム/` | 記入例を設計種別ごとに分割した版。画面モックアップを含む |
| `.github/` | Markdownリンク検査などのCI設定 |

廃止済みの英字ディレクトリや中間ディレクトリは再作成しない。

## 2. 正本と同期ルール

| No | ルール |
|---:|---|
| 1 | フォーマット、セクション構成、定義ルールを変更する場合は、最初に`テンプレート_統合設計書.md`を更新する |
| 2 | 統合版の変更を同じ作業で`テンプレート/`の対応章へ反映し、0〜13章の内容を一致させる |
| 3 | 記入例へ適用する変更は、`社員名簿管理システム_統合設計書.md`と`社員名簿管理システム/`の対応章を同時に更新する |
| 4 | 統合版と分割版の差は、ナビゲーション行と配置階層に応じた相対リンクだけに限定する |
| 5 | 画面モックアップは`社員名簿管理システム/mockups/`でHTML原本とPNGを同じ画面IDで対にして管理する |
| 6 | 章、画面、テーブル、API、JOB、モジュール、SQLの追加・削除・再採番では、README、一覧、トレーサビリティ、レビュー項目、相対リンクを同じ作業で更新する |
| 7 | 機能要件では対象範囲内の全F-IDをUCへ対応させ、利用者操作、クライアント共通操作、外部イベント、定期・非同期起動を含む全UCを一覧・個別定義・シーケンス要否・トレーサビリティで同期する |
| 8 | §3.1を全シーケンスのactor/participant、正式名称、接続順、SCR/D1/API/JOB/M対応の構成正本とし、全図の宣言集合と完全一致させる。§4〜§8は§3.1・§3.1.1のID・名称・依存方向を変更せず詳細化する |

## 3. アーキテクチャ規約

- バックエンドはCloudflare Workers Paid、永続化データベースはCloudflare D1（SQLite互換）を標準とする。
- D1 Binding `env.DB`、Prepared Statement、SQLを扱えるのはM-006 D1データアクセスだけとする。
- APIはログイン以外をM-002、ログインをM-003へ委譲し、JOBはM-002へ委譲する。API・JOBからM-006やD1を直接呼ばない。
- API、JOB、モジュールの各処理は、番号と名称が一致する「処理フロー」と「処理詳細」で記載する。
- M-006が実行するSQLはクエリ設計にSQL-ID付きで定義し、SQLite/D1構文と順序付きplaceholderを使用する。

## 4. READMEの責務

| 階層 | READMEの役割 |
|---|---|
| ルート | リポジトリの目的、メイン文書、利用方法、技術前提 |
| `テンプレート/` | 分割テンプレートの章一覧と利用方法 |
| `社員名簿管理システム/` | 記入例の章一覧、システム前提、統合版・テンプレートへの導線 |

## 5. CI

[.github/workflows/markdown-link-check.yml](.github/workflows/markdown-link-check.yml)で、ルートREADME、両統合版、両分割版のMarkdownリンク切れを検査する。移動・改名時は旧パスへの参照を残さない。
