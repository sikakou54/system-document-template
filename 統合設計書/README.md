# 統合設計書（詳細設計統合様式）

要求定義から詳細設計（シーケンス・画面・API・DB・モジュール・JOB・クエリ）までを **1つの文書に統合** して記述する様式。
章構成は 0. はじめに 〜 13. 設計の落とし込み関係とし、状態パターン、代替/例外フロー、画面イベント、API・モジュール・JOBの処理フローと処理詳細、DB制約、モジュール実行SQLまでを定義する。

`docs/` と `template/` の粒度別・多ファイル方式（正本1箇所＋ID参照）とは **別系統の成果物** として提供する。小〜中規模のシステムで、1冊にまとまった設計書を短期に作りたい場合に使う。

## 目次構成

| No | セクション | 主な内容 |
|---|---|---|
| 0 | はじめに | 本書の目的、前提、基本設計と詳細設計の境界 |
| 1 | 要求定義 | 背景・課題、目的(OBJ)、利用者と役割、業務要求(BR)、非機能要求(NFR)、制約、対象範囲 |
| 2 | 機能要件 | 機能一覧(F)、主要ユースケース(UC：状態パターン SP-x・基本/代替(ALT)/例外(EXC)フロー)、機能要求詳細、シーケンス要否 |
| 3 | シーケンス設計 | 論理構成要素、主要 UC の連携シーケンス（Mermaid） |
| 4 | 画面設計 | 画面一覧(SCR)、遷移、HTML/PNGモックアップ、全画面の項目・イベント・状態・バリデーション・メッセージ |
| 5 | API設計 | 方針、API一覧、全APIの入出力・処理フロー・処理詳細・エラー・バリデーション |
| 6 | データベース設計 | 論理/物理モデル、全テーブル定義、制約/インデックス、トランザクション、共通カラム |
| 7 | モジュール設計 | 論理モジュール構成(M)、責務、公開IF、処理フロー・処理詳細、TX・排他、利用SQL |
| 8 | JOB設計 | JOB一覧、起動条件、入出力、処理フロー・処理詳細、再実行・監視・障害対応 |
| 9 | クエリ設計 | M-006だけが実行するSQLの入出力・SQL文・結果・性能・排他定義 |
| 10 | 要求・設計トレーサビリティ | 要求→機能→シーケンス→画面/API/モジュール/JOB/SQL/DB の対応マトリクス |
| 11 | 設計レビュー用チェックリスト | 全成果物とDBアクセス境界のレビュー観点 |
| 12 | 詳細設計への引継ぎ事項 | 実装技術、運用値、試験具体化など本書外で確定する事項 |
| 13 | 設計の落とし込み関係 | 要求から各設計、およびAPI/JOB→モジュール→SQL→DBの流れ図 |

## 使い方

1. [テンプレート_統合設計書.md](テンプレート_統合設計書.md) をコピーする。
2. 各セクション直上の定義ルールコメントを読み、上から順に埋める（0→13）。
3. 記入の具体例は [社員名簿管理システム_統合設計書.md](社員名簿管理システム_統合設計書.md) を参照する。

> 注: D1 Binding・DB接続・Prepared Statement・SQL実行はデータアクセスモジュールだけに許可する。画面、API、JOB、その他モジュールからの直接アクセスは禁止する。`env.DB`はデータアクセスモジュールの実装だけへ渡す。定期JOBはCron TriggerからCloudflare Queuesへ投入し、Workers Paidの`queue` consumerで40件ずつ継続する。
