## 3.6 定期・非同期JOB

```mermaid
sequenceDiagram
    autonumber
    participant CRON as Cron Trigger
    participant SCHED as scheduled JOB
    participant Q as Cloudflare Queue
    participant JOB as Queue consumer JOB
    participant APP as M-002 アプリケーション
    participant DATA as M-006 データアクセス
    participant DB as Cloudflare D1

    CRON->>SCHED: 予定時刻で起動する
    SCHED->>SCHED: 業務日と初回継続情報を確定する
    SCHED->>Q: 初回messageを投入する
    Note over SCHED,Q: scheduledは初回投入だけで終了する
    Q->>JOB: 1件のmessageを配送する
    JOB->>APP: 最大40件・実行予算900でチャンク処理を要求する
    APP->>DATA: 対象取得・反映を論理公開IFで依頼する
    DATA->>DB: 対象を参照・更新する（概念エンティティ名）
    DB-->>DATA: 実行結果
    DATA-->>APP: 論理結果とStatement実測数
    APP-->>JOB: 件数・次cursor・継続有無・実測数
    alt 続きあり
        JOB->>Q: 同じchainの次messageを投入する
        Q-->>JOB: 投入受付
        JOB-->>Q: 現messageをackする
    else 完了
        JOB-->>Q: 現messageをackする
    else 一時障害・実行基盤障害
        JOB--xQ: 失敗を返して再配信またはDLQへ委ねる
    end
```

**連携定義**

| 観点 | 固定設計 |
|---|---|
| 初回message | `cursor=null`、`chainRunId`、`chunkNo=1`、`businessDate` |
| 継続message | M-002が返したcursor、同じchainRunId/businessDate、連続するchunkNo |
| consumer設定 | `max_batch_size=1`、`max_concurrency=1`、`max_retries=3`、DLQ必須 |
| チャンク上限 | 最大40件、D1 Statement内部予算900、Paid hard limit 1,000 |
| 成功順序 | チャンク確定 → 続きがあれば次message投入確定 → 現message ack |
| 再試行 | allowlistだけ即時再試行（exponential backoff + full jitter）。結果不明は状態/version再読込み。過負荷、timeout、CPU、memoryは即時再試行せずQueue redelivery |
| データアクセス | JOB → M-002 → M-006 → Cloudflare D1。JOB・scheduled・QueueからD1への直接経路なし |

D1原子実行境界は、当該チャンク内の対象単位ごとに§3.x/§7/§9のTX-IDへ追跡する。Queue message全体を1つのD1 transactionとはみなさず、再配信は `chainRunId + chunkNo`と対象version/冪等キーで無害化する。
