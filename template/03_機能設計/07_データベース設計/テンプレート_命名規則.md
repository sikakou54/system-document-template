【1. 基本方針】
定義内容: テーブル・カラム・主キー・外部キー・インデックス・一意インデックスなど、物理オブジェクトの命名規則を対象別に一括定義する。
定義する条件: 全プロジェクトで必須。DB設計に着手する前に確定する。
項目説明:
- 対象: 命名規則を定める物理オブジェクトの種別。
- 規則: その対象の命名パターン。接頭辞・接尾辞・大文字/小文字・複数形/単数形・区切り文字・可変部({テーブル}・{カラム}等)を明示する。
定義ルール:
- テーブル物理名の接頭辞(種別接頭辞)を採用する場合は、接頭辞→テーブル種別(マスタ/トランザクション/履歴/一時 等)の対応を規則列に明記する。論理名にも対応する種別接尾辞を定める。
- カラム・PK・FK・インデックス・一意インデックスの規則は、可変部を {テーブル}・{カラム}・{参照先} などのプレースホルダで示し、大文字/小文字と区切り文字を統一する。
- インデックス名は接頭辞(例 IDX_/UX_)+基幹名で衝突しないよう、テーブル基幹名(接頭辞を除いた部分)と対象カラム/内容を含める規則にする。
- 予約語・長さ上限・大文字小文字の扱いは採用DB基盤の制約に合わせて調整する(調整前提の汎用値を置く)。
-->
# 1. 基本方針

| 対象 | 規則 |
|---|---|
| テーブル(物理名) | {種別接頭辞} + {複数形/単数形}{区切り(スネークケース等)}{大文字/小文字}。接頭辞例: {M_=マスタ / T_=トランザクション / H_=履歴 / TP_=一時テーブル} |
| テーブル(論理名) | 「{基幹名} + 種別」。種別接尾辞例: {マスタ / トランザクション / 履歴 / 一時テーブル} |
| カラム | {区切り(スネークケース等)}{大文字/小文字} |
| PK | {主キーカラム名(例 ID)} |
| FK | {参照先単数形}{区切り}{ID接尾辞(例 _ID)} |
| Index | {接頭辞(例 IDX_)}{テーブル基幹名}{区切り}{内容}({大文字/小文字}。テーブルは接頭辞を除いた基幹名) |
| UQ(一意インデックス) | {接頭辞(例 UX_)}{テーブル基幹名}{区切り}{カラム}({大文字/小文字}) |

<!--
【2. 命名例】
定義内容: §1 の各規則を具体的な命名に適用した例を、対象・例・補足で示す。
定義する条件: 必須。§1 の各対象について最低1件ずつ例を示す。
項目説明:
- 対象: §1 の対象種別に対応する具体例の分類(テーブル(マスタ)・テーブル(トランザクション)・カラム・PK・FK・Index・UQ 等)。
- 例: §1 の規則を適用した具体的な物理名(必要に応じて論理名も併記)。
- 補足: 接頭辞・接尾辞・変換(複数形→単数形など)や、対応する TBL-ID・IDX-ID など、命名の根拠・対応関係を記す。
定義ルール:
- 例は §1 の規則と矛盾させない。矛盾が生じたら §1 を正として例を直す。
- FK の例では、参照先テーブル(複数形)→単数形への変換過程を補足に示す。
- Index・UQ の例では、対応する TBL-XXX/IDX-x や 対象カラムを補足に示す(逆参照の網羅列挙はしない。理解に必要な対応のみ)。
-->
# 2. 命名例

| 対象 | 例 | 補足 |
|---|---|---|
| テーブル(マスタ) | 物理名 {M_XXXXS} / 論理名 {XXXXマスタ} | 接頭辞 {M_}、接尾辞 {マスタ} |
| テーブル(トランザクション) | 物理名 {T_YYYYS} / 論理名 {YYYYトランザクション} | 接頭辞 {T_}、接尾辞 {トランザクション} |
| カラム | {START_AT} | {区切り(スネークケース等)}{大文字/小文字} |
| PK | {ID} | 全テーブル共通(各TBL §2 カラム定義) |
| FK | {XXXX_ID} | {M_XXXXS}(複数形)→ {XXXX}(単数形)+ {_ID} |
| Index | {IDX_YYYYS_XXXX_TIME} | {TBL-YYY/IDX-1} に対応 |
| UQ | {UX_XXXXS_CODE} | {XXXXS.CODE} の一意インデックス |

<!--
【3. 型方針】
定義内容: 採用DB基盤(RDBMS製品・SQL方言)と、その型体系に沿った型注記の方針を、種別・型・補足で定義する。
定義する条件: 必須。採用DB基盤を1つ明記したうえで、扱う代表的な種別(整数・文字列・真偽・日時・主キー・区分値 等)の型注記を定める。
項目説明:
- 種別: 論理的なデータ種別(整数・文字列・真偽・日時・主キー・区分値など)。
- 型: その種別に対して記載する物理型注記(採用DB基盤の型体系に合わせる)。
- 補足: 用途・値の解釈・付随制約(真偽の 0/1 規約、日時の保存形式、区分値の CHECK と値の正本など)。
定義ルール:
- 冒頭にDB基盤(製品・SQL方言)を1文で明記し、以降の型注記はその方言に合わせる。
- 型で文字数上限や符号などを表せない基盤では、上限・符号の担保方法(制約・アプリケーション検証)を補足に明記する。
- 真偽・日時・区分値など、基盤機能に依存する既定値・関数式・保存形式は基盤に合わせて固定し、混在を禁じる旨を箇条書きで補足する。
- 区分値は型注記に加え、値の正本(10_共通定義設計/共通コード定義.md 等)への参照を補足に示す(値そのものを本ファイルに再記載しない)。
- 採用基盤で使用しない型・記法(他方言の型など)があれば、使用しないものを箇条書きで明示する。
-->
# 3. 型方針({DB基盤(製品・SQL方言)})

DB基盤は {DB基盤(製品・SQL方言。例: RDBMS製品名)}。{その型体系}に合わせ、以下の型注記で記載する。

| 種別 | 型 | 補足 |
|---|---|---|
| 整数 | {整数型} | {用途例(数量・金額・時間など)} |
| 文字列 | {文字列型} | {文字数上限の担保方法(型で表さず制約・アプリケーション検証で担保する等)} |
| 真偽 | {真偽の表現型} | {0=偽 / 1=真 など、真偽の表現規約} |
| 日時 | {日時型} | {保存形式・タイムゾーン(例: RFC3339 UTC固定、表示は現地TZ)。例値を添える} |
| 主キー | {主キーの型注記} | 各TBL §2 カラム定義に PK=◯ で記載 |
| 区分値 | {区分値の型注記(例 整数型 + CHECK)} | 値は 10_共通定義設計/共通コード定義.md で定義 |

- {採用DB基盤で使用しない型・記法を列挙(例: 他方言固有の型・記法は使用しない)}。
- {接頭辞・大文字小文字・PK・FK など §1 の命名規則を本DB基盤前提でも維持する旨}。
- {既定値・関数式の方針(日時の既定値に用いる現在時刻の返し方など、基盤方言に合わせた固定ルール)}。
