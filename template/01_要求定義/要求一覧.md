# 1. 概要

対象システムに対して利用者・管理者・組織が求めることを、要求(RQ)として定義する。
本ディレクトリは 00_企画/プロット.md で定義した企画・構想を受け、要求定義として利用者・管理者・組織が求めることを整理する。

<!--
【2. 定義ルール】
定義内容: 要求定義一覧を作成・更新する際のルールを定義する。
定義する条件: 全プロジェクトで必須。原則として固定文言を使用し、プロジェクト固有の補足が必要な場合のみ追記する。
項目説明:

- 粒度: 要求をどの単位に分解するか。
- 網羅範囲: 要求として洗い出す対象。
- 上流文書: 要求定義の元になる企画・構想文書。
- 書かない内容: 要求定義に含めない後続工程の情報。
- 正本: 要求一覧と個別RQファイルの位置づけ。
定義ルール:
- 後続工程ID、設計詳細、物理名、処理方式は書かない。

# 2. 定義ルール

| 項目 | 内容 |
|---|---|
| 目的 | 利用者・管理者・組織が何を求めているかを、実現手段に寄せずに定義する |
| 粒度 | 受入条件・業務ルール・ユースケース上の期待結果を、単独で充足可否を判断できる詳細要求に分解する |
| 網羅範囲 | 業務課題、利用者の期待結果、利用シナリオ、品質・運用上の期待から読み取れる要求を対象とする |
| 上流文書 | 00_企画/プロット.md |
| 書かない内容 | 画面、API、DB、モジュール、外部サービス、物理名、処理方式などの設計詳細 |
| 正本 | 要求一覧は本ファイルを正本とする。個別RQファイルは、補足説明が必要な要求に限り作成する |

| 優先度 | 意味 |
|---|---|
| M (Must) | 必須。これが無いとサービスとして成り立たない |
| S (Should) | 推奨。必須ではないが重要な要求 |
| C (Could) | できれば。あればよいレベルの要求 |
| W (Won't) | 今回は見送り |
