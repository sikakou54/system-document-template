# マルチエージェント編成

本フェーズは次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・ルール・既存文書を読み、作成対象の洗い出し・分類・ID採番(一覧の最大値+1)・網羅性ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・要件と設計の非混在・実装判断できる粒度・トレース鎖の観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(企画・要求)との網羅性・業務妥当性・可読性・優先度の妥当性・専門用語の平易さの観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本フェーズの前提文書・テンプレートを読み、作業計画を立てる
2. リサーチAgent: 上流を分析し、作成対象・分類・ID採番・網羅性ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(品質系要求の抽出・分類)
- 要求一覧から品質・運用に関わる要求(性能・可用性・セキュリティ・データ保護・監査・運用性など)を抽出し、品質特性で分類する
- 各 NFR の対象範囲(全機能／システム全体／特定機能名)を整理する。特定機能固有か機能横断かを判定する
- 各 NFR がどの FR に適用されるかを整理し、機能非機能トレーサビリティ.md 更新用の FR→適用NFR 案を作る(機能横断の NFR は全 FR に適用、監査など重要操作系は該当 FR のみ)
- 達成基準の候補(測定可能な数値・単位・測定条件)を洗い出し、確定できないものは「要確認」＋確認先を明示する
- NFR-ID の採番(index.md の最大値+1、欠番の再利用禁止)を確認する
- 品質系 RQ がすべて NFR(または CFR-000 のシステム全体要件)に対応づくかの網羅性ギャップ表を作る

## 作業Agent(非機能要件作成)
- docs/02_要件定義/02_非機能要件/index.md に先に行を追加してから、テンプレート_非機能要件.md をコピーして NFR-XXX_名称.md を作成する
- §1 基本情報(要件ID・分類・対象範囲・概要)と §2 品質要求・達成基準・検証方法を、定義ルールコメントに従って埋める
- 達成基準は測定可能な形(数値・単位・測定条件・対象範囲)で書く。「速いこと」等の定性表現にしない。未確定は「要確認」とし確認先を明記する
- 機能非機能トレーサビリティ.md の各 FR 行に適用NFRを転記し、要求要件トレーサビリティ.md の品質系 RQ 行に実現要件として NFR を追記する
- 実装方式・画面・API・DB などの設計(how)は書かない。機能設計(SCR/API/MOD/TBL/JOB)を参照・言及しない

## レビューAgent(エンジニア視点)
- NFR-ID が重複・欠番の再利用なく、対象範囲が明確か
- 達成基準が測定可能(数値・単位・測定条件)で、定性表現になっていないか
- 検証方法が達成基準を確認できる現実的な手段か
- 機能設計への参照・言及や逆参照がないか(§5)。要件定義と機能設計が混在していないか
- 機能非機能トレーサビリティ.md の FR→適用NFR が妥当か(機能横断/特定機能の適用範囲が正しいか)

## レビューAgent(非エンジニア視点)
- 品質系の要求(RQ)が NFR に漏れなく落ちているか(要求要件トレーサビリティ.md で品質系 RQ が NFR を持つか)
- 達成基準の水準が利用者・組織の期待や業務実態と合っているか(過剰・過少でないか)
- 対象範囲(全機能/特定機能)が業務観点で妥当か
- 分類・概要が専門用語なしで業務担当者に理解できるか
- 「要確認」の残存項目に確認先が明記され、確定を急いでいないか

# このフェーズの必須ルール

- NFR はトレース参照先・機能設計(SCR/API/MOD/TBL/JOB)を参照・言及しない(§5)。FR との適用関係は 機能非機能トレーサビリティ.md で管理する
- NFR は 02_要件定義/02_非機能要件/ に個別文書として定義する(機能横断・特定機能固有のいずれも同ディレクトリ。§8-9)
- 達成基準は測定可能な形で書き、「未定」「要確認」を根拠なく確定させない(§8-3・§10-6)
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md・機能非機能トレーサビリティ.md・要求要件トレーサビリティ.md を更新する(§8-5・§10-7)
- ユーザーが記法・セクション構成を指摘したら、テンプレート_非機能要件.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- 品質系の RQ がすべて NFR(または CFR-000 のシステム全体要件)に対応づき、両レビュー観点が全てパスしている
- 達成基準がすべて測定可能で、「未定」「要確認」がゼロ(残る場合は確認先を明記)
- index.md・機能非機能トレーサビリティ.md・要求要件トレーサビリティ.md の行と各 NFR 文書の基本情報が一致している
