# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(UC→画面の分析)
- トレース元 UC の基本/代替/例外フロー・入力/出力データから、必要な画面と画面上の項目(入力・表示・ボタン)を洗い出す
- 各画面が呼び出す API(初期表示・イベント)を API 一覧から特定する(未作成の API は「予定」と注記し、API設計側との整合を促す)
- 選択肢が区分値の項目について、参照する DEF-XXX/CODE-XXX を特定する
- 画面遷移(遷移先 SCR・トリガ)と、表示するメッセージ(完了・確認・警告・エラー・情報)を洗い出す
- SCR-ID・ローカルID(ITM-XX/EVT-01/MSG-XXX)の採番方針を確認する(MSG は全文書横断で最大値+1)

## 作業Agent(画面設計作成 + モック作成)
- docs/03_機能設計/02_画面設計/index.md に先に行を追加してから、テンプレート_画面.md をコピーして SCR-XXX_画面名.md を作成し、§1〜§9 を全て埋める
- 画面レイアウトは HTML で作成し PNG に書き出して mockups/SCR-XXX.png として保存し、![キャプション](mockups/SCR-XXX.png) で貼る(HTML→PNG 化には html-to-png スキルを利用してよい)
- 表示文言は §9 メッセージ一覧に MSG-XXX としてインライン定義し、本文各所(初期表示・イベント・入力チェック・表示制御)からは MSG-XXX で参照する(文言を重複記載しない)
- 呼び出す API は API-ID、区分値は DEF-XXX/CODE-XXX で参照し、値・表示名・API内部仕様を再記載しない
- 入力チェックはクライアント側チェックのみ記載する(サーバ側バリデーションは API 文書に定義)
- traceability_uc_design.md(UC→SCR)・traceability_scr_api_db.md(SCR→API)へ転記する

## レビューAgent(エンジニア視点)
- SCR-ID・ローカルID(ITM-XX/EVT-01)が画面内一意で、MSG-XXX が全文書横断で重複していないか
- トレース元 UC(FR-XXX/UC-01)が実在し完全修飾形か。呼び出す API-ID・遷移先 SCR-ID・DEF-XXX が実在するか(§5)
- 逆参照(この画面を呼ぶ側の一覧など)がないか。API 内部処理・DB 構造を書いていないか(要件・他設計の混在なし)
- 文言が §9 のみで定義され、本文では MSG-XXX 参照になっているか(重複記載禁止)
- 各項目・イベントが実装判断できる粒度(種別・必須・初期値・発火条件・成否時挙動)か

## レビューAgent(非エンジニア視点)
- トレース元 UC の基本/代替/例外フローが、画面の項目・イベント・メッセージとして漏れなく表現されているか(網羅性)
- モック画像のレイアウトが利用者にとって自然で、操作手順が直感的に理解できるか
- メッセージ文言(完了・確認・警告・エラー)が利用者に分かりやすい表現か(専門用語・内部用語が出ていないか)
- 利用可能ロール・表示制御が業務上の権限と一致しているか
- 画面名・項目名が業務用語と一致しているか

# この設計の必須ルール

- 画面(SCR)のトレース元はシステムユースケースの完全修飾ID(FR-XXX/UC-01)。API は呼び出し(依存参照)、他 SCR は遷移先として参照する(§5)
- 表示文言は §9 メッセージ一覧に MSG-XXX として定義し、本文では MSG-XXX で参照する(文言の重複記載禁止)。区分値は DEF-XXX/CODE-XXX 参照(値・表示名を再記載しない)
- サーバ側バリデーション・API 内部処理・DB 構造は書かない(それぞれ API 設計・モジュール設計・DB 設計が正本)
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md・traceability_uc_design.md・traceability_scr_api_db.md を更新する(§8-5・§10-7)
- ユーザーが記法・セクション構成を指摘したら、テンプレート_画面.md の定義ルールコメントへ反映する(§8-10)
