# マルチエージェント編成

本フェーズは次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・ルール・既存文書を読み、作成対象の洗い出し・分類・ID採番(一覧の最大値+1)・網羅性ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・要件と設計の非混在・実装判断できる粒度・トレース鎖の観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(企画・要求)との網羅性・業務妥当性・可読性・優先度の妥当性・専門用語の平易さの観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本フェーズの前提文書・テンプレートを読み、作業計画を立てる
2. リサーチAgent: 上流を分析し、作成対象・分類・ID採番・網羅性ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(要求→機能要件のグルーピング)
- 要求一覧の各 RQ を機能単位にグルーピングし、機能要件(FR)候補と、複数機能に横断する共通機能要件(CFR)候補に振り分ける(認証・認可・入力検証・エラー表示・一覧検索など横断的なものは CFR)
- 各 FR/CFR が実現する RQ-ID を対応づけ、traceability_rq_fr.md 更新用の RQ→実現要件 対応案を作る
- 各 FR について、システムユースケース(UC)候補を洗い出す。データ・利用者のパターンで UC が分かれる場合は UC 単位に分解する。システム自動実行(リマインド等)も主アクター=システムの UC として洗い出す(「なし」にしない)
- FR-ID・CFR-ID の採番(各一覧の最大値+1、欠番の再利用禁止。廃止IDは欠番として扱う)を確認する
- 全 RQ が少なくとも1つの FR/CFR/NFR に対応づくかの網羅性ギャップ表を作る(NFR で実現する品質系 RQ は「非機能要件作成」フェーズ側で対応づくため、その旨を注記する)

## 作業Agent(機能要件作成)
- docs/02_要件定義/01_機能要件/index.md に先に行を追加してから、テンプレート_機能要件.md をコピーして FR-XXX_機能名.md を作成する
- 各 FR は §1 基本情報・§2 業務ルール(制約事項)・§3 受入条件・§4 システムユースケース を、定義ルールコメントに従って全て埋める
- システムユースケースは UC 単位(### 4.1【UC-01】…)で、事前条件・事後条件・入力データ・出力データ・基本フロー・代替フロー(ALT-x)・例外フロー(EXC-x)をそれぞれ太字ラベル＋テーブルで定義する。UC-ID は FR ごとのローカル連番
- 共通横断要件は テンプレート_共通機能要件.md をコピーして CFR-XXX_名称.md を作成し、CFR-000_システム共通要件.md の CFR一覧へ行を追加する
- traceability_rq_fr.md の該当 RQ 行に実現要件(FR/CFR)を転記する(NFR は非機能要件作成フェーズで追記)
- 設計(how)は書かない: 画面項目・API パラメータ・型・入力形式・DB 項目・処理ロジックは記載しない

## レビューAgent(エンジニア視点)
- FR-ID・CFR-ID・UC-ID(ローカル連番)が重複・欠番の再利用なく、参照先 RQ-ID が実在するか
- FR がトレース参照先(上流ID)を本文に書いていないか。列挙型逆参照(自分を実現する SCR/API 一覧など)がないか(§5・§10-2)
- 要件定義と機能設計の内容が混在していないか(画面・API・DB・処理ロジックが書かれていないか)
- 受入条件がテストの合否判定に使える粒度か。UC の基本/代替/例外フローが実装判断できる粒度か
- 例外フローが1行1条件で、業務レベルのエラーメッセージを持つか。ALT/EXC の分岐Stepが基本フローのStep番号に対応するか
- システム自動実行が「なし」で済まされず、主アクター=システムの UC として定義されているか

## レビューAgent(非エンジニア視点)
- 対象 RQ が FR/CFR に漏れなく落ちているか(traceability_rq_fr.md で全 RQ が実現要件を持つか)
- §2 業務ルール(制約事項)が業務観点で妥当か、矛盾や現場と合わない制約がないか
- §3 受入条件が「要求が満たされたこと」の確認になっているか(要求と受入条件が対応しているか)
- UC のシナリオ(基本・代替・例外)が実際の業務フロー・利用シーンとして自然か
- 対象ロール・用語が業務担当者に理解できるか。機能名・UC名が内容を表しているか

# このフェーズの必須ルール

- FR / CFR / UC は機能設計(SCR/API/MOD/TBL/JOB)を参照・言及しない(§5)。要件→設計のトレースは対応表と設計側トレース元で辿る
- FR は上流ID(RQ)を本文に書かない(自明・自己参照を避ける)。RQ→FR の対応は traceability_rq_fr.md が担う
- 「一覧に行追加 → 文書作成」の順を守り、文書作成と同一作業内で index.md・traceability_rq_fr.md を更新する(§8-5・§10-7)
- 共通横断要件(認証・認可・入力検証・エラー表示・一覧検索など)は各 FR に重複記載せず CFR に集約し、FR からは CFR を適用対象として扱う(適用関係は traceability で管理)
- 不明点は推測で埋めず「未定」「要確認」＋確認先を残し、根拠なく確定しない(§8-3・§10-6)
- ユーザーが記法・セクション構成を指摘したら、該当テンプレート(テンプレート_機能要件.md 等)の定義ルールコメントへ反映する(§8-10)

# 完了条件

- 「未定」「要確認」がゼロ
- index.md・CFR一覧・traceability_rq_fr.md の行と各文書の基本情報が一致している
