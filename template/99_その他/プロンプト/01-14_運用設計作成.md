# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(NFR→運用の分析)
- トレース元となる運用系 NFR を特定して読み、達成基準(数値・単位・測定条件・検証方法)から運用で監視・保証すべき指標・目標値を抽出する(NFRが未確定・未承認なら設計を開始せずギャップとして報告する)
- シーケンス設計トレーサビリティ.md で運用対象となる SEQ/JOB/API/MOD/TBL を特定し、監視・再処理・復旧・切戻しの対象となる連携・処理・データを洗い出す(処理仕様は再記載せず対象IDで参照する)
- 概要設計の該当 CMP/IFC・外部サービス連携方針から、運用の適用境界・実行基盤・外部依存を整理する
- 分類ごとに運用観点を整理する(監視=監視対象・閾値・通知先、バックアップ=取得契機・保持期間・復旧目標、デプロイ=切替方式・切戻し契機、シークレット=保管・ローテーション、JOB再処理=失敗検知・再実行、障害対応=事象・一次対応・エスカレーション)
- 障害・失敗の事象にエラーが対応する場合、返す ERR-XXX を エラーメッセージ一覧.md から特定する(コード・文言・HTTPステータスは再記載しない)
- OPS-ID の採番方針(一覧の最大値+1)を確認する

## 作業Agent(運用設計作成)
- docs/03_機能設計/09_運用設計/index.md に先に行を追加してから、テンプレート_運用設計.md をコピーして OPS-XXX_運用設計名.md を作成し、§1〜§7 を全て埋める
- §1 基本情報に運用名・分類・対象範囲・トレース元 NFR・概要・状態を定義する。分類は index.md の固有列と一致させる
- §2 関連設計は、関連NFR・関連CMP/IFC・関連SEQ/JOB/API・対象データ/ストアを ID参照のみで記載する(参照先の仕様は再記載しない。逆参照の網羅列挙は禁止)
- §3 運用方針・手順は、方針を箇条書き(- )、実施手順・タイミング・担当・使用ツールを表で定義する(§8-2)。実行基盤の製品名・キュー名など運用対象の固有名は必要な範囲で最小限記載してよい(§8-13 例外)
- §4 検知・判定基準は、監視/判定対象・基準/閾値・通知/発報条件・通知先を表で定義する。基準/閾値は トレース元 NFR の達成基準に整合させる(NFR本文の達成基準は再記載せず、運用で用いる判定値を定義する)。検知を伴わない運用は「なし」
- §5 対応・復旧手順は、事象・一次対応・復旧手順・エスカレーションを表で定義する。事象にエラーが対応する場合は ERR-XXX を併記し、復旧手順は対象JOB/APIの処理内容を再記載せず対象IDと操作(再実行・切戻し等)で示す。対応を伴わない運用は「なし」
- §6 制限事項・前提は前提条件と制限事項に分けて箇条書きで定義する(§8-8)。§7 状態は §1 と index.md に一致させる
- シーケンス設計トレーサビリティ.md に OPS→NFR/対象設計(SEQ/JOB/API/MOD/TBL)/ERR の対応を追記し、OPS の状態を「未着手」から「作成中」以降へ遷移させる

## レビューAgent(エンジニア視点)
- トレース元 NFR の達成基準(数値・単位・測定条件)が §4 検知・判定基準の閾値・§3 手順・§5 復旧目標へ運用実現として落ちているか(NFR本文の再記載になっていないか)
- 参照する NFR-ID・CMP/IFC・SEQ/JOB/API/MOD/TBL・ERR-XXX が実在するか。参照が §5 参照方向マトリクスの OPS 行(NFR、CMP/IFC、SEQ、Step 8 個別設計)に収まり、上流の RQ/FR 本文を参照していないか
- 運用対象の処理仕様(JOB/APIの内部処理)・TBLの構造を再記載せず、対象IDと操作で参照しているか
- ERR の文言・HTTPステータスを再記載していないか(コード参照か)。物理実装詳細(物理名・スキーマ・具体的コマンド)を書いていないか(§8-13。実行基盤の製品名・キュー名の最小記載は例外)
- §2 関連設計に逆参照の網羅列挙(この運用を利用する側の一覧)が混入していないか(§5・§10-2)

## レビューAgent(非エンジニア視点)
- トレース元 NFR の運用系要求(可用性・復旧目標・保持期間・監査 等)が、監視・検知・復旧・障害対応として漏れなく表現されているか(網羅性)
- 検知基準・通知先・エスカレーションが業務運用として妥当か(重要事象が黙って見逃されないか、通知先が役割で明確か)
- バックアップ保持期間・復旧手順・切戻し契機が事業継続・データ保護の観点で妥当か
- 手順・担当・タイミングが運用担当者にとって実施可能な粒度で書かれているか
- 運用名・分類が運用の内容を表しているか

# この設計の必須ルール

- OPS のトレース元は NFR。NFRの達成基準を運用手順(§3)・検知基準(§4)・復旧手順(§5)へ落とし込む。NFR本文の達成基準は再記載せず、運用で用いる判定値・目標値として定義する
- 参照方向は OPS→NFR、CMP/IFC、SEQ、Step 8 の個別設計(運用対象)に限る。上流の RQ/FR 本文は参照しない(§5)
- 運用対象の処理・データ・エラーは ID参照とし、SEQ/JOB/API/MOD/TBL の仕様・ERR の文言/HTTPステータスを再記載しない(§3.2・§8-4・§10-3)
- 物理実装詳細(物理名・物理スキーマ・具体的なコマンド/スクリプト)を書かない(§8-13)。実行基盤(Cloudflare/Stripe/Resend)の製品名・キュー名など運用対象の固有名は運用上必要な範囲で最小限に留める(§8-13 例外)
- §2 関連設計に逆参照の網羅列挙を書かない。運用の理解に必要な直接の対象IDだけを記載する(§5・§10-2)
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md・シーケンス設計トレーサビリティ.md を更新する(§8-5・§10-7)
- Step 8 の設計として、関連 Step 5責務・Step 6論理契約・Step 7 SEQ を確認する。契約の意味を運用設計だけで変更せず、必要な場合は Step 5/6 へ戻す(§8-26)
- ユーザーが記法・セクション構成を指摘したら、テンプレート_運用設計.md の定義ルールコメントへ反映する(§8-10)

- 「未定」「要確認」がゼロ
