1. 統括Agent: 前提文書を読み、対象 SEQ・関連 UC・予約/確定済み下流IDを確定する
2. 各レビューAgent: 担当観点を独立に確認し、必ずファイル・節・IDの証跡付きで指摘する
3. 統括Agent: 指摘を重複排除し、重要度、修正先の正本、影響ID、修正責任を決定する
4. 修正対象が本レビューの作業範囲なら正本を修正し、範囲外なら変更管理へ起票する。上位要件を SEQ から直接変更しない
5. 各レビューAgent: 修正後を再確認し、指摘を「解消」または根拠付き「非該当」にする
6. 全必須観点がパスし、未解消指摘がゼロになったら統括Agentが完了を宣言する

# 必須レビュー観点

## 1. 構成・ID

- 02_シーケンス設計だけが SEQ ディレクトリとして存在し、README・一覧・テンプレート・プロンプトの参照先が一致する
- シーケンスID、名称、目的、対象範囲、関連FR/CFR/UC、事前/事後条件、構成要素が定義される(要求RQは紐付けない)
- 正常系、代替系、例外系、条件分岐、データ参照/更新、トランザクション境界、補足が定義される
- SEQ-ID が index.md と一致し、関連 FR/CFR/UC・MDL が実在する
- SCR/API/JOB/MOD/TBL/SQL-ID は各 index.md に状態「未着手」以上で実在し、未予約IDや XXX-IDを参照しない

## 2. 上位設計との整合

- 各シーケンスが FR/CFR/UC の振る舞いを根拠とし、要件にない処理を独自追加していない
- UC の事前/事後条件、基本/代替/例外フロー、入力/出力データ、業務ルールが対応する
- 概要設計の構成境界、同期/非同期、外部連携、信頼境界と矛盾しない
- データ参照/更新・状態変更が概念MDL・関連・CRUDで表現できる。不足は要件変更管理へ差し戻される

## 3. シーケンス図・責務・粒度

- 原則として具体的ロールの利用者、UI、API、モジュール、単一の「データベース」(参照する概念MDLを集約)が登場し、必要時だけJOB/外部/非同期基盤を追加する。データモデルごとに参加者を分けず、往路メッセージ末尾に参照データモデル名を列挙する
- 処理は上から下へ時系列で、呼び出し元/先、要求/応答、alt/else、opt、loop、非同期が明確である
- UI に業務ロジック、API に過剰な業務ロジック、DB にアプリケーション責務を持たせていない
- モジュールの責務、トランザクション開始/終了、排他、再試行、タイムアウトが適切な要素へ割り当てられる
- SQL文、インデックス方式、物理カラム、変数操作、ライブラリ固有メソッド、具体的なERR-ID・MSG-ID、個別設計の詳細を記載していない(エラー・メッセージは責務レベルで記述)
- メッセージ名が業務上の動詞形式で、単なる実装メソッド名になっていない

## 4. 下流設計への落とし込み

- UIごとに操作、入力、イベント、API契機、正常/エラー表示、遷移、MSG候補を抽出している
- APIごとに呼び出し元、入出力、認証認可、MOD委譲、結果、トランザクション責務、ERR候補を抽出している
- JOBごとに契機、スケジュール、非同期、冪等、再試行、通知、MOD委譲を抽出している
- MODごとに責務、入出力、業務ルール、分岐、データ操作、他MOD、例外を抽出している
- MDL/TBL/SQLについて参照/更新、状態変更、必要属性・関連・制約、クエリ目的/入出力/対象を抽出している
- 認証・認可・所有者/ロール確認が CFR-002 と整合し、外部IFの契機・送受信・成功/失敗・再試行が抽出される
- シーケンス設計トレーサビリティ.md と下流index/文書が同じID・責務を示す(SEQ→下流設計の対応はSEQ本文に再掲せず対応表で一元管理する)

## 5. トレーサビリティ・正本

- UCシーケンストレーサビリティ.md の FR/UC→SEQ が SEQ 基本情報と一致する
- シーケンス設計トレーサビリティ.md の SEQ→SCR/API/JOB/MOD/MDL/TBL/SQL/ERR/MSG が本文と一致する
- 対応表は正本の転記にとどまり、独自仕様を持たない
- 個別設計の内容を SEQ に重複せず、SEQ の連携順序を個別設計側で別定義していない

# 指摘の記録形式

| 指摘ID | 重要度 | 観点 | 証跡(ファイル・節・ID) | 現状 | 期待する状態 | 修正先の正本 | 影響ID | 状態 |
|---|---|---|---|---|---|---|---|---|
| SEQR-001 | Critical/High/Medium/Low |  |  |  |  |  |  | 未対応/対応中/解消/非該当 |

- 重要度は、要件逸脱・欠落・誤更新リスクを Critical/High、実装不整合を Medium、可読性等を Low とする。
- 「問題なし」は観点ごとの確認件数と証跡を伴う。根拠のない一括パスは禁止する。

# 完了条件

- 全必須観点を対象 SEQ ごとに確認し、確認件数と証跡を記録している
- Critical/High/Medium/Low の未解消指摘がゼロである
- SEQ、index、下流予約/設計、2つの集中トレーサビリティが一致する
- 上位変更が必要な指摘は変更管理へ起票され、未承認の上位文書を直接変更していない

# 出力

- 対象 SEQ と関連 FR/CFR/UC・下流IDの一覧
- 観点別レビュー結果(確認件数・パス/指摘数)
- 指摘一覧と修正・再確認結果
- UC基本/代替/例外→SEQ→下流設計のカバレッジ表
- 未解消事項(完了時は「なし」)
