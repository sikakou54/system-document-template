# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(呼び出し元→モジュールの分析)
- API/JOB の処理詳細で | MOD-ID | 処理名 | として委譲されている処理を集約し、このモジュールが公開すべきインターフェース(メソッド)を特定する
- 各インターフェースの命名を「XXXX取得/登録/更新/削除/判定処理」(該当しない送信・発行・算出等は末尾「処理」)に整える
- 各インターフェースの入力・出力・例外(ERR-XXX)、内部処理フロー、アクセスするテーブル(TBL)と実行クエリ(SQL)を洗い出す。テーブルアクセスには必ず対応する SQL-XXX が要る(無ければクエリ設計07で先に用意)
- トランザクション境界・排他制御、外部ライブラリ・標準実行基盤 API の利用(利用専用モジュールに限る)を整理する
- MOD-ID の採番と種別(Service/Repository/Utility/Domain)を確認する

## 作業Agent(モジュール設計作成)
- docs/03_機能設計/05_モジュール設計/index.md に先に行を追加してから、テンプレート_モジュール.md をコピーして MOD-XXX_モジュール名.md を作成する
- §3 インターフェースは公開メソッドごとに ## (n) 処理名 を作り、### 1.概要/2.入力/3.出力/4.例外/5.処理フロー/6.処理詳細 を集約する。入出力は物理名・型配列表記(Room[]等)を禁止し Object/Object[] ＋インデント展開で定義する
- 処理フロー(§3-5)は処理と判定を別ステップに分け(§8-19)、委譲(MOD/SQL/外部サービス)する処理は二重角括弧、内部処理・処理結果は矩形、判定は {"…"}(§8-18)。ノードに設計IDを添えない(§8-16)
- 処理詳細(§3-6)でテーブル参照・更新は | SQL-ID | クエリ名 | で対応クエリを示す(クエリを伴わないテーブル参照禁止。無ければ07で追加。§8-20)。戻り値を返す処理は末尾に「処理結果」ブロックを置く(§8-15)
- §4 トランザクション・排他制御、§5 データアクセス(TBL の CRUD)、§6 エラー・例外(ERR 参照)、§7 利用ライブラリ/基盤 を埋める。外部ライブラリ・基盤の直接利用は本モジュール(利用専用)内に限り、他文書は再記載しない
- traceability_scr_api_db.md の MOD→主なTBL を転記し、伴って作成した SQL は 07_クエリ設計/index.md にも反映する

## レビューAgent(エンジニア視点)
- MOD-ID・参照する TBL-ID・SQL-ID・DEF・他MOD・ERR-XXX が実在するか(§5)
- 公開インターフェース名(§3 の ## (n) 処理名)が、呼び出し元 API/JOB の | MOD-ID | 処理名 | と厳密一致しているか(呼び出し規約の整合)
- テーブルアクセスが必ず SQL-XXX を伴っているか(クエリを伴わないテーブル参照がないか。§8-20)。外部ライブラリ・基盤の直接利用が利用専用モジュールに閉じているか(§8-14)
- 処理・ステップの命名規則(取得/登録/更新/削除/判定処理)、処理と判定の分離(§8-19)、ノード形状(§8-18)、処理結果ブロック(§8-15)を満たすか
- 入出力に物理名・型配列表記(Room[]等)が使われていないか(Object/Object[]＋インデント展開か。§8-13/§9系ルール)

## レビューAgent(非エンジニア視点)
- 呼び出し元(API/JOB)が期待する処理・業務ルールが、モジュールの責務・インターフェースとして漏れなく実現されているか(網羅性)
- 各インターフェースの概要・責務が、何をする処理か非エンジニアにも読めるか(具体的な判定ロジックが概要に混ざっていないか)
- 例外・エラーが業務シナリオ(重複・不存在・期限切れなど)を正しくカバーしているか
- トランザクション境界・排他制御が、業務データの整合性(二重予約防止など)の要件と合っているか
- モジュール名・処理名が業務の意味を表しているか

# この設計の必須ルール

- SQL を直接実行してよいのは MOD のみ。テーブルのデータ参照・更新は必ず対応するクエリ(SQL-XXX)を設計し | SQL-ID | クエリ名 | で示す(クエリを伴わないテーブル参照禁止。§8-20)
- 外部ライブラリ・標準実行基盤 API の直接利用は外部ライブラリ利用専用モジュールに限る。他モジュール・API・JOB は MOD-ID＋処理名で参照する(§8-14)
- 公開インターフェース名は「XXXX取得/登録/更新/削除/判定処理」等で統一し、呼び出し元の | MOD-ID | 処理名 | と厳密一致させる(§8-12)
- 入出力に物理名・型配列表記(Room[]等)を書かず Object/Object[]＋インデント展開で定義する。処理と判定の分離(§8-19)・ノード形状(§8-18)・処理結果ブロック(§8-15)を守る
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md・traceability_scr_api_db.md、および伴うクエリ(07)を更新する(§8-5・§10-7)
- ユーザーが記法・セクション構成を指摘したら、テンプレート_モジュール.md の定義ルールコメントへ反映する(§8-10)

- 「未定」「要確認」がゼロ。テーブルアクセスに対応するクエリ(SQL)がすべて用意されている
- index.md・traceability_scr_api_db.md の行と本文の基本情報が一致し、呼び出し元の処理名と厳密一致している
