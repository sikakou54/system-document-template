# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(全体構成の把握)
- 要件定義(FR/CFR/NFR/MDL)と採用技術前提から、システムを構成するブロック(フロント・API・モジュール・データ・非同期処理・外部サービス)を洗い出す
- 各ブロックが担う責務と、ブロック間の依存・呼び出し方向を整理する
- 外部サービス連携(決済・メール送信・スケジューラ等)の連携点・方式・失敗時方針を洗い出す
- 非機能・運用方針(可用性・セキュリティ・データ保護・監査・運用)を、実現元の NFR/CFR-ID と対応づける
- 図に載せる各要素に対応する既存/予定の設計ID(SCR/API/MOD/TBL/JOB)を確認する(未作成なら「予定」と注記)

## 作業Agent(概要設計作成)
- docs/03_機能設計/01_概要設計/テンプレート_概要設計.md をコピーして NN_名称.md を作成し、§1 概要・§2 システム構成図・§3 構成要素と責務・§4 外部連携・§5 非機能・運用方針を全て埋める
- システム構成図は mermaid flowchart で、subgraph で層を区切り、ノードラベルは「表示名<br/>関連ID」で設計IDを併記する。ラベルは必ず " で囲む
- 構成図に描いた全要素を §3 構成要素と責務の表に載せる(図と表を一致させる)。詳細仕様は関連ドキュメントIDで参照し再記載しない
- 外部連携のエラーはエラーコード＋発生条件のみ記載する(文言・HTTPステータスは エラーメッセージ一覧.md 参照)
- 非機能・運用方針は方針の俯瞰にとどめ、NFR/CFR-ID で参照する(NFR 本体は再記載しない)

## レビューAgent(エンジニア視点)
- 構成図の矢印が「呼び出す側 → 呼び出される側」の依存方向で描かれているか
- 図の全要素が §3 の表に過不足なく載っているか(図と表の一致)
- 共通仕様(API-COM・共通カラム・共通定義設計)やエラー文言・メッセージ文言を再記載していないか(ID参照になっているか)
- 参照している設計ID(SCR/API/MOD/TBL/JOB)・NFR/CFR が実在するか(未作成なら「予定」と明示されているか)
- 個別画面・個別API・DBカラムの詳細まで踏み込んでいないか(概要の粒度を保っているか)

## レビューAgent(非エンジニア視点)
- 構成図・責務説明が、非エンジニアでもシステムの全体像を把握できる分かりやすさか
- 提供する機能(要件)が構成要素として過不足なく表現されているか(網羅性)
- 外部連携の用途・失敗時方針が業務観点で妥当か
- 非機能・運用方針が、利用者・組織の期待(可用性・セキュリティ等)と対応しているか
- 用語・ブロック名が要件定義の用語と一致しているか

# この設計の必須ルール

- 概要設計は俯瞰であり、個別画面(SCR)・個別API(API)・DBカラム(TBL)の詳細は各設計文書が正本。本文書ではIDで参照し再記載しない
- 共通仕様(API-COM・共通カラム・共通定義設計)、エラー(エラーメッセージ一覧.md)、メッセージ(各SCRインライン)の内容は再記載せずID参照のみとする
- 構成図のノードラベルは " で囲み、図と §3 の表を一致させる(§8-16 の簡潔性に従う)
- ユーザーが記法・セクション構成を指摘したら、テンプレート_概要設計.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- 構成図・構成要素・外部連携・非機能運用方針が要件と整合し、両レビュー観点が全てパスしている
- 「未定」「要確認」がゼロ
- 参照する設計ID・NFR/CFR がすべて実在(または「予定」と明示)している

# 出力

- 作成した概要設計文書の全文
- 参照した設計ID・NFR/CFR の一覧(実在/予定の別)
- 要件 → 構成要素 の網羅性チェック結果
- 本文に反映できなかった確認事項(なければ「なし」)
