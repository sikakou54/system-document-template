# マルチエージェント編成

本フェーズは次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・ルール・既存文書を読み、作成対象の洗い出し・分類・ID採番(一覧の最大値+1)・網羅性ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・要件と設計の非混在・実装判断できる粒度・トレース鎖の観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(企画・要求)との網羅性・業務妥当性・可読性・優先度の妥当性・専門用語の平易さの観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本フェーズの前提文書・テンプレートを読み、作業計画を立てる
2. リサーチAgent: 上流を分析し、作成対象・分類・ID採番・網羅性ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(プロット分析)
- プロット §3(ターゲットと課題)・§4(提供価値・期待する状態・開発者ゴール)・§5(体験プロセス)・§6(対象範囲・コア機能MVP・見送り)から、利用者・管理者・組織が求めること(要求候補)を抽出する
- 抽出した要求候補を要求領域(要求一覧.md §3)で分類する。分類は要求を探しやすくするための単位とする
- 各要求候補を「要求主体・目的(なぜ)・手段(何がほしい)」に分解できるか確認する。分解できないものは粒度が粗すぎる/細かすぎるため統括Agentへ差し戻す
- MVP の最小成立条件をプロット §6.2 コア機能から特定し、MoSCoW(M/S/C/W)の判定基準案を作る
- RQ-ID の採番方針(要求一覧.md の最大値+1、欠番の再利用禁止)を確認する
- プロットの課題・提供価値・体験プロセスが要求候補で網羅されているかのギャップ表(カバー済み/未カバー)を作る

## 作業Agent(要求一覧作成)
- docs/01_要求定義/要求一覧.md の各セクション定義ルールコメントに従い、§3 要求領域・§4 要求一覧・§5 要求間の関係・§6 未確認事項を埋める
- 各要求はユーザーストーリー形式(「(要求主体)として、(目的)を達成するために、(手段)がほしい」)で、目的(なぜ)と手段(何がほしい)を必ず分けて1行1件で記載する
- 優先度は MoSCoW(M/S/C/W)で付与し、判定基準を §4 冒頭に明記する(MVP は原則 M のみで構成する)
- 補足説明が必要な要求のみ、docs/01_要求定義/テンプレート_要求.md をコピーして RQ-XXX_要求名.md を作成する(先に一覧へ行を追加してから作成する)
- 後続工程ID(FR/NFR/画面/API/DB 等)・設計詳細・物理名・処理方式は書かない

## レビューAgent(エンジニア視点)
- RQ-ID が連番で、重複・欠番の再利用がないか
- 各要求が単独で充足可否を判断できる粒度か(粗すぎ・曖昧でないか)
- 設計詳細(画面・API・DB・処理方式・物理名・技術名)が混入していないか(要求は why / want のみ)
- 目的(なぜ)と手段(何がほしい)が分離され、手段だけ・目的だけの行がないか
- 要求領域の分類が一貫し、要求間の関係(§5)に矛盾がないか

## レビューAgent(非エンジニア視点)
- プロットの解決したい課題(§3.3)・提供価値(§4.1)・期待する状態(§4.2)・体験プロセス(§5)が、要求として漏れなく落ちているか(網羅性)
- 各ユーザーストーリーが利用者・管理者・組織の立場で自然か、実際の利用シーンとして納得できるか
- 優先度(MoSCoW)が企画の MVP・対象範囲(§6)と整合しているか
- 専門用語・略語なしで業務担当者が読めるか。要求名が内容を表しているか
- 見送り要件(§6.3)・対象外(§6.1)が誤って必須要求(M)に混ざっていないか

# このフェーズの必須ルール

- 要求(RQ)は後続工程のIDを参照・言及しない(docs/CLAUDE.md §5 参照方向マトリクス: RQ = なし)
- 要求一覧は 01_要求定義/要求一覧.md を正本とする。個別RQファイルは補足が必要な要求に限る
- 「一覧に行追加 → 個別ファイル作成」の順を守る(§8-5)
- 不明点は推測で埋めず、§6 未確認事項(Q-XXX)に分離する(§8-3)
- ユーザーが記法・セクション構成を指摘したら、要求一覧.md / テンプレート_要求.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- プロットの課題・価値・体験・MVP が要求で網羅され、両レビュー観点が全てパスしている
- 「未定」「要確認」がゼロ(残る確認事項は §6 未確認事項に Q-XXX として分離)
- 要求一覧.md の行と、作成した個別RQファイルの基本情報が一致している

- 作成した個別RQファイル(あれば)

- 本文に反映できなかった確認事項(Q-XXX。なければ「なし」)
