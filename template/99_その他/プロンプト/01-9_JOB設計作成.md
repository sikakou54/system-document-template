# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(FR→ジョブの分析)
- 集中トレーサビリティで対象 FR/JOB に対応する SEQ を特定して全文を読む。対応 SEQ が無い、未確定、または予約 JOB-ID と作成対象が不一致なら設計を開始せずギャップとして報告する
- SEQ から実行契機、開始/終了、MOD委譲、データ参照/更新、正常/例外結果、非同期、再試行、外部システム連携を抽出する
- トレース元 FR のシステム自動実行(主アクター=システムのユースケース)から、ジョブの目的・実行契機・スケジュール・処理対象を洗い出す
- 多重起動可否・冪等性・リトライ方針・想定処理件数/時間を整理する(実行基盤側で単一起動が保証されるかも確認)
- 処理フローの各ステップで呼び出すモジュール(MOD)と、処理対象テーブル(TBL)を特定する(データアクセスは MOD 経由。未作成 MOD/TBL は「予定」と注記)
- エラー時の対応(スキップして継続 / ジョブ中断)と、返す ERR-XXX を エラーメッセージ一覧.md から特定する(無ければ新規採番)
- SEQ の外部インターフェース候補について、呼出契機、送受信データの目的、同期/非同期、タイムアウト、再試行、冪等性、失敗通知を抽出し、概要設計の外部連携方針と照合する
- JOB-ID の採番方針を確認する

## 作業Agent(JOB設計作成)
- docs/03_機能設計/05_JOB設計/index.md に先に行を追加してから、テンプレート_ジョブ.md をコピーして JOB-XXX_ジョブ名.md を作成し、§1〜§7 を全て埋める
- §1 に実行契機・スケジュール・多重起動・冪等性・リトライ・想定処理件数/時間・トレース元 FR を定義する。多重起動が禁止ならフロー先頭に多重起動チェックを含める(基盤側保証時は §1 に方式を記し省略可)
- §4 処理フローは mermaid で、処理と判定を別ステップに分け(§8-19)、委譲する処理は二重角括弧、内部処理・処理結果は矩形、判定は {"…"}(§8-18)。ノードに設計ID・実行方式・通知内容等の詳細を書かない(§8-16)
- §5 処理詳細でデータアクセスは | MOD-ID | 処理名 | で MOD を呼び出す(JOB から SQL 直接実行禁止。§8-20)。取得・実行結果を返す処理は末尾に「処理結果」ブロックを置く(§8-15)
- §6 実行結果・出力に対象/成功/失敗/スキップ件数と実行ログを定義し、§7 エラー時の対応はエラーコード＋発生条件＋対応(継続/中断)＋通知要否で定義する(ERR 文言は再記載しない)
- traceability_sequence_design.md の SEQ→JOB/MOD/TBL/ERR/外部連携を確定内容へ更新し、JOB予約行の状態を「未着手」から「作成中」以降へ遷移させる

## レビューAgent(エンジニア視点)
- 関連 SEQ の契機、処理順、委譲、非同期、再試行、外部連携、正常/例外結果が JOB 設計へ対応しているか
- JOB-ID・参照する MOD-ID・TBL-ID・ERR-XXX が実在し、トレース元 FR が実在するか(§5)
- | MOD-ID | 処理名 | が呼び出し先 MOD の公開インターフェース名(「〜処理」)に厳密一致しているか。JOB から SQL を直接実行していないか(§8-20)
- 多重起動制御・冪等性・リトライが定義され、1件エラー時の継続/中断方針が明確か
- 処理と判定の分離(§8-19)、ノード形状(§8-18)、フローの簡潔性(§8-16)、処理結果ブロック(§8-15)を満たすか
- ERR 文言・HTTPステータスを再記載していないか(エラーコード参照か)

## レビューAgent(非エンジニア視点)
- トレース元 FR のシステム自動実行の要件が、ジョブの目的・処理対象・実行結果として漏れなく表現されているか(網羅性)
- 実行契機・スケジュールが業務運用(実行タイミング・頻度)と合っているか
- 失敗時の対応・通知が業務上妥当か(重要処理が黙ってスキップされないか)
- 実行結果・ログの項目が、運用担当者が処理状況を把握できる内容か
- ジョブ名・処理名が業務の内容を表しているか

# この設計の必須ルール

- JOB のトレース元は FR。処理は MOD へ委譲し、データアクセスは必ず MOD 経由とする(JOB から SQL 直接実行禁止。§8-20)
- 作成前に集中トレーサビリティから関連 SEQ を特定して全文を読み、SEQ の実行契機・委譲・データ操作・例外・外部連携を JOB 責務へ落とす
- ジョブは共通APIバリデーションフローを通らないため、起動パラメータの構文チェックはジョブ本体で行う(§2 に制約を記載)
- ERR はエラーメッセージ一覧.md が正本。文言・HTTPステータスを再掲せず、エラーコード＋発生条件で記載する
- 処理と判定の分離(§8-19)、ノード形状の使い分け(§8-18)、処理結果ブロック(§8-15)、フローの簡潔性(§8-16)を守る
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md を更新する(§8-5・§10-7)
- SEQ と矛盾するスケジュール、再試行、外部連携を JOB 側だけで追加しない。変更時は SEQ と traceability_sequence_design.md を同時更新する
- ユーザーが記法・セクション構成を指摘したら、テンプレート_ジョブ.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- 「未定」「要確認」がゼロ
- index.md の行と本文の基本情報が一致している
- 関連 SEQ の全 JOB 呼び出し・正常/代替/例外・ERR・外部連携が設計へ対応し、traceability_sequence_design.md と一致している
