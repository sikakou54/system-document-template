# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(共通コード・区分値の抽出)
- 集中トレーサビリティで対象 共通コード定義 または参照側設計に対応する SEQ を特定して全文を読み、状態変更、分岐条件、初期/成功/失敗状態、複数設計で共有する定数候補を抽出する
- MDL の区分値・FR/CFR の業務ルール・各設計(SCR/API/MOD/TBL)の参照ニーズから、複数文書で共有されるコード値・区分値・定数・共通ルールを洗い出す
- 各コードについて、値・表示名・説明・状態(有効/廃止)を整理する。コード種別ごとに CODE-XXX を割り当てる
- 同一コードを設定・判定する場面(初期値・成功時値・失敗時値・判定対象値)があれば設定基準 SET-XXX として整理する
- 既存 共通コード定義 に同義の定義が無いか(重複・統合の要否)を確認する
- 共通コード定義・CODE-ID(共通コード定義 内 CODE-001 連番)・SET-ID(SET-001 連番)の採番を確認する

## 作業Agent(共通定義設計作成)
- docs/03_機能設計/10_共通定義設計/index.md に先に行を追加してから、テンプレート_共通定義.md をコピーして 共通コード定義.md を作成し、§1〜§4 を埋める
- §1 基本情報(種別 Code/Rule/Constant)、§2 定義ルール(正本・参照方法・値の扱い・物理制約)を定義する
- §3 コード定義はコード種別ごとに ## CODE-XXX 名称 の見出しを作り、値・表示名・説明・状態を定義する(この節を正本とし他文書では再記載しない)
- §4 設定基準は SET-XXX で、対象コード・利用場面・設定/判定する状態(コード定義に存在する表示名)を定義する
- コード値の削除・再利用は禁止。廃止は状態「廃止」として残す
- シーケンス設計トレーサビリティ.md の SEQ→共通コード定義/CODE・SET を確定内容へ更新する。SEQ 内の候補名と確定した 共通コード定義/CODE・SET の対応を記録する

## レビューAgent(エンジニア視点)
- 関連 SEQ の状態変更・分岐・設定契機が 共通コード定義/CODE・SET へ対応し、SEQ・SCR/API/MOD/TBL 間で同じ状態を別定義していないか
- 共通コード定義・CODE-ID・SET-ID が一意・連番で、欠番の再利用・廃止コードの再利用がないか
- 値・表示名・説明が本書(§3)にのみ定義され、参照側(SCR/API/MOD/TBL)で再記載されていないか(重複記載の排除。§10-3)
- 設定基準(SET)が参照する表示名が §3 コード定義に実在するか
- テーブルの CHECK 制約等の物理制約と、値の意味の正本(本書)の役割分担が明確か
- 廃止コードが削除されず状態「廃止」で残されているか(§4 ID規則)

## レビューAgent(非エンジニア視点)
- 業務で使う区分・状態(予約状態・支払い方法・ロールなど)が、コードとして漏れなく定義されているか(網羅性)
- 表示名・説明が業務担当者に理解できる自然な言葉か(内部コード値だけになっていないか)
- コード値の意味・状態遷移が、業務ルール(MDL の区分値・FR の業務ルール)と一致しているか
- 設定基準(初期値・成功/失敗時値)が業務上の運用と合っているか
- 定義名・コード名が業務概念を表しているか

# この設計の必須ルール

- 共通コード・区分値・定数・共通ルールの正本は本書(共通コード定義)。他文書は 共通コード定義/CODE-XXX・共通コード定義/SET-XXX で参照し、値・表示名・説明を再記載しない(§3.2 カタログ正本・§10-3)
- 作成前に集中トレーサビリティから関連 SEQ を特定して全文を読み、SEQ の状態変更・分岐・設定契機を共通定義へ落とす。SEQ はコード値や表示名の正本にしない
- コード値の削除・再利用は禁止。廃止する場合は状態を「廃止」として残す(§4 ID規則)
- 値・表示名・説明は §3 コード定義でのみ定義する。テーブルの CHECK 制約など物理制約はテーブル設計に記載してよいが、値の意味は本書を正本とする
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md・シーケンス設計トレーサビリティ.md を更新する(§8-5・§10-7)
- ユーザーが記法・セクション構成を指摘したら、テンプレート_共通定義.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- 複数文書で共有されるコード値・区分値・定数が 共通コード定義 に集約され、両レビュー観点が全てパスしている
- 「未定」「要確認」がゼロ。参照側での値・表示名の重複記載がゼロ
- index.md の行と本文の基本情報が一致している
- 関連 SEQ の状態・分岐・設定契機が 共通コード定義/CODE・SET に対応し、シーケンス設計トレーサビリティ.md と一致している
