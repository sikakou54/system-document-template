# マルチエージェント編成

本フェーズは次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・ルール・既存文書を読み、作成対象の洗い出し・分類・ID採番(一覧の最大値+1)・網羅性ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・要件と設計の非混在・実装判断できる粒度・トレース鎖の観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(企画・要求)との網羅性・業務妥当性・可読性・優先度の妥当性・専門用語の平易さの観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本フェーズの前提文書・テンプレートを読み、作業計画を立てる
2. リサーチAgent: 上流を分析し、作成対象・分類・ID採番・網羅性ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 第1段階の概念MDLを「確定」にし、シーケンス設計へ引き渡す
8. シーケンス作成後、統括AgentとレビューAgent: 第2段階の整合検証を行い、ギャップは要件変更管理へ起票する。承認された変更だけを FR/CFR/UC→MDL の順で反映する

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 二段階の作成・検証ルール

| 段階 | 入力 | 実施内容 | 変更の扱い |
|---|---|---|---|
| 第1段階: 概念MDL作成 | FR/CFR/UC、要求、業務用語 | エンティティ、論理属性、関連、多重度、CRUD、整合性・セキュリティを定義し、シーケンス設計へ渡す | 通常の要件定義作成として更新 |
| 第2段階: SEQ後検証 | 確定済みMDL、該当SEQ、集中トレーサビリティ | SEQ のデータ参照・更新・状態変更・事前/事後条件が概念MDLで表現できるか検証する | SEQからMDLを直接変更しない。要件変更管理で根拠・影響FR/CFR/UCを確認し、承認後に上位から順に反映 |

- 第1段階では SEQ を入力にしない。概念MDLはシーケンスを作るための上流成果物である。
- 第2段階は循環依存を作る再設計ではなく、下流で発見した要件ギャップのフィードバック検証である。
- 物理テーブル、カラム、SQL、実装都合だけを根拠に概念MDLを追加・変更しない。
- ギャップが要件変更でない場合は、MDLを変えず、シーケンスまたは下流設計側の誤りとして修正する。

# 各エージェントの作業指示

## リサーチAgent(エンティティ抽出・関連整理)
- 各 FR/CFR のシステムユースケース(入力データ・出力データ)と業務ルールから、システムが取り扱う情報(エンティティ)を抽出する
- エンティティ間の関連(1:N / M:N / 1:0..1 など)を整理する。多対多は中間エンティティの要否を判断する
- 各エンティティのデータ項目候補(属性名・業務レベルの型区分・必須・主キー・区分値の意味)を洗い出す
- 各 FR/CFR がどのエンティティに対して C/R/U/D のどれを行うかを整理し、機能データCRUD相関.md 更新用のマトリクス案を作る。利用者操作を伴わない自動処理は「システム(自動処理)」行にまとめる
- MDL-ID の採番(index.md の最大値+1、欠番の再利用禁止)を確認する
- 機能要件が扱うデータがすべてエンティティ化されているかの網羅性ギャップ表を作る
- 第2段階では、SEQ の各データ参照・更新・状態変更を MDL/属性/関連へ対応づけ、未定義・過剰定義・多重度不整合を「要件ギャップ」「SEQ誤り」「下流詳細」のいずれかに分類する

## 作業Agent(データモデル作成)
- docs/02_要件定義/03_データモデル/index.md に先に行を追加してから、テンプレート_データモデル.md をコピーして MDL-XXX_○○情報.md を作成する
- §1 基本情報・§2 データ項目定義・§3 データライフサイクル(CRUD)・§4 関連機能/業務ルール・§5 セキュリティ を、定義ルールコメントに従って全て埋める
- 物理名(英語のテーブル名・カラム名)・物理型(varchar/int 等)は書かない。データ型は業務レベルの型区分(文字列/数値/日時/真偽 等)で記載する
- 区分値は説明列に「`値`:意味」で業務レベルの意味として記載する。設計層ID(DEF/CODE/TBL)は参照しない(要件レベルで自己完結)
- 関連機能は同層の FR/CFR/NFR(実在ID)のみ参照する。CRUD・制約は具体ロジックでなく処理の目的で記載する(論理削除等の方式は明記)
- 全エンティティを作成後、概念ER図.md(Mermaid erDiagram ＋ §3 関連一覧)を更新し、機能データCRUD相関.md を更新する
- 第2段階で要件ギャップを検出した場合は、根拠となる SEQ/ステップ、影響する FR/CFR/UC・MDL、変更案、非変更時の影響を変更管理票へ記録する。承認前に MDL 本文を直接修正しない

## レビューAgent(エンジニア視点)
- MDL-ID が重複・欠番の再利用なく、関連機能の FR/CFR/NFR-ID が実在するか
- 物理名・物理型・設計層ID(DEF/CODE/TBL)への参照が混入していないか(§5・§8-13。要件定義は機能設計を参照しない)
- データ項目が実装判断できる粒度か(主キー・必須・型区分・区分値の意味が揃っているか)
- 概念ER図の関連(多重度)と各 MDL のデータ項目(他エンティティ参照)が整合しているか。矛盾時は MDL 側を正とできるか
- 機能データCRUD相関.md の CRUD が各 MDL の §3 CRUD・§4 関連機能、および各 FR/UC の業務ルールと整合しているか
- 第2段階で SEQ→MDL の直接逆参照や無承認変更を行わず、要件変更管理を経て FR/CFR/UC→MDL→SEQ の順に再整合しているか

## レビューAgent(非エンジニア視点)
- 機能要件が扱う業務上のデータが、漏れなくエンティティ化されているか(網羅性)
- データ管理者・機密性レベル・参照/更新権限が業務観点で妥当か
- 区分値・属性の意味が業務用語で理解でき、専門用語・物理名なしで読めるか
- CRUD 相関のデータの流れ(誰が作り・参照し・更新し・消すか)が実際の業務フローと合っているか
- エンティティ名・概要が業務概念を正しく表しているか

# このフェーズの必須ルール

- データモデル(MDL)は同じ要件定義層の FR/CFR/NFR(関連要件)と他 MDL(エンティティ間の関連)のみ参照する。設計層ID(DEF/TBL 等)は参照しない(逆参照禁止。§5)
- 物理名(英語のテーブル名・カラム名)・物理型・インデックスは書かない(§8-13)。区分値・権限は要件レベルで自己完結的に記載する
- CRUD・制約は具体的な判定ロジック(SQL-ID・カラム名・具体値)でなく処理の目的で記載する(§8-11)。削除は原則、論理削除の方式を明記する
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md・概念ER図.md・機能データCRUD相関.md を更新する(§8-5・§10-7)
- 各エンティティの正本は各 MDL 文書。概念ER図・CRUD相関は転記/俯瞰であり、独自情報を持たせない(§6)
- 概念MDLは第1段階で SEQ より先に確定する。SEQ は MDL の直接入力ではなく、第2段階の妥当性検証材料としてのみ使用する
- SEQ から見つかった不足は要件変更管理へ差し戻し、承認前に MDL を直接修正しない。承認後は FR/CFR/UC、MDL、SEQ、トレーサビリティを上位から順に更新する
- ユーザーが記法・セクション構成を指摘したら、テンプレート_データモデル.md 等の定義ルールコメントへ反映する(§8-10)
