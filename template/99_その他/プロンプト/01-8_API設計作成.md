# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(UC→APIの分析)
- 集中トレーサビリティで対象 UC/API に対応する SEQ を特定して全文を読む。対応 SEQ が無い、未確定、または予約 API-ID と作成対象が不一致なら設計を開始せずギャップとして報告する
- SEQ から呼び出し元、リクエスト/レスポンス、認証、認可・所有者/ロール確認、呼び出す MOD、正常/エラーの結果、トランザクション開始・終了責務を抽出する
- トレース元 UC の入力/出力データ・基本/代替/例外フロー・業務ルールから、API のリクエスト/レスポンス項目と、DB参照・業務ルールを伴う判定(存在確認・重複など)を洗い出す
- 認証・認可・入力バリデーション(必須・型・形式・単項目・項目間相関)は共通処理フロー(API-COM §7)で実施されるため個別フローに含めない、という切り分けを行う
- 業務処理で呼び出すモジュール(MOD)とその公開処理名を特定する(データアクセスは必ず MOD 経由。未作成の MOD は「予定」と注記しモジュール設計側と整合)
- SEQ の代替/例外と ERR 候補から発生条件・処理方針・利用者/呼び出し元への結果を抽出し、既存 ERR を特定する。無ければ新規 ERR-XXX(全体連番の最大値+1)を採番する
- SEQ の権限確認を CFR-002 と照合し、APIで強制する認証・認可・所有者確認・ロール条件と、拒否時 ERR を整理する。権限定義を API 独自に拡張しない
- API-ID の採番、メソッド・パス・認可(ロール別可否)を確認する

## 作業Agent(API設計作成)
- docs/03_機能設計/04_API設計/index.md に先に行を追加してから、テンプレート_API.md をコピーして API-XXX_API名.md を作成し、§1〜§6 を全て埋める
- §4 処理フローは mermaid で、認証・認可・入力バリデーションは共通処理フローのため記載せず、存在確認・重複など業務判定のみを含める。ノードはフロー出現順に (1)(2)… で採番する
- 処理と判定を必ず別ステップに分ける(処理ノードから直接分岐させない。判定は「(x) 処理名の結果」を参照。§8-19)。委譲する処理は二重角括弧 [["(n)…"]]、内部処理・処理結果・エラーは矩形、判定は {"(n)…"}(§8-18)
- §5 処理詳細は処理型・判定型・エラー型で定義する。データアクセスは | MOD-ID | 処理名 | で MOD を呼び出す(API から SQL 直接実行禁止。§8-20)。レスポンスを返す API は末尾に「処理結果」ブロックを置き | 項目名 | データ型 | 値 | 説明 | で定義する(§8-15)
- エラーレスポンス封筒(API-COM §4)・エラー文言(エラーメッセージ一覧.md)は再記載せず、エラー型ステップでは | エラーコード | 引数 | 値 | のみ定義する。認証方式・ページネーション等 API-COM の共通事項は「準拠」等も書かず差分のみ記載する
- §6 バリデーションは成立条件(AND/OR 論理式)＋違反時の details[].message(利用者向け文言)を項目ごとに定義する(コードは常に ERR-006 のため記載しない)
- 新規 ERR は エラーメッセージ一覧.md に追加し、traceability_sequence_design.md の SEQ→API/MOD/ERR へ転記する
- traceability_sequence_design.md の SEQ→API/MOD/ERR/権限確認を確定内容へ更新し、API予約行の状態を「未着手」から「作成中」以降へ遷移させる

## レビューAgent(エンジニア視点)
- 関連 SEQ の呼び出し、入出力、委譲、結果、トランザクション境界が API 文書へ対応し、SEQ で抽出された ERR・認可確認が欠落していないか
- API-ID・参照する MOD-ID・ERR-XXX・共通コード定義 が実在するか。トレース元 UC が完全修飾で実在するか(§5)
- | MOD-ID | 処理名 | の処理名が、呼び出し先 MOD の §3 インターフェース公開メソッド名(「〜処理」)に厳密一致しているか(内部ステップ名・外部サービス名でないか)
- 認証・認可・入力バリデーションを個別フローに書いていないか。API から SQL を直接実行していないか(データアクセスが MOD 経由か。§8-20)
- 処理と判定が分離され(§8-19)、ノード形状が委譲有無に対応し(§8-18)、ノード/エッジに設計ID・外部サービス名・ステータス値等の詳細を書いていないか(§8-16)
- 処理結果ブロック(§8-15)・エラー型ステップが定義され、共通仕様(API-COM・エラー文言)を再記載していないか

## レビューAgent(非エンジニア視点)
- トレース元 UC の基本/代替/例外フロー・業務ルールが、処理フロー・判定・エラーとして漏れなく表現されているか(網羅性)
- リクエスト/レスポンス項目が、その UC で扱う入力/出力データと一致しているか
- 認可(ロール別可否)が業務上の権限と合っているか
- バリデーションの違反時メッセージ(details[].message)が利用者に分かりやすい文言か(形式名・型名・内部IDを出していないか)
- 返すエラーが業務シナリオ(会議室なし・重複予約など)を正しくカバーしているか

# この設計の必須ルール

- API のトレース元はシステムユースケースの完全修飾ID(FR-XXX/UC-01)。実装は MOD へ委譲し(依存参照)、データアクセスは必ず MOD 経由とする(API・JOB から SQL 直接実行禁止。§8-20)
- 作成前に traceability_uc_sequence.md・traceability_sequence_design.md から関連 SEQ を特定して全文を読み、SEQ の呼び出し・入出力・認証認可・委譲・正常/例外結果・トランザクション責務を API へ落とす
- 認証・エラー封筒・ページネーション等の共通事項は API-COM_共通設計.md が正本。本文書は差分のみ記載し「準拠」等の言及も書かない
- ERR はエラーメッセージ一覧.md が正本(全体連番)。本文書は文言・HTTPステータスを再掲せず、エラーはフローのエラーノード＋エラー型ステップで表現する
- 処理と判定の分離(§8-19)、ノード形状の使い分け(§8-18)、処理結果ブロック(§8-15)、フローの簡潔性(§8-16)を守る
- | MOD-ID | 処理名 | は呼び出し先 MOD の公開インターフェース名に厳密一致させる。該当処理が MOD に無ければモジュール設計(06)に定義してから参照する
- SEQ と矛盾する API 仕様や権限条件を API 側だけで追加しない。変更が必要なら影響する SEQ・UC・CFR-002 へ変更管理で戻す
- ユーザーが記法・セクション構成を指摘したら、テンプレート_API.md の定義ルールコメントへ反映する(§8-10)
