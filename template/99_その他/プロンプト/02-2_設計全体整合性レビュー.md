# 必須レビュー観点

## 1. 構成・番号・参照

- 機能設計が 01_概要、02_シーケンス、03_画面、04_API、05_JOB、06_モジュール、07_データベース、08_クエリ、10_共通定義、11_トレーサビリティの構成である
- 旧 02_画面、03_API、04_JOB、05_モジュール、06_データベース、07_クエリ、08_シーケンスへの参照が残っていない
- README、CLAUDE、IDルール、目次、index、テンプレート、作成/レビュープロンプトの番号・パス・説明が一致する
- Markdownリンク、画像参照、相対パスが実在し、一覧の文書リンクが切れていない
- `09` の扱い(権限正本を CFR-002 に集約し欠番を維持する等)が明文化され、別の権限正本と競合しない

## 2. ID・一覧・状態

- RQ/CFR/FR/NFR/MDL/SEQ/SCR/API/JOB/MOD/TBL/SQL/共通コード定義/ERR/MSG が採番規則に従い、重複・欠番再利用・未登録参照がない
- 各個別文書の基本情報と index.md の名称・概要・トレース元・依存・状態が一致する
- SEQ が参照する下流IDは参照前に各 index.md へ「未着手」で予約され、文書作成後に適切な状態へ遷移している
- 「未着手」なのに個別文書が確定済み、または「確定」なのに文書が無い等の状態矛盾がない

## 3. 上位→SEQ

- すべての対象 FR/CFR/UC が SEQ または省略理由へ対応し、UCシーケンストレーサビリティ.md から追跡できる
- SEQ の正常/代替/例外、事前/事後条件、データ操作が UC・業務ルール・概念MDLと一致する
- 要件に無い処理が SEQ に混入せず、SEQで発見した概念MDL不足が 01-4 の第2段階/要件変更管理を経ている
- 概要設計の構成境界・外部連携・同期/非同期・非機能制約が SEQ に反映される

## 4. SEQ→下流設計

- シーケンス設計トレーサビリティ.md で SEQ→SCR/API/JOB/MOD/MDL/TBL/SQL/ERR/MSG を追跡できる(SEQ本文は連携・責務を定義し、下流設計IDの網羅対応は本対応表に一元化する)
- SCR が操作、入力、イベント、API契機、結果表示、遷移、MSGを反映する
- API が呼び出し元、入出力、認証認可、MOD委譲、正常/エラー結果、トランザクション責務、ERRを反映する
- JOB が契機、スケジュール、非同期、冪等、再試行、通知、MOD委譲を反映する
- MOD が責務、入出力、業務ルール、分岐、データ操作、他MOD、例外、トランザクションを反映する
- TBL/SQL が参照/更新、状態、属性、関連、制約、クエリ目的/入出力/対象、ロックを反映する
- 権限確認が CFR-002 と一致し、外部インターフェースが概要の方針と SEQ の連携順へ対応する

## 5. 契約・責務・技術整合

- SCR→API の呼び出し、API/JOB→MOD の公開処理名、MOD→SQL→TBL の対象と入出力が一致する
- UI/API/MOD/DB の責務が分離され、API/JOB が SQL を直接実行せず、外部ライブラリ利用が専用MODに閉じる
- トランザクション境界、排他、冪等、再試行、タイムアウト、エラー伝播、利用者/運用者通知が文書間で矛盾しない
- APIのERR、SCRのMSG、ログ/通知の発生契機が SEQ の代替/例外と一対一または根拠ある多対一で対応する
- 共通コード定義/CODE・SET、共通カラム、API共通仕様等の共通定義を参照し、個別文書で値・文言・共通仕様を再定義していない

## 6. 正本・参照方向・重複

- 要件、連携順、画面、API、業務ロジック、論理/物理データ、SQL、ERR、MSG、権限、外部IF、共通定義の正本が各1箇所である
- 下位成果物一覧を上位本文へ逆参照として重複せず、逆方向追跡は集中トレーサビリティで行う
- 対応表・index・概要図は正本の転記/俯瞰にとどまり、独自仕様を持たない
- 同じ条件・値・文言・処理順が複数文書で矛盾または重複していない

# 指摘の記録形式

| 指摘ID | 重要度 | 経路/観点 | 証跡(ファイル・節・ID) | 現状 | 期待する状態 | 修正先の正本 | 影響ID | 状態 |
|---|---|---|---|---|---|---|---|---|
| XDR-001 | Critical/High/Medium/Low |  |  |  |  |  |  | 未対応/対応中/解消/非該当 |

- 指摘は必ず具体的なファイル・節・IDと、正本からの修正順を含める。
- 単に「整合すること」と書かず、期待するID・契約・状態を明記する。
- 問題なしの観点も、確認対象数・一致数・除外数と証跡を記録する。

# 完了条件

- 全必須観点を対象ID全件について確認し、件数と証跡を記録している
- Critical/High/Medium/Low の未解消指摘、旧パス、参照切れ、未登録ID、正本重複がゼロである
- RQ→FR/CFR→UC→SEQ→下流設計のトレースが断絶せず、分岐する ERR/MSG/権限/外部IF/共通コード定義 を追跡できる
- index・個別文書・集中トレーサビリティのID・状態・責務が一致する
- 02-1 の全対象 SEQ がパスし、レビュー後の変更について再レビュー済みである
