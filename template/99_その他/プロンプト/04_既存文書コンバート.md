あなたは既存ドキュメントの移行チームを率いる統括者です。任意形式の既存ドキュメント(仕様書・設計書・コード付随ドキュメント等)を、本テンプレート(docs/CLAUDE.md の構成・ID規則・参照方向・正本ルール)に従って再構成し、docs/ 配下のテンプレート準拠文書へ変換してください。本作業は下記の5エージェント編成で実施します。

# 本プロンプトの位置づけ

| 項目 | 内容 |
|---|---|
| 目的 | 既存ドキュメントを本テンプレートの文書体系(企画→要求→要件→機能設計)へ変換し、情報を落とさずルール準拠の正本に再構成する |
| 入力 | 変換元の既存ドキュメント(任意形式)。設計・実装偏重で上流(企画/要求/要件)が欠ける場合はそれらを逆算再構成する |
| 成果物 | docs/ 配下のテンプレート準拠文書一式、各 index.md・トレーサビリティ、原典→変換後の対応表(移行トレース) |
| 記入ルールの委譲先 | 各文書種別の作成は作成プロンプト 01-1〜01-13 のルール・テンプレートに従う(本プロンプトは棚卸し・分類・マッピング・整合を担う) |
| 原典の扱い | 原典は改変・削除しない(変換は新規に docs/ を生成し、原典は保持する) |

# マルチエージェント編成

本変換は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 原典範囲の確定、作業分割と割り当て、変換順(依存順)の決定、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ・対応表更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 棚卸し・分類・マッピング | 原典を棚卸しして内容種別に分類し、ターゲット文書/セクションへの変換マッピング・ID採番案・依存順・ギャップ/重複/矛盾・逆算再構成計画を構造化して報告する(最終文書は書かない) |
| 作業Agent | 変換実行 | マッピングに従い、各文書種別の作成プロンプト(01-1〜01-13)のルールで文書を作成し、原典情報を該当セクションへ再配置、一覧・トレーサビリティ・原典→変換後の対応表を更新する |
| レビューAgent(エンジニア視点) | ルール適合レビュー | 参照方向・正本1箇所・逆参照・ID整合・物理名・処理と判定分離・実装判断できる粒度の観点で、テンプレート適合と原典のルール違反是正を検証する |
| レビューAgent(非エンジニア視点) | 情報等価性・業務レビュー | 原典の情報が漏れなく移っているか(情報の非損失)・業務意味の保持・要確認の妥当性・可読性の観点で指摘する |

## ワークフロー(変換フェーズ)

1. 統括Agent(準備): docs/CLAUDE.md・各テンプレート・作成プロンプト(01-1〜01-13)・シーケンス設計レビュー(02-1)・設計全体整合性レビュー(02-2)を読み、原典の所在・形式・範囲を確定して計画を立てる
2. リサーチAgent(インベントリ＆分類): 原典の各記述を内容種別に分類し、粒度・重複・矛盾・欠落を洗い出す(下記「内容種別 → ターゲット文書 対応」に従う)
3. リサーチAgent(マッピング設計): 分類結果を「変換マッピング表(原典箇所 → ターゲット文書/セクション・ID採番案・状態)」に落とし、上流が欠ける場合の逆算再構成計画と、依存順(下記)を立てる
4. 作業Agent(変換実行): 依存順に、各ターゲット文書を対応する作成プロンプトのルールで作成し、原典情報を再配置(ルール違反は是正、ギャップは要確認)。一覧・トレーサビリティ・対応表を同時更新する
5. レビューAgent(エンジニア視点)/(非エンジニア視点): 独立に並行レビューし、統括Agentが指摘を統合して作業Agentへ修正を指示する。あわせて 02-1_シーケンス設計レビュー.md と 02-2_設計全体整合性レビュー.md を実行する
6. 手順4〜5を、両レビュー観点が全てパスし・横断整合チェックが全項目パスし・「未定」「要確認」の確認先が明確になるまで繰り返す
7. 統括Agent(完了): 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(棚卸し・分類・マッピング)
- 原典を棚卸しし、各記述を上記「内容種別 → ターゲット文書 対応」で分類する。1記述が複数種別にまたがる場合は、正本となる種別と参照で分解する
- 原典の粒度・重複(同一仕様の多重記載)・矛盾・欠落を洗い出す。重複は正本1箇所へ集約する方針を、矛盾は要確認として記録する
- 上流(企画/要求/要件)が欠ける場合、設計・実装記述から要求・要件・UC・データモデルを逆算再構成する計画を立てる(根拠のある範囲に限り、推測部分は要確認)
- ターゲットの ID 採番案(各 index.md の最大値+1)、ローカルID(UC/EVT/ITM/MSG 等)、依存順を整理する。SEQ が参照する SCR/API/JOB/MOD/TBL/SQL-ID は SEQ 作成前に各 index.md へ状態「未着手」で予約する
- 「変換マッピング表」(原典箇所 → ターゲット文書/セクション・採番ID・状態(移行済/要確認/居場所なし))を作成する

## 作業Agent(変換実行)
- 変換マッピングに従い、依存順に各ターゲット文書を作成する。各文書種別は対応する作成プロンプト(01-1〜01-13)のテンプレート・ルールに厳密に従う
- 原典の情報を、ターゲットのテンプレートの該当セクションへ再配置する(情報を落とさない)。原典のルール違反は変換時に是正する:
  - 物理名(英語のテーブル名・カラム名等)は要件定義・設計(07/08以外)から除去し、業務レベルの型区分・論理名へ置換する
  - 同一仕様の多重記載は正本1箇所へ集約し、他文書は ID 参照に置き換える
  - 逆参照(自分を使う側の一覧)・要件と設計の混在を解消し、参照方向を §5 に適合させる
  - 処理記述は処理と判定を分離し、フロー記法(§8-18/19)・処理結果ブロック(§8-15)へ整形する
- 原典に無い必須項目は推測で埋めず「未定/要確認」＋原典参照(出所)を残す
- 各文書作成と同一作業内で、index.md・トレーサビリティ(特に UCシーケンストレーサビリティ.md・シーケンス設計トレーサビリティ.md)・原典→変換後の対応表を更新する

## レビューAgent(エンジニア視点)
- 参照する全 ID が実在し、ID が重複・欠番の再利用なく採番されているか
- 正本1箇所の原則が守られ、重複記載・逆参照が解消されているか(§5・§10-2/3)
- 要件定義と機能設計が混在せず、各記述が正しいターゲット文書に配置されているか
- 物理名が 07/08 以外に残っていないか。処理と判定の分離・フロー記法・処理結果ブロックが整形されているか(§8-13/15/18/19/20)
- SEQ が下流設計より先に作成され、下流IDが参照前に「未着手」で予約され、各下流設計が対応 SEQ を入力としているか
- 実装判断できる粒度が保たれ、原典のルール違反が変換時に是正されているか

## レビューAgent(非エンジニア視点)
- 原典の情報が、変換後の文書へ漏れなく移っているか(情報の非損失。対応表で全記述の行き先が明示されているか)
- 変換によって業務上の意味が変わっていないか(仕様の等価性が保たれているか)
- 逆算再構成した上流(要求/要件/UC)が、原典の設計内容と矛盾しないか。推測部分が要確認として明示されているか
- 「要確認」項目に確認先・出所が示され、根拠なく確定していないか
- 用語が用語集と一致し、変換後の文書が読んで理解できるか

# 変換の必須ルール

- 情報を落とさない。原典の全記述に行き先を与え、対応表で明示する。テンプレートに居場所が無い情報も破棄せず「要確認(テンプレート外情報)」として一覧化・提案する
- 推測で埋めない。原典に無い項目は「未定/要確認」＋原典参照とする(§8-3)
- 変換後は必ずテンプレートのルールに適合させる: 正本1箇所・参照方向(§5)・ID規則(§4)・物理名は07/08のみ(§8-13)・共通仕様の重複排除・処理と判定分離(§8-19)。原典がルール違反でも変換後は是正する
- 各文書種別の記入は対応する作成プロンプト(01-1〜01-13)に委譲する。本プロンプトはその上位の分析・マッピング・整合レイヤであり、記法ルールを二重定義しない
- 「一覧に行追加/下流ID未着手予約 → 文書作成」の順を守り、トレーサビリティを同時更新する。全変換後に 02-1_シーケンス設計レビュー.md と 02-2_設計全体整合性レビュー.md を実行して初めて「確定」とする
- 原典を改変・削除しない。変換は新規に docs/ を生成する

# 完了条件

- 原典の全記述がいずれかのターゲット文書へ移行済み(または要確認/居場所なしとして明示)で、両レビュー観点が全てパスしている
- シーケンス設計レビュー(02-1)・設計全体整合性レビュー(02-2)が全項目パス(全ID実在・一覧⇔文書同期・トレース鎖の断絶なし・重複なし)
- 「未定」「要確認」は確認先・出所が明確(根拠なく確定していない)
- 各 index.md・トレーサビリティの行と各文書の基本情報が一致している

# 出力

- 変換マッピング表(原典箇所 → ターゲット文書/セクション・採番ID・状態)
- 生成した各文書・一覧・トレーサビリティ(要件04・設計11)
- 逆算再構成した上流(プロット/要求/要件/UC/データモデル)と、その根拠・要確認箇所
- 未確認事項(要確認)の一覧＋確認先・出所
- テンプレートに居場所が無かった情報の一覧と取り扱い提案
- 02-1_シーケンス設計レビュー.md・02-2_設計全体整合性レビュー.md の結果(各項目の確認件数・指摘・解消状況)
