## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(モジュール処理→クエリの分析)
- 集中トレーサビリティで対象 SQL/MOD/TBL に対応する SEQ を特定して全文を読む。対応 SEQ が無い、未確定、または予約 SQL-ID と作成対象が不一致なら設計を開始せずギャップとして報告する
- SEQ のデータ参照・更新から、クエリの目的、入力、出力、対象 MDL/TBL、処理順、トランザクション・ロック・排他要件を抽出する。SEQにはSQL文を逆流させない
- MOD の処理詳細で | SQL-ID | クエリ名 | として要求されているクエリを集約し、作成対象のクエリを特定する(使用元 MOD-ID を控える)
- 各クエリの対象テーブル(TBL)・物理名、結合・抽出・集計の要件、入力バインドパラメータ(:name・型・必須・制約)を洗い出す
- 出力列(別名・論理名・型・算出方法)を整理する。集計列は算出方法(COUNT/SUM/CAST 等)を明確化する
- 論理削除(DELETED_AT IS NULL で有効行限定)・区分値(DEF-XXX/CODE-XXX)・日時規約を確認する
- 利用インデックス(TBL-XXX/IDX-x)・想定件数・性能上の注意点を洗い出す。SQL-ID の採番を確認する
- SEQ の参照/更新ごとに対応 SQL があるかを確認し、同一トランザクション内の順序・行ロック・競合時結果を MOD/DB 設計と照合する

## 作業Agent(クエリ設計作成)
- docs/03_機能設計/08_クエリ設計/index.md に先に行を追加してから、テンプレート_クエリ.md をコピーして SQL-XXX_クエリ名.md を作成し、§1〜§6 を全て埋める
- §1 使用元は MOD-ID で参照する。§2 対象テーブルは TBL-ID＋物理名で示す(カラム定義は再記載しない)
- §4 クエリは採用RDB方言の実SQLで記載する。バインド変数は §3 のパラメータ名(:name)を用い、論理削除は DELETED_AT IS NULL で有効行に限定する。区分値は DEF-XXX/CODE-XXX を参照し値の意味を再定義しない
- §5 出力(列)・§6 補足(利用インデックス TBL-XXX/IDX-x・想定件数・注意点)を定義する
- 日時・月の受け取り形式はプロジェクトの日時規約に従い、可変長リスト(IN 句)はアプリ側展開の旨を制約に記す
- traceability_sequence_design.md の SEQ→MOD/SQL/TBL を確定内容へ更新し、SQL予約行の状態を「未着手」から「作成中」以降へ遷移させる

## レビューAgent(エンジニア視点)
- 関連 SEQ の全データ参照・更新に対応 SQL があり、目的・順序・トランザクション/ロック要件が MOD/TBL と整合しているか
- SQL-ID・使用元 MOD-ID/API-ID・対象 TBL-ID・利用インデックス TBL-XXX/IDX-x・DEF が実在するか
- SQL 内のテーブル・カラム物理名が データベース設計(TBL)と一致し、命名規則に沿っているか
- 論理削除条件(DELETED_AT IS NULL)が有効行限定に入っているか。区分値の意味を再定義せず DEF 参照になっているか
- バインドパラメータが §3 と §4 で一致し、型・必須・制約が実装判断できる粒度か
- 使用元 MOD の処理詳細の | SQL-ID | クエリ名 | と、本クエリの ID・クエリ名が対応しているか(相互整合)

## レビューAgent(非エンジニア視点)
- クエリの用途・概要が、使用元モジュールの処理目的(何のためのデータ取得・更新か)と一致しているか
- 出力列が、業務で必要な情報(一覧表示項目・集計値など)を過不足なく返すか(網羅性)
- 集計・抽出条件が業務ルール(対象期間・有効なデータのみ 等)と合っているか
- クエリ名・列の論理名が業務用語で理解できるか
- 想定件数・性能上の注意点が、利用場面(一覧・レポートなど)の実態に合っているか

# この設計の必須ルール

- クエリ設計(08)は物理名(テーブル・カラム)・実SQLを記載してよい例外領域(§8-13)。テーブル・カラムの物理名/型の正本は データベース設計(TBL)、区分値の正本は 共通定義設計(DEF)。本文書は再記載せず ID で参照する
- 作成前に集中トレーサビリティから関連 SEQ を特定して全文を読み、SEQ のデータ参照/更新を目的・入力・出力・対象MDL/TBL・トランザクション要件へ落とす
- SQL を実行するのは MOD のみ(§8-20)。本クエリは MOD の処理詳細から | SQL-ID | クエリ名 | で参照される。テーブルアクセスにはクエリ設計が必ず伴う(クエリを伴わないテーブル参照は禁止)
- 論理削除は DELETED_AT IS NULL で有効行に限定する(共通カラム.md)。日時規約・IN 句の扱いはプロジェクト規約に従う
- 「一覧の未着手予約行を確認/状態更新 → 文書作成」の順を守り、同一作業内で index.md・traceability_sequence_design.md を更新する(§8-5・§10-7)
- SEQ と矛盾するデータ操作を SQL 側だけで追加しない。実SQL・インデックスなど物理詳細を SEQ へ重複記載しない
- ユーザーが記法・セクション構成を指摘したら、テンプレート_クエリ.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- 「未定」「要確認」がゼロ。SQL のテーブル・カラム物理名が TBL と一致している
- index.md の行と本文の基本情報が一致し、使用元 MOD の | SQL-ID | クエリ名 | と対応している
- 関連 SEQ の全データ参照/更新が SQL に対応し、traceability_sequence_design.md と一致している
