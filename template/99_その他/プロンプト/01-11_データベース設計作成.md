# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(エンティティ→テーブルの分析)
- 集中トレーサビリティで対象 MDL/TBL に対応する SEQ を特定して全文を読む。対応 SEQ が無い、未確定、または予約 TBL-ID と作成対象が不一致なら設計を開始せずギャップとして報告する
- SEQ から参照/更新対象のエンティティ、状態変更、必要属性、関連、整合性制約、トランザクション境界、排他・一意性の要求を抽出し、概念MDLと照合する
- MDL のデータ項目定義から、テーブルのカラム(論理名・物理名・型・NULL・PK・Default)を洗い出す。物理名は命名規則.md に従う(テーブル: M_/T_/H_/TP_ 接頭辞、大文字)
- 他エンティティを参照する項目を外部キー(FK)として特定し、参照先「テーブル名.物理名」と削除時挙動(RESTRICT/CASCADE/SET NULL)を整理する
- 検索・整合性のためのインデックス(IDX-x・一意性)と、複数カラムにまたがる制約(UNIQUE/CHECK)を洗い出す
- 区分値カラムが参照する 共通コード定義/CODE-XXX を特定する(値・表示名・説明は再記載しない)
- 共通カラム(CREATED_AT/UPDATED_AT/DELETED_AT)は共通カラム.md が正本のため本文書に含めない点を確認する。TBL-ID の採番を確認する
- SEQ に必要だが MDL に無い属性・関連・状態を見つけた場合は、物理設計だけで補わず 01-4 の第2段階ルールに従って要件変更管理へ差し戻す

## 作業Agent(データベース設計作成)
- docs/03_機能設計/07_データベース設計/index.md に先に行を追加してから、テンプレート_テーブル.md をコピーして TBL-XXX_テーブル名.md を作成し、§1〜§5 を全て埋める
- §2 カラム定義は主キー ID を先頭に、業務カラムを物理名(大文字)・型・NULL・PK/FK・Default・補足で定義する。共通カラム(CREATED_AT/UPDATED_AT/DELETED_AT)は記載しない
- FK は参照先「テーブル名.物理名」を記し削除時挙動を補足に、区分値カラムは補足に 共通コード定義/CODE-XXX を記載する。§3 インデックス(IDX-x)・§4 制約・§5 共通定義参照を定義する
- 全テーブル作成後、ER図.md を更新する(エンティティ・関連・多重度。物理カラムは書かず概念/主要関連中心)
- 論理名は種別接尾辞(〜マスタ/〜トランザクション/〜履歴/〜一時テーブル)を付ける
- traceability_sequence_design.md の SEQ→MDL/TBL を確定内容へ更新し、TBL予約行の状態を「未着手」から「作成中」以降へ遷移させる

## レビューAgent(エンジニア視点)
- 関連 SEQ の参照/更新、状態変更、整合性、トランザクション・排他要求が TBL のカラム/FK/制約/インデックスへ適切に反映されているか
- TBL-ID・FK 参照先テーブル・共通コード定義/CODE-XXX が実在するか。物理名(テーブル・カラム・インデックス)が命名規則.md に従い大文字か
- 共通カラム(CREATED_AT/UPDATED_AT/DELETED_AT)を再記載していないか(共通カラム.md 参照か)。区分値の値・表示名を再記載していないか(共通コード定義 参照か)
- 主キー・外部キー・一意制約・インデックスが整合し、FK の削除時挙動が定義されているか
- ER図.md の関連・多重度が各 TBL のカラム定義(FK)と整合しているか(矛盾時は TBL 側を正とできるか)
- 逆参照(このテーブルを使う MOD 一覧など)を書いていないか(§5・§10-2)

## レビューAgent(非エンジニア視点)
- トレース元 MDL のデータ項目が、テーブルのカラムとして漏れなく物理化されているか(網羅性)
- 論理名・概要が業務のデータ概念を正しく表しているか(物理名だけでなく論理名で意味が追えるか)
- 区分値・制約が業務ルール(重複禁止・有効期間など)と対応しているか
- 論理削除(DELETED_AT)などのデータライフサイクルが MDL の CRUD 方針と一致しているか
- テーブルの粒度・関連が、業務の実体(会議室・予約・利用者など)と自然に対応しているか

# この設計の必須ルール

- データベース設計(07)は物理名(英語のテーブル名・カラム名)を記載してよい例外領域(§8-13)。物理名は命名規則.md に従い大文字で記載する
- 作成前に集中トレーサビリティから関連 SEQ を特定して全文を読み、SEQ のデータ参照/更新・状態変更・整合性・トランザクション要求を MDL とともに物理設計へ落とす
- 共通カラム(CREATED_AT/UPDATED_AT/DELETED_AT)は共通カラム.md が正本。各 TBL に再記載しない(主キー ID は §2 に記載する)
- 区分値カラムの値・表示名・説明は共通定義設計が正本。本文書は 共通コード定義/CODE-XXX 参照とし再記載しない
- テーブル(TBL)は他TBL(FK)・共通コード定義 のみ参照する(逆参照禁止。§5)。トレース元として MDL を参照する場合は対応表で管理する
- 「一覧の未着手予約行を確認/状態更新 → 文書作成」の順を守り、同一作業内で index.md・ER図.md・traceability_sequence_design.md を更新する(§8-5・§10-7)
- SEQ で発見した概念データの不足を TBL だけで補わず、要件変更管理へ差し戻す。SEQ と矛盾する制約をDB側だけで追加しない
- ユーザーが記法・セクション構成を指摘したら、テンプレート_テーブル.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- トレース元 MDL のデータ項目がテーブルへ物理化され、両レビュー観点が全てパスしている
- 「未定」「要確認」がゼロ。ER図.md が最新のテーブル・関連を反映している
- index.md・ER図.md の内容と各 TBL 文書の基本情報・カラム(FK)が整合している
- 関連 SEQ の全データ参照/更新・状態変更・整合性要求が TBL に対応し、traceability_sequence_design.md と一致している
