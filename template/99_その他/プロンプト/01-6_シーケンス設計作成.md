# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割、下流ID予約、成果統合、レビュー指摘の採否、一覧・トレーサビリティ更新、状態管理、最終出力 |
| リサーチAgent | 上流分析・設計対象抽出 | FR/CFR/UC・概要・概念MDLを読み、シナリオ、構成要素、分岐、データ操作、下流設計対象、整合ギャップを構造化して報告する(最終文書は書かない) |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md、上流文書、テンプレート、既存一覧、トレーサビリティを読み、対象 UC と作成単位を確定する
2. リサーチAgent: 上流だけを根拠に、正常系・代替系・例外系、構成要素、データ参照/更新、下流設計対象を抽出する
3. 統括Agent: SEQ-ID を採番し、シーケンスが参照する予定の SCR/API/JOB/MOD/TBL/SQL-ID を対応 index.md に状態「未着手」で予約する
4. 作業Agent: 予約済みIDだけを用いて SEQ 文書・SEQ一覧・2つのトレーサビリティ表を作成する
5. レビューAgent 2名: 独立にレビューし、要件逸脱、漏れ、責務混在、詳細化しすぎ、ID不整合を指摘する
6. 統括Agent: 指摘を統合して修正を指示し、全観点がパスするまでレビューを繰り返す
7. 統括Agent: 下流設計への引き渡し一覧と未着手予約行を確認し、SEQ を「確定」にして最終成果を出力する

各Agentは docs/CLAUDE.md の正本管理、参照方向、状態、作成、禁止事項の各ルールに従う。

# 各エージェントの作業指示

## リサーチAgent(上流→連携・下流設計対象の分析)

- UC の事前条件・事後条件・基本/代替/例外フロー・入力/出力データ・業務ルールを根拠として、正常系・代替系・例外系を抽出する
- 構成要素は原則として具体的なロールの利用者、UI、API、モジュール、単一の「データベース」(参照する概念MDLを集約)とする。ジョブ・外部システム・非同期基盤は該当時だけ追加する
- UI の操作・入力・イベント・表示・遷移、API の入出力・認証認可・ステータス・委譲先、モジュールの責務・分岐・データ操作を下流設計候補として抽出する
- 参照/更新する概念 MDL、状態変更、整合性制約、トランザクション境界、排他制御、非同期、再試行を整理する
- 代替系・例外系から ERR 候補、利用者向け MSG 候補、ログ/通知候補を抽出する。ここでは文言や未採番IDを確定しない
- アクター・操作・データ所有関係から認証・認可・所有者確認・ロール確認を抽出し、CFR-002 との整合を確認する
- 外部システム呼び出しについて、契機、送受信、同期/非同期、成功/失敗、再試行・タイムアウトの設計対象を抽出する
- 静的表示や自明な単純CRUDで省略する場合も、SEQ一覧に対象 UC と省略理由を記録する

## 統括Agent(下流IDの先行予約)

- シーケンス内で参照する SCR/API/JOB/MOD/TBL/SQL-ID は、文書作成前でも必ず対応 index.md に名称・概要・トレース元/使用元を記載し、状態を「未着手」として予約する
- 予約時は各一覧の最大値+1を用い、欠番を再利用しない。既存IDで責務を満たせる場合は重複採番しない
- 予約されていない下流ID、文書にも一覧にも存在しないID、仮の XXX-ID を確定済み SEQ から参照しない
- ID予約は設計詳細の確定ではない。下流設計で責務変更が必要になった場合は SEQ と集中トレーサビリティへ変更影響を戻す

## 作業Agent(シーケンス設計作成)

- 02_シーケンス設計/index.md に SEQ 行を追加し、テンプレートをコピーして SEQ-XXX_シーケンス名.md を作成する
- 基本情報に目的、対象範囲、関連FR/CFR/UC、事前・事後条件、構成要素を記載する(SEQは機能要件を入力とし、要求RQは紐付けない)
- Mermaid sequenceDiagram で、処理順序、要求/応答、alt/else、opt、loop、非同期を表現する。メッセージ名は業務上の動詞形式とする。データベースは単一participant「データベース」に集約し、往路(→DB)メッセージ末尾に参照データモデル名を全角括弧で列挙する(復路には付けない。MDL-ID・TBL-ID・具体的なERR-ID・MSG-IDは図に書かず、エラー・メッセージは責務レベルで記述する)
- 正常系・代替系・例外系、条件分岐、データ参照/更新、トランザクション境界、補足を定義する
- 下流設計への引き渡しとして、関連 SCR/API/JOB/MOD/MDL/TBL/SQL、ERR候補、MSG候補、認証認可、外部インターフェースを整理する
- SQL文、物理アクセス方式、変数操作、ライブラリ固有メソッド、画面/API/モジュールの詳細仕様は書かない
- UCシーケンストレーサビリティ.md に FR/UC→SEQ、シーケンス設計トレーサビリティ.md に SEQ→予約済み下流ID・MDL・ERR/MSG候補を転記する

## レビューAgent(エンジニア視点)

- シーケンスが FR/CFR/UC・概要・概念MDLだけを根拠に作られ、未作成の下流詳細を入力としていないか
- 参照する SCR/API/JOB/MOD/TBL/SQL-ID が対応 index.md に「未着手」以上の状態で実在するか
- ユーザー/UI/API/モジュール/DBの責務と方向が自然で、UI/APIに業務ロジック、DBにアプリケーション責務を持たせていないか
- 正常/代替/例外、データ参照/更新、トランザクション境界、認証認可、外部連携、非同期・再試行が必要範囲で表現されているか
- 具体的SQL、物理テーブル中心の説明、変数操作、実装メソッドが混入していないか
- SEQ一覧、下流予約一覧、2つのトレーサビリティ表、本文のIDが一致するか

## レビューAgent(非エンジニア視点)

- 対象 UC の基本・代替・例外フローが、開始から結果まで時系列に漏れなく表現されているか
- 具体的ロールの利用者が何を行い、UIが何を返すかを追跡できるか
- 権限不足、入力不備、データ不整合、外部サービス失敗時の利用者/運用者への結果が明確か
- 業務上必要な画面、API、ジョブ、通知、データ更新が下流引き渡しに漏れていないか
- 用語が要件・概念MDLと一致し、内部実装を知らなくても理解できるか

# この設計の必須ルール

- シーケンス設計は下流個別設計の共通入力であり、下流設計の寄せ集めとして後から作らない
- 上流の振る舞いを連携順序へ具体化するが、個別設計の詳細仕様を重複記載しない
- 下流IDは参照前に対応 index.md へ「未着手」で予約する。予約済みIDだけを本文・図・トレーサビリティで使用する
- ERR/MSG は候補と発生・表示契機を抽出し、正本での採番・文言確定は API/画面設計へ引き渡す
- 要件または概念MDLの不足を発見しても SEQ から上位文書を直接書き換えず、要件変更管理へ差し戻す
- 文書作成と同一作業内で SEQ一覧、下流予約一覧、UCシーケンストレーサビリティ.md、シーケンス設計トレーサビリティ.md を更新する

# 完了条件

- 対象 UC の正常・代替・例外がシーケンスへ反映され、下流設計への引き渡し項目が揃っている
- 参照する SCR/API/JOB/MOD/TBL/SQL-ID がすべて各 index.md に状態「未着手」以上で存在する
- SEQ本文・SEQ一覧・下流予約一覧・2つのトレーサビリティ表のIDが一致している
- 詳細設計の重複、具体的SQL、ライブラリ固有処理、「未定」「要確認」が残っていない
- 両レビュー観点がすべてパスしている

# 出力

- 作成した SEQ 文書の全文(Mermaidシーケンス図を含む)
- SEQ一覧に追加した行
- 下流各 index.md に「未着手」で予約した SCR/API/JOB/MOD/TBL/SQL の行
- UCシーケンストレーサビリティ.md・シーケンス設計トレーサビリティ.md に追加・更新した行
- UC の基本/代替/例外カバレッジと、画面/API/JOB/MOD/DB/クエリ/ERR/MSG/権限/外部連携への引き渡し一覧
- 要件または概念MDLへ変更管理で差し戻すギャップ(なければ「なし」)
