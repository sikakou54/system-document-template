# マルチエージェント編成

本設計は次の5エージェントで分担する。統括Agentが全体を管理し、各Agentへ作業を割り当て、成果を統合する。

| Agent | 役割 | 主な責務 |
|---|---|---|
| 統括Agent | 全体統括・意思決定 | 前提の読み込み確認、作業分割と割り当て、各成果の統合、レビュー指摘の採否判断、一覧・トレーサビリティ更新の担保、状態遷移(未着手→作成中→レビュー中→確定)の管理、最終成果物の出力 |
| リサーチAgent | 上流分析・抽出 | 上流文書・共通仕様・既存設計を読み、作成対象の洗い出し・依存関係・ID採番(一覧の最大値+1)・整合ギャップを構造化して報告する(最終文書は書かない) |
| 作業Agent | 文書作成 | リサーチ結果とテンプレートに基づき文書を作成し、全セクションを埋め、一覧に行を追加し、トレーサビリティを更新する |
| レビューAgent(エンジニア視点) | 技術・整合レビュー | ID整合・参照方向・逆参照・共通仕様との重複排除・実装判断できる粒度・トレース鎖・記法ルールの観点で指摘する |
| レビューAgent(非エンジニア視点) | 業務・網羅性レビュー | 上流(要件・ユースケース)との整合・業務妥当性・利用者にとっての分かりやすさ・網羅性の観点で指摘する |

## ワークフロー

1. 統括Agent: docs/CLAUDE.md と本設計の前提文書・テンプレート・共通仕様を読み、作業計画を立てる
2. リサーチAgent: 上流・依存・共通仕様を分析し、作成対象・依存関係・ID採番・整合ギャップを報告する
3. 作業Agent: リサーチ結果を基に、文書・一覧・トレーサビリティを作成する
4. レビューAgent(エンジニア視点)と(非エンジニア視点): それぞれ独立に並行してレビューし、指摘一覧を出す
5. 統括Agent: 両レビューの指摘を統合し採否を判断して、作業Agentへ修正を指示する
6. 手順4〜5を、両レビュー観点が全てパスし「未定」「要確認」がゼロになるまで繰り返す
7. 統括Agent: 状態を「確定」にし、最終成果物(下記「出力」)をまとめる

各Agentは docs/CLAUDE.md の作成ルール(§8)・参照方向ルール(§5)・状態定義(§7)・禁止事項(§10)に従う。

# 各エージェントの作業指示

## リサーチAgent(連携の分析)
- 対象の連携(利用者操作・定期実行・Webhook 受信など)について、契機と登場要素(SCR/API/MOD/JOB/TBL/外部サービス)を洗い出す
- 各要素間のメッセージ授受(要求・応答)の時系列を、既存の SCR・API・MOD 等の設計から組み立てる
- 分岐(alt/else)・繰り返し(loop)・補足(Note)が必要な箇所と、例外・代替の流れ(返す ERR-XXX・更新状態・表示 MSG-XXX)を洗い出す
- 認証・認可・入力検証は API-COM §7 共通前処理として Note で示す方針を確認する
- 登場要素の設計IDが実在するかを確認し、SEQ-ID の採番を確認する

## 作業Agent(シーケンス設計作成)
- docs/03_機能設計/08_シーケンス設計/index.md に先に行を追加してから、テンプレート_シーケンス.md をコピーして SEQ-XXX_シーケンス名.md を作成し、§1〜§5 を全て埋める
- §2 登場要素を種別・ID/参照・役割で列挙し、§3 シーケンス図(mermaid sequenceDiagram)を participant/actor で宣言して時系列に描く
- 分岐は alt/else、繰り返しは loop、補足は Note で表す。エラー・メッセージは ERR-XXX/MSG-XXX で参照する(文言・HTTPステータスは書かない)
- 認証・認可・入力検証は API-COM §7 共通前処理として Note で示す(各段で再記載しない)
- §4 ステップ説明(番号順・送信元→送信先・内容)と §5 例外・代替(分岐・分岐後の流れ)を定義する。内部仕様は各文書を参照し再記載しない

## レビューAgent(エンジニア視点)
- 登場要素の設計ID(SCR/API/MOD/JOB/TBL/外部)・ERR-XXX/MSG-XXX が実在するか(§5)
- §1 関連要素・§2 登場要素・§3 図の participant が一致しているか
- メッセージの方向・順序が各要素の設計(呼び出し関係)と矛盾しないか(SCR→API→MOD→TBL の依存方向と整合)
- 内部ロジックの詳細(各要素の内部処理)を再記載していないか(連携順序の俯瞰にとどまっているか)
- 共通前処理(認証・認可・入力検証)を各段で重複記載せず Note で示しているか。文言・HTTPステータスを書いていないか

## レビューAgent(非エンジニア視点)
- 対象ユースケースの一連の流れ(利用者操作→結果)が、時系列として漏れなく・自然に表現されているか(網羅性)
- シーケンス図が、非エンジニアでも「誰が→何を→次に何が起きるか」を追える分かりやすさか
- 例外・代替の分岐(失敗時・キャンセル時など)が業務シナリオを正しくカバーしているか
- 外部サービス連携(決済・メール等)の順序・タイミングが業務の実態と合っているか
- 要素名・ステップ内容が業務用語で理解できるか

# この設計の必須ルール

- シーケンスは連携順序の俯瞰。各要素(SCR/API/MOD/JOB/TBL/外部)の内部仕様は各文書が正本で、本文書は再記載せず ID 参照する
- エラー・メッセージは ERR-XXX/MSG-XXX で参照する(文言・HTTPステータスは書かない)。認証・認可・入力検証は API-COM §7 共通前処理として Note で示す
- メッセージの向き・順序は各要素の依存方向(呼び出す側→呼び出される側)と整合させる
- 「一覧に行追加 → 文書作成」の順を守り、同一作業内で index.md を更新する(§8-5・§10-7)
- ユーザーが記法・セクション構成を指摘したら、テンプレート_シーケンス.md の定義ルールコメントへ反映する(§8-10)

# 完了条件

- 「未定」「要確認」がゼロ。登場要素の設計IDがすべて実在している
- index.md の行と本文の基本情報(契機・関連要素)が一致している
