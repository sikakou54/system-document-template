# 3. 文書種別ごとの追加観点

| 文書 | 観点 |
|---|---|
| FR | 受入条件が検証可能な形で書かれている／画面・API・DB の詳細が書かれていない |
| NFR | 達成基準と検証方法が分離され、数値・条件・対象期間などで判定できる |
| UC | 基本・代替・例外フローが分離されている／事前・事後条件がある |
| MDL | FR/CFR/NFRを根拠とし、SEQで判明した不足が要件変更として反映済み／設計層IDを逆参照していない |
| SEQ | FR/CFR・UCを根拠とし(要求RQは紐付けない)、目的・範囲・事前事後条件・正常/代替/例外・条件分岐・CRUD・トランザクション境界がある／参加者は利用者・UI・API/JOB・MOD・単一の「データベース」で、データモデルごとに参加者を分けず往路メッセージ末尾に参照データモデル名を括弧で列挙し、MDL-ID・TBL-IDを図に書かない／§4.2データ参照更新は概念データモデル(MDL)で記載し物理テーブル(TBL)を併記しない(TBL・SQLは11_トレーサビリティ・07/08で管理)／エラー・メッセージは責務レベルで記述し具体的なERR-ID・MSG-IDを図に書かない／具体SQL・変数・物理メソッドがない |
| SCR | 関連SEQの操作・入力・イベント・API契機・正常/エラー表示・遷移を反映し、初期表示・権限・0件時・失敗時が定義されている |
| API | 関連SEQの呼出元・入出力・認証認可・MOD・正常/異常結果・トランザクション責務を反映し、エラーがERR参照になっている |
| JOB | 関連SEQの契機・非同期・再試行を反映し、リトライ・冪等性・多重起動制御が定義されている／TBL・SQLへ直接アクセスしない |
| MOD | 関連SEQの責務・入出力・業務ルール・分岐・CRUD・ERRを反映し、トランザクション境界・排他制御が定義されている |
| TBL | 関連SEQのMDL参照更新・状態変更・整合性を物理化し、共通カラムを再記載していない／FK削除時挙動がある |
| SQL | 関連SEQのデータ参照更新を目的・入力・出力・対象MDL/TBLへ落とし、使用元がMODだけである／具体SQLは本書だけにある |
| DEF | 関連SEQ・各設計で共通化が必要な値だけを定義し、利用元の逆引き一覧を持たない |
