# 5. このドキュメント群の目的

対象システム「{プロダクト名}」の企画・要求定義・要件定義・機能設計ドキュメント一式である。
以下の開発プロセスのうち「企画・構想」「要求定義」「要件定義」「機能設計」の成果物を docs/ 配下で管理する。

設計は「1.目的・スコープ → 2.ユースケース・業務ルール → 3.非機能要件 → 4.データ・状態 → 5.責務・境界 → 6.論理契約 → 7.シーケンス検証 → 8.個別・運用設計」の順で進める。シーケンスは Step 5・6 で確定した責務と論理契約を検証する成果物である。

```text
企画・構想 ＞ 要求定義 ＞ 要件定義 ＞ 機能設計 ＞ コーディング ＞ テスト ＞ リリース
```

# 7. 必須作成順と推奨読み順

作成・更新は次の順序を必須とする。各 Step の確定条件と承認状態は [99_その他/工程ゲート.md](99_その他/工程ゲート.md) で管理する。

| Step | 必須成果物 | 次工程への引き渡し |
|---|---|---|
| 1 目的・スコープ | README、00_企画、01_要求定義 | 対象者、解決課題、対象/対象外、今回の設計ベースライン |
| 2 UC・業務ルール | FR/CFR、完全修飾UC、受入条件 | 正常・代替・例外フロー、状態パターン、不変条件 |
| 3 非機能要件 | NFR、FR適用表 | 数値化した達成基準・測定条件・検証方法 |
| 4 データ・状態 | MDL、概念ER図、CRUD相関 | データ所有、正本、状態遷移、整合性・保持ルール |
| 5 責務・境界 | 01_概要設計の責務・境界節 | 構成要素の責務/非責務、所有データ、依存方向 |
| 6 論理契約 | 01_概要設計の論理契約節 | 呼出元/先、論理入出力、成否、認証認可、冪等性・タイムアウト |
| 7 シーケンス検証 | SEQ、UC・契約カバレッジ | Step 2〜6の整合確認、正常・失敗・再試行・並行更新の時系列 |
| 8 個別・運用設計 | SCR/API/JOB/MOD/TBL/SQL/共通定義/OPS | 実装契約、物理設計、監視・復旧・障害対応 |

前 Step のゲートが「承認済」でないとき、次 Step の成果物を「確定」にしない。上流成果物の変更時は、当該 Step 以降のゲートを「失効」に戻し、上位から再レビューする。
