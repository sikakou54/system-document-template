# 9. 要求・設計トレーサビリティ

本章は、全業務要求とログイン・ログアウトの根拠となる非機能要求から機能・想定される全ユースケース・画面・API・モジュール・JOB・SQL・DBまでの実現経路を定義する。API/JOBの実行境界とデータアクセスの一元化（M-002/M-003委譲・M-006限定・直接アクセス禁止）は[サンプル・技術基盤としての前提](00_はじめに.md#02-サンプル技術基盤としての前提)・[JOB設計](06_JOB設計.md#6-job設計)・[モジュール責務](07_モジュール設計.md#72-モジュール責務)を正本とし、各行がこの境界に沿うことを確認する。

## 9.1 追跡ルール

- 正引きは「BR/NFR → F → UC → SCR/API/JOB → M → SQL → DB」、逆引きはこの逆順で確認する。
- F-001〜F-015とUC-001〜UC-015を本表へ1回以上記載し、クライアント内完結はAPI/JOB/SQL/DBを「なし」、システム起動は画面/APIを「なし」と明示する。
- 画面初期表示に必要なマスター取得API・SQLは、その画面を利用する要求行にも併記する。
- SQL-031 監査ログ登録は監査対象の更新・出力・権限変更要求へ対応付ける。
- 「なし」は設計漏れではなく、その要求に画面、APIまたはJOBの起動境界が存在しないことを示す。
- DB列は4章[テーブル一覧](04_データベース設計.md#42-テーブル一覧)で定義したTBL-ID（該当時はFTS-ID）で記載し、物理名との対応は[テーブル一覧](04_データベース設計.md#42-テーブル一覧)を正本とする。
