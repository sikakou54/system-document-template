# 8. クエリ設計

## 8.1 クエリ設計方針

- SQLはCloudflare D1で利用できるSQLite構文とし、物理テーブル・カラムは4.データベース設計を正本とする。
- Workerの`env.DB` bindingを取得しSQLを実行できるのはM-006だけとする。API・JOB・M-001〜M-005・M-007・M-008はDBへ直接アクセスせず、SQL本文、SQL-ID、binding、prepared statement、D1結果を保持しない。
- 本章のSQLコードブロックはM-006が`prepare()`する実SQLである。D1が対応するordered placeholderだけを使い、Prepared Statementごとに`?1`、`?2`…を欠番なく割り当てる。複数構成文を持つSQL-IDは構成文ごとに独立して`?1`から採番し、各「固定バインド順（構成文）」を論理名との唯一の対応表とする。名前付きplaceholderへ実行時変換しない。
- 1 statementのbind上限は100個（最大`?100`）とし、M-006が実行前に検証する。本章35件の現行最大は`?13`である。
- M-006は固定SQL文字列を`env.DB.prepare(sql).bind(v1, v2, ...)`で準備し、単一参照は`first()`/`all()`、単一更新は`run()`、複数更新は`env.DB.batch([...])`で実行する。値・JSON配列・ページング値をSQL文字列へ連結しない。
- TEXT、日付、UTC日時、JSONは`TEXT`、真偽は`INTEGER`（0/1）、件数・版数は`INTEGER`としてバインドする。日付は`YYYY-MM-DD`、日時は`YYYY-MM-DDTHH:mm:ss.SSSZ`、ID配列は重複のないJSON文字列配列とする。
- 社員検索・詳細・基本情報更新・出力の基準日はM-002が確定した業務日だけを渡し、API入力で過去日を指定させない。組織・役職候補だけは`effectiveOn`（省略時は業務日）を利用できる。
- 社員番号・氏名の部分一致は`instr(lower(対象), lower(?N)) > 0`を正本とし、入力をLIKE wildcardやFTS検索式として解釈しない。FTS5は任意の性能移行であり、本章の意味論を変えない。
- 一覧は安定ソートの末尾に主キーを含める。通常一覧はLIMIT/OFFSET、JOB対象は複合条件によるキーセットカーソルを使う。
- D1は単一DB内の`batch()` statementを順次実行し、全statement成功時だけ確定し、いずれかのstatement例外時は全体を取り消す。M-006はDB接続単位の明示制御やDBロック構文を発行しない。
- 更新バッチは、既存SQL-IDの構成文として、(A) `TX_MUTATION_GUARD` INSERT、(B) 所有者version UPDATE、(C) 所有者の更新後versionとguard存在を条件にする従属更新、(D) guard DELETEの順で組み立てる。未採番SQLや上位層の隠れSQLを追加してはならない。
- ガードINSERT時のトリガーが事前取得した所有者versionと期待対象ID集合をDB現況と双方向比較する。所有者更新が0件ならすべての従属更新も0件となる。M-006は期待1件の所有者更新0件を競合へ変換し、差分更新0件は正常結果として扱う。
- 即時再試行は[処理詳細](06_JOB設計.md#635-処理詳細)に列挙した5エラーだけを対象とし、M-002が参照からバッチ全体を初回後最大2回再実行する。書込結果不明はM-006経由の再読込でcommit確認し、overload、timeout、CPU・memory超過は即時再試行しない。制約・トリガー・業務競合も再試行しない。
- 氏名・カナはM-002でUnicode前後空白trim、NFC、空文字拒否、長さ・形式検証を済ませる。組織・役職の`active`は有効期間と独立した0/1値とする。

### 8.1.1 更新SQL-IDの構成文契約

| 対象SQL-ID | バッチ先頭の構成文 | 所有者更新 | 従属更新後の末尾構成文 |
|---|---|---|---|
| SQL-008 / 011 / 013 / 034 | 同じSQL-IDのA: EMPLOYEE所有者guard INSERT | 同じSQL-IDのB: EMPLOYEEを期待versionで更新 | 同じSQL-IDのC: guard DELETE |
| SQL-020 | SQL-020-A: ORGANIZATION所有者guard INSERT | SQL-020-B: ORGANIZATION更新 | SQL-020-C: guard DELETE |
| SQL-025 | SQL-025-A: POSITION所有者guard INSERT | SQL-025-B: POSITION更新 | SQL-025-C: guard DELETE |
| SQL-035 | SQL-035-A: USER_ACCOUNT所有者・ロール更新集合guard INSERT | SQL-035-B: USER_ACCOUNT更新 | SQL-035-C: guard DELETE |

SQL-034/013/035のCは、同じD1バッチに含むSQL-010/006/007またはSQL-029/030/007の後へ配置する。構成文A〜Cは当該SQL-IDの一部であり、IF番号を増やさない。SQL-006/007/010/029/030は`guard_id`と更新後の`guarded_version`を照合する。SQL-005を所有者とする新規登録バッチだけはguard不要で、SQL-006/007が登録直後のversion=1を確認する。

### 8.1.2 バインド値・D1結果の共通契約

各SQLの「入力」は業務入力を説明し、「固定バインド順」は実SQLが使用する業務入力・技術入力の全件を列挙する。両者に差がある場合も固定バインド順を実装正本とし、M-006以外が技術入力を生成しない。

| 論理名 | D1バインド型 | NULL | 生成・検証 |
|---|---|---|---|
| `guard_id` | TEXT | 条件付 | M-006が更新試行ごとにUUIDを生成する。SQL-005を所有者とする新規登録バッチのSQL-006/007に限りNULL可（登録直後のversion=1を確認）、他はNULL不可 |
| `guard_owner_type` | TEXT | 不可 | `EMPLOYEE`または`USER_ACCOUNT`。履歴登録時の所有者種別 |
| `guard_owner_id` | TEXT | 不可 | 履歴登録を保護する所有者ID |
| `now_utc` | TEXT | 不可 | M-006が`YYYY-MM-DDTHH:mm:ss.SSSZ`で1回生成し、同じ業務バッチ内で共用する |
| `guarded_version` | INTEGER | 不可 | M-006が`expected_version + 1`を安全な整数演算で算出する |
| `expected_assignment_ids_json` | TEXT | 不可 | SQL-009のOWN対象IDを重複なし・順序非依存のJSON文字列配列へ変換する。対象なしは`[]` |
| `expected_user_role_ids_json` | TEXT | 不可 | 終了・取消予定のUSER_ROLE IDを重複なしJSON文字列配列へ変換する。対象なしは`[]` |
| `retained_user_role_ids_json` | TEXT | 不可 | 保持するUSER_ROLE IDを重複なしJSON文字列配列へ変換する。対象なしは`[]` |
| `allowed_organization_ids_json` | TEXT | 不可 | 認可済み組織IDを重複なしJSON文字列配列へ変換する。対象なしは`[]` |
| `actor_user_id` | TEXT | 条件付 | JOBまたは認証前監査だけNULL可。それ以外は認証済み利用者ID |

M-006は参照を`first()`または`all()`、単一文DMLを`run()`、複数文業務更新を`batch()`で実行する。`RETURNING`を持つ単一文は`D1Result.results`、バッチでは対応する各`D1Result.results`から行を取得し、更新件数は`meta.changes`を正本とする。`created_at`/`updated_at`は返却行または同じ`now_utc`から構成できる。SQL-031は業務更新成功後の独立した単一文`run()`とする。

所有者更新Bは`meta.changes = 1`、guard削除Cも`meta.changes = 1`を期待する。差分更新SQL-010/029は0件以上、SQL-006/007/030は呼出計画上の期待件数と`results`件数を照合する。statement例外時はD1がバッチを取り消す。バッチ成功後の期待件数不一致は整合性事故として上位成功へ変換せず、エラーログと運用通知を行う。

## 8.3 SQL-001 社員検索

| 項目 | 内容 |
|---|---|
| 目的 | 検索条件と閲覧可能範囲に一致する社員を現所属付きでページ取得する |
| 入力 | `employee_number TEXT`（NULL可）、`name TEXT`（NULL可）、`organization_id TEXT`（NULL可）、`position_id TEXT`（NULL可）、`status TEXT`（NULL可）、`scope_type TEXT`（必須）、`actor_employee_id TEXT`（SELF時必須）、`allowed_organization_ids_json TEXT（JSON文字列配列）`（必須、対象なしは空配列）、`as_of_date TEXT（YYYY-MM-DD）`（必須）、`limit INTEGER`（1以上）、`offset INTEGER`（0以上） |
| 固定バインド順 | `?1` = `as_of_date`、`?2` = `employee_number`、`?3` = `name`、`?4` = `organization_id`、`?5` = `position_id`、`?6` = `status`、`?7` = `scope_type`、`?8` = `actor_employee_id`、`?9` = `allowed_organization_ids_json`、`?10` = `limit`、`?11` = `offset` |
| 出力 | 社員ID、社員番号、表示氏名、組織・役職、在籍状態、版数を0件以上、最大`limit`件 |
| 0件時 | 0行。空の社員一覧を返す。 |
| ソート・ページング | 社員番号、社員IDの昇順。`LIMIT/OFFSET`。 |
| 利用インデックス | IDX-001, IDX-006, IDX-007, IDX-021 |
| ロック・競合 | 読取ロックなし。閲覧範囲条件を必須適用する。 |
| 注意 | `as_of_date`にはAPI入力値ではなくM-002が確定した業務日だけを設定する。employee_number・nameはUnicode trim/NFC後のリテラル値であり、`instr(lower(...), lower(?N))`で検索する。許可組織はNULLでなくJSON空配列`[]`を使用する。 |

```sql
WITH ranked_assignment AS (
    SELECT ea.employee_id, ea.organization_id, ea.position_id,
           row_number() OVER (
               PARTITION BY ea.employee_id
               ORDER BY ea.effective_from DESC, ea.assignment_id
           ) AS rn
    FROM "EMPLOYEE_ASSIGNMENT" ea
    WHERE ea.deleted_at IS NULL
      AND ea.effective_from <= ?1
      AND (ea.effective_to IS NULL OR ea.effective_to >= ?1)
)
SELECT e.employee_id,
       e.employee_number,
       e.last_name || ' ' || e.first_name AS display_name,
       a.organization_id,
       o.organization_name,
       a.position_id,
       p.position_name,
       e.status,
       e.version
FROM "EMPLOYEE" e
LEFT JOIN ranked_assignment a
  ON a.employee_id = e.employee_id AND a.rn = 1
LEFT JOIN "ORGANIZATION" o
  ON o.organization_id = a.organization_id AND o.deleted_at IS NULL
LEFT JOIN "POSITION" p
  ON p.position_id = a.position_id AND p.deleted_at IS NULL
WHERE e.deleted_at IS NULL
  AND (?2 IS NULL
       OR instr(lower(e.employee_number), lower(?2)) > 0)
  AND (?3 IS NULL
       OR instr(lower(e.last_name || ' ' || e.first_name), lower(?3)) > 0)
  AND (?4 IS NULL OR a.organization_id = ?4)
  AND (?5 IS NULL OR a.position_id = ?5)
  AND (?6 IS NULL OR e.status = ?6)
  AND (
      ?7 = 'ALL'
      OR (?7 = 'SELF' AND e.employee_id = ?8)
      OR (
          ?7 = 'ORGANIZATION'
          AND EXISTS (
              SELECT 1
              FROM json_each(?9) j
              WHERE CAST(j.value AS TEXT) = a.organization_id
          )
      )
  )
ORDER BY e.employee_number, e.employee_id
LIMIT ?10 OFFSET ?11;
```

## 8.4 SQL-002 社員検索件数取得

| 項目 | 内容 |
|---|---|
| 目的 | SQL-001と同じ条件の総件数を取得する |
| 入力 | `employee_number TEXT`（NULL可）、`name TEXT`（NULL可）、`organization_id TEXT`（NULL可）、`position_id TEXT`（NULL可）、`status TEXT`（NULL可）、`scope_type TEXT`（必須）、`actor_employee_id TEXT`（SELF時必須）、`allowed_organization_ids_json TEXT（JSON文字列配列）`（必須、対象なしは空配列）、`as_of_date TEXT（YYYY-MM-DD）`（必須） |
| 固定バインド順 | `?1` = `as_of_date`、`?2` = `employee_number`、`?3` = `name`、`?4` = `organization_id`、`?5` = `position_id`、`?6` = `status`、`?7` = `scope_type`、`?8` = `actor_employee_id`、`?9` = `allowed_organization_ids_json` |
| 出力 | `total_count INTEGER`を1行 |
| 0件時 | 0行にはならず、該当なしは`total_count = 0`の1行。 |
| ソート・ページング | なし。 |
| 利用インデックス | IDX-001, IDX-006, IDX-007, IDX-021 |
| ロック・競合 | 読取ロックなし。SQL-001と同一の検索・閲覧範囲条件を適用する。 |
| 注意 | SQL-001からページング値だけを除き、業務日・リテラル部分一致・閲覧範囲の条件差異を作らない。 |

```sql
WITH ranked_assignment AS (

           row_number() OVER (
               PARTITION BY ea.employee_id

    WHERE ea.deleted_at IS NULL
      AND ea.effective_from <= ?1
      AND (ea.effective_to IS NULL OR ea.effective_to >= ?1)
)
SELECT count(*) AS total_count
FROM "EMPLOYEE" e
LEFT JOIN ranked_assignment a
  ON a.employee_id = e.employee_id AND a.rn = 1
WHERE e.deleted_at IS NULL
  AND (?2 IS NULL
       OR instr(lower(e.employee_number), lower(?2)) > 0)
  AND (?3 IS NULL
       OR instr(lower(e.last_name || ' ' || e.first_name), lower(?3)) > 0)
  AND (?4 IS NULL OR a.organization_id = ?4)
  AND (?5 IS NULL OR a.position_id = ?5)
  AND (?6 IS NULL OR e.status = ?6)
  AND (
      ?7 = 'ALL'
      OR (?7 = 'SELF' AND e.employee_id = ?8)
      OR (
          ?7 = 'ORGANIZATION'
          AND EXISTS (
              SELECT 1
              FROM json_each(?9) j
              WHERE CAST(j.value AS TEXT) = a.organization_id
          )
      )

```

## 8.5 SQL-003 社員詳細取得

| 項目 | 内容 |
|---|---|
| 目的 | 閲覧可能範囲をSQL条件で強制し、対象社員の基本情報と基準日時点の所属・役職・上長を取得する |
| 入力 | `employee_id TEXT`、`scope_type TEXT`（ALL / SELF / ORGANIZATION）、`allowed_organization_ids_json TEXT（JSON文字列配列）`（必須、対象なしは空配列）、`as_of_date TEXT（YYYY-MM-DD）`（以上必須）、`actor_employee_id TEXT`（SELF時必須） |

```sql
        WHEN effective_from <= ?1 THEN ?1
        ELSE effective_to
    END,
    deleted_at = CASE
        WHEN effective_from > ?1 THEN ?2
        ELSE deleted_at
    END,

    updated_by = ?3,

      SELECT 1
      FROM "TX_MUTATION_GUARD" g
      JOIN "EMPLOYEE" e

       AND e.deleted_at IS NULL

        AND g.owner_type = 'EMPLOYEE'
        AND g.owner_id = ?4

      (
          effective_from <= ?1
          AND (effective_to IS NULL OR effective_to > ?1)
      )
      OR effective_from > ?1
  )
RETURNING assignment_id,
          CASE
              WHEN effective_from > ?1 THEN 'CANCELED'
```

## 8.19 SQL-017 組織取得

| 項目 | 内容 |

| 固定バインド順 | `?1` = `reference_as_of_date`、`?2` = `organization_id` |
| 出力 | 組織情報と、参照基準日現在・将来の所属参照件数、参照最小開始日、有限の参照最遠終了日、無期限参照有無を0または1行 |

| 利用インデックス | ORGANIZATION主キー、IDX-007 |
| ロック・競合 | 読取ロックなし。更新時はSQL-020のversion guardと組織階層・所属参照トリガーが再検証する。 |

```sql
    SELECT ea.organization_id,
           count(*) AS assignment_reference_count,
           min(ea.effective_from) AS reference_min_effective_from,
           max(CASE WHEN ea.effective_to IS NOT NULL
                    THEN ea.effective_to END) AS reference_max_finite_effective_to,
           max(CASE WHEN ea.effective_to IS NULL THEN 1 ELSE 0 END)
               AS has_open_ended_reference
    FROM "EMPLOYEE_ASSIGNMENT" ea
    WHERE ea.deleted_at IS NULL
      AND (ea.effective_to IS NULL
           OR ea.effective_to >= ?1)
    GROUP BY ea.organization_id
)
SELECT o.organization_id, o.organization_code, o.organization_name,
       o.parent_organization_id, o.effective_from, o.effective_to,
       o.active, o.version,
       coalesce(refs.assignment_reference_count, 0)
           AS assignment_reference_count,
       refs.reference_min_effective_from,
       refs.reference_max_finite_effective_to,
       coalesce(refs.has_open_ended_reference, 0)
           AS has_open_ended_reference
FROM "ORGANIZATION" o
LEFT JOIN refs ON refs.organization_id = o.organization_id
WHERE o.organization_id = ?2
  AND o.deleted_at IS NULL;
```

## 8.24 SQL-022 役職取得

| 項目 | 内容 |
|---|---|
| 目的 | 役職IDで役職を1件取得し、存在・有効性と現在・将来所属による期間変更可否の判定に使用する |
| 入力 | `position_id TEXT`、`reference_as_of_date TEXT（YYYY-MM-DD）`（すべて必須） |
| 固定バインド順 | `?1` = `reference_as_of_date`、`?2` = `position_id` |
| 出力 | 役職情報と、参照基準日現在・将来の所属参照件数、参照最小開始日、有限の参照最遠終了日、無期限参照有無を0または1行 |
| 0件時 | 0行。対象役職不存在。 |
| ソート・ページング | なし。 |
| 利用インデックス | POSITION主キー、IDX-021 |
| ロック・競合 | 読取ロックなし。更新時はSQL-025のversion guardと役職・所属参照トリガーが再検証する。 |
| 注意 | 論理削除済み役職は返さない。`assignment_reference_count > 0`のとき、更新後active=0、参照最小開始日を包含しない開始日、有限の参照最遠終了日を包含しない終了日、または`has_open_ended_reference=true`に対する有限終了日はM-005がMASTER_PERIOD_INVALIDとする。 |

```sql
WITH refs AS (
    SELECT ea.position_id,
           count(*) AS assignment_reference_count,
           min(ea.effective_from) AS reference_min_effective_from,
           max(CASE WHEN ea.effective_to IS NOT NULL
                    THEN ea.effective_to END) AS reference_max_finite_effective_to,
           max(CASE WHEN ea.effective_to IS NULL THEN 1 ELSE 0 END)
               AS has_open_ended_reference
    FROM "EMPLOYEE_ASSIGNMENT" ea
    WHERE ea.deleted_at IS NULL
      AND (ea.effective_to IS NULL
           OR ea.effective_to >= ?1)
    GROUP BY ea.position_id
)
SELECT p.position_id, p.position_code, p.position_name, p.position_level,
       p.effective_from, p.effective_to, p.active, p.version,
       coalesce(refs.assignment_reference_count, 0)
           AS assignment_reference_count,
       refs.reference_min_effective_from,
       refs.reference_max_finite_effective_to,
       coalesce(refs.has_open_ended_reference, 0)
           AS has_open_ended_reference
FROM "POSITION" p
LEFT JOIN refs ON refs.position_id = p.position_id
WHERE p.position_id = ?2
  AND p.deleted_at IS NULL;
```

## 8.28 SQL-026 認証利用者・ロール取得

| 項目 | 内容 |

| 利用インデックス | IDX-013、USER_ACCOUNT主キー、EMPLOYEE主キー、IDX-006、IDX-017、ROLE主キー |
| ロック・競合 | 読取ロックなし。認証・認可判定用の一貫した1回の読取とする。 |
| 注意 | `identity_subject`と`user_id`はXOR条件とする。社員に紐づく利用者はACTIVEかつ退職日前だけ返し、退職日当日はJOB反映前でも0行とする。社員外管理者は在籍条件の対象外。認証・操作権限・社員閲覧範囲はいずれもロール・所属の両基準日を業務日に固定し、APIから過去基準日を受け付けない。ロールなしはJSON空配列。 |

```sql
WITH ranked_assignment AS (
    SELECT ea.employee_id, ea.organization_id, ea.position_id,
           row_number() OVER (
               PARTITION BY ea.employee_id
               ORDER BY ea.effective_from DESC, ea.assignment_id
           ) AS rn
    FROM "EMPLOYEE_ASSIGNMENT" ea
    WHERE ea.deleted_at IS NULL
      AND ea.effective_from <= ?1
      AND (ea.effective_to IS NULL
           OR ea.effective_to >= ?1)
)
SELECT ua.user_id, ua.employee_id, ua.active, ua.version,
       a.organization_id, a.position_id,
       coalesce((
           SELECT json_group_array(role_code)
           FROM (
               SELECT DISTINCT r.role_code
               FROM "USER_ROLE" ur
               JOIN "ROLE" r
                 ON r.role_id = ur.role_id
                AND r.active = 1
                AND r.deleted_at IS NULL
               WHERE ur.user_id = ua.user_id
                 AND ur.deleted_at IS NULL
                 AND ur.effective_from <= ?2
                 AND (ur.effective_to IS NULL
                      OR ur.effective_to >= ?2)
               ORDER BY r.role_code
           )
       ), json('[]')) AS role_codes_json
FROM "USER_ACCOUNT" ua
LEFT JOIN "EMPLOYEE" e
  ON e.employee_id = ua.employee_id
 AND e.deleted_at IS NULL
LEFT JOIN ranked_assignment a
  ON a.employee_id = ua.employee_id AND a.rn = 1
WHERE (
      (?3 IS NOT NULL AND ?4 IS NULL
       AND ua.identity_subject = ?3)
      OR
      (?3 IS NULL AND ?4 IS NOT NULL
       AND ua.user_id = ?4)
  )
  AND ua.active = 1
  AND ua.deleted_at IS NULL
  AND (
      ua.employee_id IS NULL
      OR (
          e.employee_id IS NOT NULL
          AND e.status = 'ACTIVE'
          AND (e.retirement_date IS NULL OR ?5 < e.retirement_date)
      )
  );
```

## 8.34 SQL-032 出力用社員検索

| 項目 | 内容 |
|---|---|
| 目的 | 閲覧範囲と検索条件に一致する社員を業務出力用の安定順で取得する |
| 入力 | `employee_number TEXT`（NULL可）、`name TEXT`（NULL可）、`organization_id TEXT`（NULL可）、`position_id TEXT`（NULL可）、`status TEXT`（NULL可）、`scope_type TEXT`（必須）、`actor_employee_id TEXT`（SELF時必須）、`allowed_organization_ids_json TEXT（JSON文字列配列）`（必須、対象なしは空配列）、`as_of_date TEXT（YYYY-MM-DD）`（必須）、`export_limit INTEGER`（1以上10000以下） |
| 固定バインド順 | `?1` = `as_of_date`、`?2` = `employee_number`、`?3` = `name`、`?4` = `organization_id`、`?5` = `position_id`、`?6` = `status`、`?7` = `scope_type`、`?8` = `actor_employee_id`、`?9` = `allowed_organization_ids_json`、`?10` = `export_limit` |
| 出力 | 社員番号、姓名、姓名カナ、メール、電話番号、入社日、退職日、雇用区分、状態、組織コード・名称、役職コード・名称を0件以上、最大`export_limit + 1`件 |
| 0件時 | 0行。出力候補なし。 |
| ソート・ページング | 社員番号、社員IDの昇順。`LIMIT (export_limit + 1)`で最大10001件。 |
| 利用インデックス | IDX-001, IDX-006, IDX-007, IDX-021 |
| ロック・競合 | 読取ロックなし。閲覧範囲条件を必須適用する。 |
| 注意 | `as_of_date`にはAPI入力でなくM-002が確定した業務日だけを設定する。10001件取得時はM-002が413相当とし、ファイルを生成しない。検索語はリテラル値として`instr()`で評価し、出力順・整形はM-002が行う。 |

```sql
WITH ranked_assignment AS (
    SELECT ea.employee_id, ea.organization_id, ea.position_id,
           row_number() OVER (
               PARTITION BY ea.employee_id
               ORDER BY ea.effective_from DESC, ea.assignment_id
           ) AS rn
    FROM "EMPLOYEE_ASSIGNMENT" ea
    WHERE ea.deleted_at IS NULL
      AND ea.effective_from <= ?1
      AND (ea.effective_to IS NULL OR ea.effective_to >= ?1)
)
SELECT e.employee_id, e.employee_number,
       e.last_name, e.first_name, e.last_name_kana, e.first_name_kana,
       e.email, e.phone_number, e.hire_date, e.retirement_date,
       e.employment_type_code, e.status,
       a.organization_id, o.organization_code, o.organization_name,
       a.position_id, p.position_code, p.position_name
FROM "EMPLOYEE" e
LEFT JOIN ranked_assignment a
  ON a.employee_id = e.employee_id AND a.rn = 1
LEFT JOIN "ORGANIZATION" o
  ON o.organization_id = a.organization_id AND o.deleted_at IS NULL
LEFT JOIN "POSITION" p
  ON p.position_id = a.position_id AND p.deleted_at IS NULL
WHERE e.deleted_at IS NULL
  AND (?2 IS NULL
       OR instr(lower(e.employee_number), lower(?2)) > 0)
  AND (?3 IS NULL
       OR instr(lower(e.last_name || ' ' || e.first_name), lower(?3)) > 0)
  AND (?4 IS NULL OR a.organization_id = ?4)
  AND (?5 IS NULL OR a.position_id = ?5)
  AND (?6 IS NULL OR e.status = ?6)
  AND (
      ?7 = 'ALL'
      OR (?7 = 'SELF' AND e.employee_id = ?8)
      OR (
          ?7 = 'ORGANIZATION'
          AND EXISTS (
              SELECT 1
              FROM json_each(?9) j
              WHERE CAST(j.value AS TEXT) = a.organization_id
          )
      )
  )
ORDER BY e.employee_number, e.employee_id
LIMIT (?10 + 1);
```
