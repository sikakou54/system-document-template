## 12.2 データアクセス境界

- オンライン経路は「SCR-006 → M-001 → Cloudflare Workers `fetch` → API-006 → M-002/IF-06」、定期経路は「Cloudflare Cron Trigger → Worker `scheduled`ハンドラー → Cloudflare Queues → Worker `queue`ハンドラー → JOB-001 → M-002/IF-07」とする。`scheduled`は初回投入だけ、`queue`は1メッセージ・最大40社員の1チャンクだけを担当する。
- API-006とJOB-001はD1 Binding `env.DB`、Prepared Statement、`batch()`、TBL、SQL-ID、トランザクションを扱わない。両者からD1・SQLへの直接アクセスは禁止する。
- M-002、M-003、M-004、M-007は論理的な公開IFで連携し、DBデータが必要な場合はM-006の公開IFへ委譲する。
- composition rootだけがWorker環境からD1 Bindingを取り出してM-006実装を構築し、M-006だけが`env.DB.prepare(...).bind(...)`と`env.DB.batch(...)`によりSQL-003、SQL-007、SQL-009〜SQL-013、SQL-031を実行する。
- 更新処理はM-002が論理境界を宣言し、M-006が社員versionガードと業務条件を含むD1 mutation batchを全成功または全ロールバックで実行する。即時再試行はCloudflareが再試行推奨とする明示的エラーだけを対象にfull jitter付きで初回+最大2回とし、書込み結果不明時はM-006経由で状態・versionを再確認する。過負荷・タイムアウト・CPU・メモリ・実行文予算超過は即時再試行せず、定期経路ではQueue再配信へ委ねる。
- `queue` invocationはD1実行文数900以下（Paid上限1000）、対象40社員以下、保守的最悪841文とする。後続ありではM-002がJOB-001経由で返す退職日・社員IDカーソルをqueueハンドラーが次メッセージへ引き継ぎ、1件・並列1・再試行3回、失敗後はDLQとする。
- BR-001〜BR-008の全経路は9章、実装基盤・運用環境・容量・試験実装の引継ぎは11章を正本とする。
- NFR-001のログアウト経路は「共通ヘッダー／認証失効 → UC-013(F-013) → M-001/IF-04 → SCR-010」で完結し、API・JOB・M-006・SQL・D1を使用しない。

## 12.3 認可済み参照条件の伝播

- 社員検索は「SCR-001 → M-001 → API-001 → M-002/IF-01 → M-003/IF-03 → M-006/IF-26・IF-16 → M-003/IF-03 → M-002/IF-01 → M-006/IF-01・IF-02」とする。社員詳細は同じ認可往復後にM-002/IF-02がM-006/IF-03を実スコープ条件付きで1回呼び、0件は閲覧範囲外の存在を秘匿して不存在として扱う。
- M-003/IF-03は[共通区分定義](02_機能要件.md#25-共通区分定義)の固定ロールと[ユースケース](02_機能要件.md#23-ユースケース)の各UC認可定義から`ALL` / `ORGANIZATION` / `SELF`、操作者社員ID、許可組織ID集合、認可基準日、返却可能項目を確定する。社員検索・詳細・更新・出力の認可基準日は呼出時点のAsia/Tokyo業務日に固定し、部門管理者の許可組織集合はその日の主所属組織と全子孫である。組織・役職候補取得の`effectiveOn`を過去・未来の認可範囲へ流用しない。
- M-002は認可結果を縮小してよいが拡張せず、`scopeType`、操作者社員ID、許可組織ID集合、認可基準日をM-006へすべて渡す。M-006は8章の固定マッピングで論理項目をD1対応の順序付きパラメータ`?NNN`へ変換して`.bind(...)`し、独自に認可範囲を追加しない。
- APIやJOBは認可用データを取得するためにもM-006を直接呼ばない。M-006/IF-26・IF-16はM-003からの認可材料取得、M-006/IF-01〜IF-03はM-002からの業務データ取得であり、呼出主体と目的を分離する。

## 12.4 D1原子実行の落とし込み

~~~text
業務原子境界    M-002/IF-07 が TX-005 を宣言
        ▼ 1つの論理commandを1回呼出
データアクセス  M-006/IF-13 退職確定
        ▼ 全Statementを事前にprepare・bind
クエリ順        SQL-013-A #1 → SQL-013-B #2 → SQL-010 #3 → SQL-007 #4 → SQL-013-C #5
        ▼ env.DB.batch([...]) を1回実行
D1              全Statement成功時だけ確定 / 1件失敗時はbatch全体ロールバック
~~~

- 1つのTX-005を複数M-006公開IFの逐次呼出で組み立てず、M-002/IF-07がM-006/IF-13を社員1件につき1回だけ呼ぶ。[データベース設計](04_データベース設計.md#4-データベース設計)、[モジュール設計](07_モジュール設計.md#7-モジュール設計)、[SQL-013 退職確定](08_クエリ設計.md#815-sql-013-退職確定)、[要求・設計トレーサビリティ](09_トレーサビリティ.md#9-要求設計トレーサビリティ)でM-006 IF、SQL順、成功条件、全体失敗条件を一致させる。
- batch成功後の上位判定でロールバックできると仮定せず、社員version・在籍状態・退職日範囲・上長逆参照0件・期待所属ID集合一致の不可分条件は、SQL-013-Bのversion・status・hire_date・business_date条件付きDML、SQL-013-AのTX_MUTATION_GUARD参照とトリガーによる期待ID集合検証、EMPLOYEE_ASSIGNMENTの制約・期間トリガー、または同一batch内の失敗可能Statementで保証する。

## 12.6 Cron・Queue・JOB継続の落とし込み

~~~text
Cron Trigger      `15 15 * * *`（UTC 15:15 = Asia/Tokyo 翌日00:15）
        ▼ scheduled() は初回messageだけを生成・投入
Cloudflare Queue  cursor=null / chainRunId=JOB-001:<businessDate>:<scheduledTime> / chunkNo=1 / businessDate
        ▼ max_batch_size=1・max_concurrency=1
queueハンドラー   JOB-001 退職日到来反映 を1 messageにつき1回呼出
        ▼ JOB本体が業務処理を委譲
JOB-001           M-002/IF-07 を1回呼出
        ▼ maxItems=40・statementBudget=900
M-002/IF-07       対象・成功・スキップ・失敗件数・次cursor・hasMore・Statement実測数
        ▼ 永続化はM-006公開IFだけ
M-006 → D1        Paid hard limit 1,000 / invocation・保守的最悪841文
        ▼ 続きあり: 同じchainRunId/businessDate、chunkNo+1、次cursor
Cloudflare Queue  次message投入確定後に現messageをack
~~~

| 障害 | 落とし込み |
|---|---|
| retryable allowlist一致 | M-006内で[処理詳細](06_JOB設計.md#635-処理詳細)の5エラーだけを上限付きexponential backoff + full jitterで初回後最大2回。未解消はQueue redelivery |
| 書込み結果不明 | M-006が社員状態・versionを再読込みし、確定状態を判定してから再試行可否を決める |
| 過負荷 / timeout / CPU / memory / 実行文予算超過 | 同一invocationで即時再試行せず`DATA_ACCESS_ERROR`（retryable）でQueue redelivery |
| 再配信上限 | `max_retries=3`、その後DLQ。復旧はbusinessDateと冪等キーを維持しメッセージ本文を変更せず再投入 |
| 重複 | `chainRunId + chunkNo`、社員version、未反映条件で無害化 |

[JOB-001 退職日到来反映](06_JOB設計.md#63-job-001-退職日到来反映)、[IF-07 退職日到来反映](07_モジュール設計.md#769-if-07-退職日到来反映)、[要求・設計トレーサビリティ](09_トレーサビリティ.md#9-要求設計トレーサビリティ)、[設計レビュー用チェックリスト](10_レビューチェックリスト.md#10-設計レビュー用チェックリスト)、[詳細設計への引継ぎ事項](11_詳細設計への引継ぎ.md#11-詳細設計への引継ぎ事項)と本節で、メッセージ項目、40件、900/1,000 Statement、Queue設定、retry/DLQ規則を一致させる。
