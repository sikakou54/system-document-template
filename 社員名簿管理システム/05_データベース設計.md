# 5. データベース設計

## 5.1 採用DBMS・物理設計方針

業務データの論理構造(エンティティ・属性・関連)は§2.3 データモデル、シーケンス上のDB要素名と接続境界は§3.1の単一participant「Cloudflare D1」を正本とし、本章はそれらのSQLite物理構造を詳細化する。M-006以外からD1へ到達する経路を追加しない。

| 項目 | 設計 |
|---|---|
| DBMS / SQL方言 | Cloudflare D1 / SQLite互換方言 |
| 実行環境 | Cloudflare Workers PaidのD1 Binding。SQL実行はM-006 D1データアクセスに限定する |
| テーブル型制約 | EMPLOYEE〜AUDIT_LOGの通常テーブルとTX_MUTATION_GUARD技術テーブルはすべて`CREATE TABLE ... STRICT`で作成する。任意導入のFTS5 virtual tableだけは`STRICT`対象外 |
| 検索機能 | 初期正本はSQLite組込み関数の`instr(lower(対象列), lower(term)) > 0`を使用する。FTS5 `trigram`は性能試験で移行基準を超えた場合だけ採用する任意候補とし、初期マイグレーションには含めない |
| 文字コード | UTF-8 |
| タイムゾーン | DB保存はUTC、業務日判定と画面表示はAsia/Tokyo。DB側の現在時刻関数を保存値に直接使用せず、M-006がUTCのISO 8601値をバインドする |
| 日付 | `TEXT`。`YYYY-MM-DD`のISO形式で保存し、文字列順を日付順として利用する。有効期間の開始日・終了日は両端を含む |
| 日時 | `TEXT`。UTCの`YYYY-MM-DDTHH:mm:ss.SSSZ`形式で保存する |
| 主キー | UUIDの正準表現を`TEXT`で保存する。Workers側のモジュールで採番して登録する |
| 真偽値 | `INTEGER NOT NULL CHECK (値 IN (0, 1))`。`1`を真、`0`を偽とする |
| JSON | `TEXT`で保存し、`json_valid()`と用途別`json_type()`をCHECKする。業務概要は`object`、更新ガードのID集合は`array`とする |
| 楽観ロック | 更新対象の`version`一致をWHERE条件に含め、成功時に1加算する |
| 論理削除 | マスター・利用者等は`deleted_at IS NULL`を有効行条件とする。社員は在籍状態で管理し物理削除しない |
| マスター利用可否 | ORGANIZATION・POSITIONの`active`は有効期間と独立した手動利用可否。指定日利用可は`active = 1 AND effective_from <= 指定日 AND (effective_to IS NULL OR 指定日 <= effective_to)`。未来開始前・期限到来後でも`active = 1`は許容する |
| 更新単位 | 参照は`first()`/`all()`、単一文更新は`run()`、複数文の業務更新はM-006がprepared statementを`D1Database.batch()`へ順番どおりに渡す。D1は全statement成功時だけ確定し、statement例外時はバッチ全体を取り消す |
| 競合制御 | D1単一DBの書込み逐次性を前提とし、分離レベル指定やロック構文は使用しない。所有者`version`ガード、DB制約、検証トリガー、更新対象ID集合の完全一致検証で事前参照後の状態変化を検出する |
| 再試行 | 即時再試行対象は§3.1.2および§7.3.5に完全一致で列挙した5エラーだけとし、指数バックオフ＋full jitterで初回後最大2回再実行する。書込結果不明は状態・versionを再読込してcommit確認後、未反映だけ再試行する。overload、timeout、CPU・memory超過は即時再試行しない。M-006は論理一時障害を返し、M-002がオンライン要求を503系、JOBをQueue再配信へ振り分ける |
| 実行予算 | Workers Paid、D1 Paid上限1,000 statement/Worker invocationに対し内部上限900、ordered placeholderは1 statement最大100個。M-006が実測statement数をM-002へ返す |
| JOB基盤 | Cron scheduledハンドラーはCloudflare Queuesへ初回メッセージを投入するだけとし、queueハンドラーが1メッセージにつきJOB-001を1回、JOB-001がM-002/IF-07を1回呼ぶ。`max_batch_size=1`、`max_concurrency=1`、`max_retries=3`、超過時はDLQへ送る |
| DBアクセス境界 | M-006 D1データアクセスだけがSQLを実行する。API・JOB・他モジュールによる直接アクセスは禁止する |

### 期間整合性

- 同一社員の有効期間は重複させない。
- 基準日時点の所属は`effective_from <= 基準日 AND (effective_to IS NULL OR effective_to >= 基準日)`で判定する。
- `effective_to IS NULL`は無期限の最終行を表し、将来開始の予約所属を含めて社員ごとに最大1件とする。
- 異動可否判定では基準日に有効な行だけでなく、基準日以降に終了していない現在・将来行を取得して期間重複を検証する。
- 異動時は異動日の直前に有効な所属を異動日前日で終了し、異動日以降に開始する既存の将来所属を論理削除してから新所属を登録する。これを指定日以降の将来予約置換の正本とする。
- 即時・到来退職時は退職日に有効な所属を退職日で終了し、退職日より後に開始する将来所属を論理削除する。
- 上長社員の退職日当日以後も有効となる`manager_employee_id`参照が1件でもある場合、その退職予定・確定を拒否し、部下の上長変更を先に行う。
- 所属が参照する組織・役職は所属期間全体で利用可能でなければならない。マスター無効化・期間短縮は、現在・将来所属の参照期間を外す場合に拒否する。
- 終了日がある場合は`effective_from <= effective_to`を満たす。

| role_code | role_name | 初期active |
|---|---|---|
| `HR` | 人事担当者 | 1 |
| `DEPARTMENT_MANAGER` | 部門管理者 | 1 |
| `EMPLOYEE` | 一般社員 | 1 |
| `SYSTEM_ADMIN` | システム管理者 | 1 |

| IDX-022 | ORGANIZATION(parent_organization_id) WHERE deleted_at IS NULL | PARTIAL INDEX | 階層循環・子組織包含検証 |

- 部分インデックスを利用するSQLは、対象インデックスと同じ論理削除・状態条件を検索条件へ含める。
- SQLiteの`TEXT`長は型名では制限されないため、最大長を持つ列はすべて`CHECK (length(列) BETWEEN 1 AND 最大長)`を持つ。NULL可の列は`列 IS NULL OR ...`とする。
- 外部キーはマイグレーションで明示し、D1で有効な外部キー検証を利用する。削除は`RESTRICT`、主キー値の更新は禁止する。

### 5.12.1 型・チェック制約

- EMPLOYEE〜AUDIT_LOGおよびTX_MUTATION_GUARDの物理DDLはすべて末尾を`) STRICT;`とする。SQLiteの`INTEGER` / `REAL` / `TEXT` / `BLOB`以外を本システムの物理型に使用せず、FTS5 virtual tableだけを例外とする。
- UUIDを保持する`TEXT`列は、`length(列) = 36`、8-4-4-4-12位置のハイフン、英小文字の16進文字だけを許可するCHECKを持つ。
- 日付`TEXT`列は`length(列) = 10`、`YYYY-MM-DD`の桁・区切り、および`strftime('%Y-%m-%d', 列) = 列`を満たすCHECKを持つ。NULL可の列はNULLを許可する。
- 日時`TEXT`列はUTCの`YYYY-MM-DDTHH:mm:ss.SSSZ`を満たすCHECKを持つ。M-006は同じ形式へ正規化した値だけをバインドする。
- 真偽列は`INTEGER CHECK (列 IN (0, 1))`とする。
- JSON列は`TEXT CHECK (json_valid(列))`を必須とし、change_summary・detail_summaryは`json_type(列) = 'object'`、更新ガードのID集合は`json_type(列) = 'array'`を追加する。
- EMPLOYEEは`retirement_date IS NULL OR hire_date <= retirement_date`を満たす。
- EMPLOYEEの`status = 'RETIRED'`では`retirement_date IS NOT NULL`を満たす。
- EMPLOYEEの`status`は`ACTIVE`または`RETIRED`のいずれかを満たす。
- EMPLOYEEは`employment_type_code IN ('REGULAR', 'CONTRACT', 'PART_TIME', 'TEMPORARY', 'OTHER')`を満たす。
- EMPLOYEEは`retirement_type_code IS NULL OR retirement_type_code IN ('VOLUNTARY', 'COMPANY', 'RETIREMENT_AGE', 'CONTRACT_END', 'OTHER')`を満たす。
- 各有効期間は終了日がある場合に`effective_from <= effective_to`を満たす。ISO日付`TEXT`のため文字列比較を使用する。
- EMPLOYEE_ASSIGNMENTは`manager_employee_id IS NULL OR manager_employee_id <> employee_id`を満たす。
- POSITIONの`position_level`は0以上とする。
- ROLEは`role_code IN ('HR', 'DEPARTMENT_MANAGER', 'EMPLOYEE', 'SYSTEM_ADMIN')`を満たす。
- EMPLOYEE_CHANGE_HISTORYの`change_type`は`REGISTER`、`BASIC_UPDATE`、`ASSIGNMENT`、`RETIREMENT_SCHEDULED`、`RETIREMENT_CONFIRMED`、`ROLE_UPDATE`のいずれか、`source_type`は`API`または`JOB`のいずれかとする。
- AUDIT_LOGの`result`は`SUCCESS`または`FAILURE`のいずれか、`source_type`は`API`、`JOB`、`AUTH`のいずれかとする。

### 5.12.2 部分一致検索とFTS5移行候補

初期正本SQLは社員番号と`last_name || ' ' || first_name`に対する`instr(lower(対象), lower(term)) > 0`である。ワイルドカード構文を使用しないため、入力値の`%`、`_`、`\`を検索演算子として解釈しない。D1の通常索引では任意位置の部分一致を高速化できないため、データ量・実測時間が非機能要件の移行基準を超えた場合に限りFTS5へ移行する。

FTS5へ移行する場合は、D1が提供するSQLite FTS5の`trigram` tokenizerを使い、次の任意マイグレーションを適用する。`employee_id`は検索語として索引化しない。このDDLは初期マイグレーションおよび初期正本SQLには含めない。

```sql
CREATE VIRTUAL TABLE "EMPLOYEE_SEARCH_FTS" USING fts5(
    employee_id UNINDEXED,
    employee_number,
    full_name,
    tokenize = 'trigram case_sensitive 0'
);

CREATE TRIGGER "trg_employee_search_fts_ai"
AFTER INSERT ON "EMPLOYEE"
WHEN NEW.deleted_at IS NULL
BEGIN
    INSERT INTO "EMPLOYEE_SEARCH_FTS" (employee_id, employee_number, full_name)
    VALUES (NEW.employee_id, NEW.employee_number, NEW.last_name || ' ' || NEW.first_name);
END;

CREATE TRIGGER "trg_employee_search_fts_au"
AFTER UPDATE OF employee_number, last_name, first_name, deleted_at ON "EMPLOYEE"
BEGIN
    DELETE FROM "EMPLOYEE_SEARCH_FTS" WHERE employee_id = OLD.employee_id;
    INSERT INTO "EMPLOYEE_SEARCH_FTS" (employee_id, employee_number, full_name)
    SELECT NEW.employee_id, NEW.employee_number, NEW.last_name || ' ' || NEW.first_name
    WHERE NEW.deleted_at IS NULL;
END;

CREATE TRIGGER "trg_employee_search_fts_ad"
AFTER DELETE ON "EMPLOYEE"
BEGIN
    DELETE FROM "EMPLOYEE_SEARCH_FTS" WHERE employee_id = OLD.employee_id;
END;
```

FTS5採用時は正規化後3文字以上をFTS5、1～2文字を初期正本の`instr()`で検索する。切替時は有効なEMPLOYEEから全件投入し、両方式の検索結果一致を検証してからFTS5経路を開放する。本番運用でEMPLOYEEの物理削除は行わないが、マイグレーション・復旧時の索引残留を避けるためDELETE同期トリガーも定義する。

### 5.12.3 期間重複防止トリガー

終了日NULLを無限未来とみなし、両端包含で重複を判定する。D1ではINSERT用とUPDATE用を分け、該当時は`RAISE(ABORT, エラーコード)`で同一バッチ全体を失敗させる。

```sql
CREATE TRIGGER "trg_assignment_period_bi"
BEFORE INSERT ON "EMPLOYEE_ASSIGNMENT"
WHEN NEW.deleted_at IS NULL AND EXISTS (
    SELECT 1
    FROM "EMPLOYEE_ASSIGNMENT" a
    WHERE a.employee_id = NEW.employee_id
      AND a.deleted_at IS NULL
      AND a.effective_from <= coalesce(NEW.effective_to, '9999-12-31')
      AND NEW.effective_from <= coalesce(a.effective_to, '9999-12-31')
)
BEGIN
    SELECT RAISE(ABORT, 'ASSIGNMENT_PERIOD_OVERLAP');
END;

CREATE TRIGGER "trg_assignment_period_bu"
BEFORE UPDATE OF employee_id, effective_from, effective_to, deleted_at
ON "EMPLOYEE_ASSIGNMENT"
WHEN NEW.deleted_at IS NULL AND EXISTS (
    SELECT 1
    FROM "EMPLOYEE_ASSIGNMENT" a
    WHERE a.employee_id = NEW.employee_id
      AND a.assignment_id <> NEW.assignment_id
      AND a.deleted_at IS NULL
      AND a.effective_from <= coalesce(NEW.effective_to, '9999-12-31')
      AND NEW.effective_from <= coalesce(a.effective_to, '9999-12-31')
)
BEGIN
    SELECT RAISE(ABORT, 'ASSIGNMENT_PERIOD_OVERLAP');
END;

CREATE TRIGGER "trg_user_role_period_bi"
BEFORE INSERT ON "USER_ROLE"
```

### 5.12.4 業務整合性トリガー

アプリケーションでの事前検証に加え、次のトリガーを最終防衛としてマイグレーションに定義する。いずれも違反時は固定エラーコードで`RAISE(ABORT, ...)`し、M-006が業務エラーへ変換する。

| Trigger-ID | タイミング・対象 | 再検証内容 | DBエラーコード |
|---|---|---|---|
| TRG-001 | EMPLOYEE_ASSIGNMENT INSERT/UPDATE | 組織・役職が`active = 1`で所属期間全体を包含すること、上長が本人以外で在籍し、上長退職日の前日までに所属が終了すること | `ASSIGNMENT_REFERENCE_INVALID` |
| TRG-002 | EMPLOYEE UPDATE retirement_date/status | 退職日当日以後に有効な部下からの上長参照がないこと | `MANAGER_REFERENCE_EXISTS` |
| TRG-003 | ORGANIZATION INSERT/UPDATE | 再帰CTEで循環がないこと、親が有効で子期間を包含すること、有効な子を孤児化しないこと、変更後条件から外れる所属参照がないこと | `ORGANIZATION_REFERENCE_INVALID` |
| TRG-004 | POSITION UPDATE | 無効化・期間短縮後の条件から外れる所属参照がないこと | `POSITION_REFERENCE_INVALID` |
| TRG-005 | USER_ROLE INSERT/UPDATE | ROLEとUSER_ACCOUNTが削除されておらず、ROLEが`active = 1`であること | `USER_ROLE_REFERENCE_INVALID` |

TRG-001とTRG-005は論理削除後の行を参照検証対象外とする。TRG-002～004は現在・将来の未削除参照を再検索し、退職、無効化、期間短縮による不整合も拒否する。トリガー内の日付比較は正規化済みISO日付`TEXT`の比較を使用する。DBエラーだけを正常系の分岐手段にはせず、M-002の事前検証結果と一致することを試験する。

次をTRG-001〜005のマイグレーション正本とする。Trigger-IDは論理ID、物理トリガー名の`_bi`/`_bu`はINSERT/UPDATE別実装を表す。

```sql
              NEW.effective_to IS NOT NULL
```

## 5.13 トランザクション境界

| TX-ID | 処理 | 同一トランザクションの更新 | 失敗時 |
|---|---|---|---|
| TX-001 | 社員登録 | 重複・組織/役職有効性の再検証、EMPLOYEE登録、EMPLOYEE_ASSIGNMENT登録、EMPLOYEE_CHANGE_HISTORY登録を一つのD1バッチで実行 | バッチ全体を取消 |
| TX-002 | 社員基本情報更新 | 所有者版数ガード、EMPLOYEE更新、EMPLOYEE_CHANGE_HISTORY登録、ガード削除を一つのD1バッチで実行 | バッチ全体を取消 |
| TX-003 | 社員異動 | 所有者版数・更新集合ガードINSERT、社員版数更新、直前所属終了・将来所属取消、新所属登録、EMPLOYEE_CHANGE_HISTORY登録、ガード削除を一つのD1バッチで実行 | バッチ全体を取消 |
| TX-004 | 未来日退職受付 | 所有者版数ガード、上長逆参照のDB再検証、EMPLOYEEの退職日・退職区分・版数更新、EMPLOYEE_CHANGE_HISTORY登録、ガード削除を一つのD1バッチで実行 | バッチ全体を取消 |
| TX-005 | 即時・到来退職反映 | 所有者版数・更新集合ガードINSERT、退職確定、退職日に有効な所属終了、後続将来所属取消、EMPLOYEE_CHANGE_HISTORY登録、ガード削除を社員1件ごとのD1バッチで実行 | 当該社員のバッチ全体を取消 |
| TX-006 | 組織登録・更新 | 更新時の所有者版数ガード、親子期間・循環・所属参照のDB再検証、ORGANIZATION登録または条件付き更新、ガード削除を一つのD1バッチで実行 | バッチ全体を取消 |
| TX-007 | 役職登録・更新 | 更新時の所有者版数ガード、所属参照のDB再検証、POSITION登録または条件付き更新、ガード削除を一つのD1バッチで実行 | バッチ全体を取消 |
| TX-008 | ロール割当更新 | 所有者版数・更新集合ガードINSERT、USER_ACCOUNT版数更新、現行USER_ROLE終了、新USER_ROLE登録、EMPLOYEE_CHANGE_HISTORY登録、ガード削除を一つのD1バッチで実行 | バッチ全体を取消 |

- M-006はバッチ内statementを表の記載順にprepared statementとして配列化する。D1は同一DB内で順次実行し、全statement成功時だけ確定する。M-006からSQLの`BEGIN`、`COMMIT`、`ROLLBACK`は発行しない。
- AUDIT_LOGは業務更新の成功後に独立した単一文`run()`で追記する。
- AUDIT_LOG記録に失敗した場合、業務更新は巻き戻さず、M-007が監査記録失敗を返して運用アラート対象とする。
- 即時再試行は§3.1.2および§7.3.5に完全一致で列挙した5エラーだけを対象に、指数バックオフ＋full jitterで初回後最大2回とする。実行時・デプロイ時の追加は許可しない。書込結果不明は状態・versionをM-006経由で再参照してcommit確認後、未反映だけ新しい`guard_id`で再試行する。overload、timeout、CPU・memory超過は即時再試行せず、M-002がオンラインは503系の一時障害、JOBだけQueue再配信へ変換する。制約・トリガー・版数・集合競合は再試行しない。
- JOB-001はSQL-012を`limit + 1`で最大41件取得し、先頭40名だけを1 Worker invocationの1チャンクとして処理する。1名当たり最大21 statement、ページ取得1 statementの最悪値は`21 × 40 + 1 = 841`で、内部上限900およびD1 Paid上限1,000を下回る。41件目は処理せず`hasMore`と`nextCursor`の判定にだけ使う。
- M-006は`first()`、`all()`、`run()`、`batch()`でD1へ送信したprepared statement数を実測し、`statementCount`（互換名`queryCount`）としてM-002へ返す。900到達前に新たな社員処理を開始しない。

### 5.13.1 所有者version・更新集合ガード

事前SELECTと更新バッチの間に別要求が確定しても、古い判定結果で更新しないよう、次の技術テーブルを作成する。成功時は同じバッチの末尾で削除され、途中失敗時はバッチとともにINSERT自体が取り消されるため残留しない。

```sql
CREATE TABLE "TX_MUTATION_GUARD" (
    guard_id TEXT PRIMARY KEY,
    owner_type TEXT NOT NULL
        CHECK (owner_type IN ('NONE', 'EMPLOYEE', 'ORGANIZATION', 'POSITION', 'USER_ACCOUNT')),
    owner_id TEXT,
    expected_version INTEGER,
    operation_type TEXT NOT NULL
        CHECK (operation_type IN ('NONE', 'ASSIGNMENT_TRANSFER', 'ASSIGNMENT_RETIREMENT', 'USER_ROLE_REPLACE')),
    pivot_date TEXT,
    expected_target_ids_json TEXT NOT NULL
        CHECK (json_valid(expected_target_ids_json) AND json_type(expected_target_ids_json) = 'array'),
    retained_ids_json TEXT NOT NULL DEFAULT '[]'
        CHECK (json_valid(retained_ids_json) AND json_type(retained_ids_json) = 'array'),
    created_at TEXT NOT NULL,
    CHECK (
        (owner_type = 'NONE' AND owner_id IS NULL AND expected_version IS NULL)
        OR
        (owner_type <> 'NONE' AND owner_id IS NOT NULL AND expected_version >= 1)
    ),
    CHECK (
        (operation_type = 'NONE' AND pivot_date IS NULL)
        OR
        (operation_type <> 'NONE' AND pivot_date IS NOT NULL)
    )
) STRICT;

CREATE TRIGGER "trg_mutation_guard_owner_bi"
BEFORE INSERT ON "TX_MUTATION_GUARD"
BEGIN
    SELECT CASE
        WHEN NEW.owner_type = 'EMPLOYEE' AND NOT EXISTS (
            SELECT 1 FROM "EMPLOYEE"
            WHERE employee_id = NEW.owner_id
              AND version = NEW.expected_version
              AND deleted_at IS NULL
        ) THEN RAISE(ABORT, 'OWNER_VERSION_CONFLICT')
        WHEN NEW.owner_type = 'ORGANIZATION' AND NOT EXISTS (
            SELECT 1 FROM "ORGANIZATION"
            WHERE organization_id = NEW.owner_id
              AND version = NEW.expected_version
              AND deleted_at IS NULL
        ) THEN RAISE(ABORT, 'OWNER_VERSION_CONFLICT')
        WHEN NEW.owner_type = 'POSITION' AND NOT EXISTS (
            SELECT 1 FROM "POSITION"
            WHERE position_id = NEW.owner_id
              AND version = NEW.expected_version
              AND deleted_at IS NULL
        ) THEN RAISE(ABORT, 'OWNER_VERSION_CONFLICT')
        WHEN NEW.owner_type = 'USER_ACCOUNT' AND NOT EXISTS (
            SELECT 1 FROM "USER_ACCOUNT"
            WHERE user_id = NEW.owner_id
              AND version = NEW.expected_version
              AND deleted_at IS NULL
        ) THEN RAISE(ABORT, 'OWNER_VERSION_CONFLICT')
    END;
    SELECT CASE
        WHEN EXISTS (
            SELECT 1 FROM json_each(NEW.expected_target_ids_json) WHERE type <> 'text'
        ) OR (
            SELECT count(*) FROM json_each(NEW.expected_target_ids_json)
        ) <> (
            SELECT count(DISTINCT value) FROM json_each(NEW.expected_target_ids_json)
        ) THEN RAISE(ABORT, 'EXPECTED_SET_INVALID')
    END;
    SELECT CASE
        WHEN NEW.operation_type IN ('ASSIGNMENT_TRANSFER', 'ASSIGNMENT_RETIREMENT')
         AND NEW.owner_type <> 'EMPLOYEE'
          THEN RAISE(ABORT, 'EXPECTED_SET_INVALID')
        WHEN NEW.operation_type = 'USER_ROLE_REPLACE'
         AND NEW.owner_type <> 'USER_ACCOUNT'
          THEN RAISE(ABORT, 'EXPECTED_SET_INVALID')
        WHEN NEW.operation_type = 'NONE'
         AND json_array_length(NEW.expected_target_ids_json) <> 0
          THEN RAISE(ABORT, 'EXPECTED_SET_INVALID')
    END;
END;
```

更新集合検証トリガーは、次の集合を`expected_target_ids_json`と双方向に比較し、片側に1件でも差があれば`UPDATE_SET_CONFLICT`で中止する。`USER_ROLE_REPLACE`では`retained_ids_json`も重複なしTEXT配列として同じ要領で検証する。

| operation_type | DB内で再計算する更新対象集合 |
|---|---|
| `ASSIGNMENT_TRANSFER` | `owner_id`の未削除所属のうち、`pivot_date`前日に有効な行、または`effective_from >= pivot_date`の行 |
| `ASSIGNMENT_RETIREMENT` | `owner_id`の未削除所属のうち、`pivot_date`当日に有効な行、または`effective_from > pivot_date`の行 |
| `USER_ROLE_REPLACE` | `owner_id`の未削除ロールのうち、`effective_to IS NULL OR effective_to >= pivot_date`で、`retained_ids_json`に含まれない行 |

次のD1/SQLiteトリガーで3種類の集合を実際に双方向突合する。片方向だけの件数比較ではなく、DB集合にだけ存在するIDと期待集合にだけ存在するIDの両方を検出する。

```sql
CREATE TRIGGER "trg_mutation_guard_set_bi"
BEFORE INSERT ON "TX_MUTATION_GUARD"
WHEN NEW.operation_type <> 'NONE'
BEGIN
    SELECT CASE
        WHEN EXISTS (
            SELECT 1 FROM json_each(NEW.retained_ids_json) WHERE type <> 'text'
        ) OR (
            SELECT count(*) FROM json_each(NEW.retained_ids_json)
        ) <> (
            SELECT count(DISTINCT value) FROM json_each(NEW.retained_ids_json)
        ) THEN RAISE(ABORT, 'EXPECTED_SET_INVALID')
    END;

    SELECT CASE
        WHEN NEW.operation_type = 'ASSIGNMENT_TRANSFER' AND (
            EXISTS (
                SELECT 1
                FROM "EMPLOYEE_ASSIGNMENT" a
                WHERE a.employee_id = NEW.owner_id
                  AND a.deleted_at IS NULL
                  AND (
                      (
                          a.effective_from <= date(NEW.pivot_date, '-1 day')
                          AND (a.effective_to IS NULL OR a.effective_to >= date(NEW.pivot_date, '-1 day'))
                      )
                      OR a.effective_from >= NEW.pivot_date
                  )
                  AND NOT EXISTS (
                      SELECT 1 FROM json_each(NEW.expected_target_ids_json) j
                      WHERE j.value = a.assignment_id
                  )
            )
            OR EXISTS (
                SELECT 1
                FROM json_each(NEW.expected_target_ids_json) j
                WHERE NOT EXISTS (
                    SELECT 1
                    FROM "EMPLOYEE_ASSIGNMENT" a
                    WHERE a.assignment_id = j.value
                      AND a.employee_id = NEW.owner_id
                      AND a.deleted_at IS NULL
                      AND (
                          (
                              a.effective_from <= date(NEW.pivot_date, '-1 day')
                              AND (a.effective_to IS NULL OR a.effective_to >= date(NEW.pivot_date, '-1 day'))
                          )
                          OR a.effective_from >= NEW.pivot_date
                      )
                )
            )
        ) THEN RAISE(ABORT, 'UPDATE_SET_CONFLICT')

        WHEN NEW.operation_type = 'ASSIGNMENT_RETIREMENT' AND (
            EXISTS (
                SELECT 1
                FROM "EMPLOYEE_ASSIGNMENT" a
                WHERE a.employee_id = NEW.owner_id
                  AND a.deleted_at IS NULL
                  AND (
                      (
                          a.effective_from <= NEW.pivot_date
                          AND (a.effective_to IS NULL OR a.effective_to >= NEW.pivot_date)
                      )
                      OR a.effective_from > NEW.pivot_date
                  )
                  AND NOT EXISTS (
                      SELECT 1 FROM json_each(NEW.expected_target_ids_json) j
                      WHERE j.value = a.assignment_id
                  )
            )
            OR EXISTS (
                SELECT 1
                FROM json_each(NEW.expected_target_ids_json) j
                WHERE NOT EXISTS (
                    SELECT 1
                    FROM "EMPLOYEE_ASSIGNMENT" a
                    WHERE a.assignment_id = j.value
                      AND a.employee_id = NEW.owner_id
                      AND a.deleted_at IS NULL
                      AND (
                          (
                              a.effective_from <= NEW.pivot_date
                              AND (a.effective_to IS NULL OR a.effective_to >= NEW.pivot_date)
                          )
                          OR a.effective_from > NEW.pivot_date
                      )
                )
            )
        ) THEN RAISE(ABORT, 'UPDATE_SET_CONFLICT')

        WHEN NEW.operation_type = 'USER_ROLE_REPLACE' AND (
            EXISTS (
                SELECT 1
                FROM "USER_ROLE" ur
                WHERE ur.user_id = NEW.owner_id
                  AND ur.deleted_at IS NULL
                  AND (ur.effective_to IS NULL OR ur.effective_to >= NEW.pivot_date)
                  AND NOT EXISTS (
                      SELECT 1 FROM json_each(NEW.retained_ids_json) r
                      WHERE r.value = ur.user_role_id
                  )
                  AND NOT EXISTS (
                      SELECT 1 FROM json_each(NEW.expected_target_ids_json) j
                      WHERE j.value = ur.user_role_id
                  )
            )
            OR EXISTS (
                SELECT 1
                FROM json_each(NEW.expected_target_ids_json) j
                WHERE NOT EXISTS (
                    SELECT 1
                    FROM "USER_ROLE" ur
                    WHERE ur.user_role_id = j.value
                      AND ur.user_id = NEW.owner_id
                      AND ur.deleted_at IS NULL
                      AND (ur.effective_to IS NULL OR ur.effective_to >= NEW.pivot_date)
                      AND NOT EXISTS (
                          SELECT 1 FROM json_each(NEW.retained_ids_json) r
                          WHERE r.value = ur.user_role_id
                      )
                )
            )
        ) THEN RAISE(ABORT, 'UPDATE_SET_CONFLICT')
    END;
END;
```

M-006は集合ガード成功後のUPDATE対象を広い日付条件で再指定せず、`expected_target_ids_json`を`json_each()`で展開したIDだけに限定する。所有者の版数更新は`WHERE 主キー = owner_id AND version = expected_version`で1件だけ更新する。これにより、ガード検証後からバッチ末尾までの逐次実行中に対象集合がずれない。

バッチ成功後、M-006は各更新statementの`RETURNING` ID集合と`expected_target_ids_json`を順序非依存で再比較し、所有者更新件数が1件であることも確認する。差分は通常到達不能な整合性事故として記録し、後続処理を停止する。業務上の競合は必ずバッチ内ガードで検出して全体を取り消す。

## 5.14 共通カラム

### 更新可能テーブル共通

EMPLOYEE、EMPLOYEE_ASSIGNMENT、ORGANIZATION、POSITION、USER_ACCOUNT、ROLE、USER_ROLEへ付与する。

| カラム | 論理名 | 型 | NULL | 制約・説明 |
|---|---|---|---|---|
| created_at | 登録日時 | TEXT | 不可 | UTCのISO 8601日時。登録時にM-006がバインド |

| updated_at | 更新日時 | TEXT | 不可 | UTCのISO 8601日時。更新時にM-006がバインド |
| updated_by | 更新者ID | TEXT | 可 | UUID正準表現。JOB実行時はNULL可 |

## 5.15 命名規則

| 対象 | 規則 |
|---|---|
| テーブル | 英大文字スネークケース、単数形。履歴は`_HISTORY`、ログは`_LOG` |
| カラム | 英小文字スネークケース |
| 主キー | `{entity}_id` |
| 外部キー | 参照先の主キー名。用途が異なる自己参照は用途を接頭する |
| 制約 | `pk_` / `fk_` / `uk_` / `ck_` + 小文字テーブル名 + 用途 |
| トリガー | `trg_` + 小文字テーブル名 + 用途 + `_bi`（INSERT前）/ `_bu`（UPDATE前）/ `_ai`（INSERT後）/ `_au`（UPDATE後） |
| インデックス | `idx_` + 小文字テーブル名 + 主要列または用途 |
| バインド変数 | 09章の実SQLはD1 ordered placeholder `?1`、`?2`…だけを使用し、Prepared Statementごとの固定バインド順でコロンなし論理入力名との対応を定義する |

- SQLはテーブル名を二重引用符で囲み、本書の英大文字物理名と厳密に一致させる。
- 同じ論理パラメーターを複数箇所で使う場合も位置番号と値の対応表をPrepared Statementごとに固定し、複数構成文では各文を`?1`から欠番なく採番し直す。文字列置換でSQLを組み立てず、値はすべて`D1PreparedStatement.bind()`へ渡す。
- 文字列連結は`||`、NULL置換は`coalesce()`、日付加減算は`date(date_text, '+1 day')`または`date(date_text, '-1 day')`を使用する。
- 配列バインドは使用せず、JSON配列`TEXT`をバインドして`json_each(json_array)`で展開する。
- 現在日時はWorkers側で1要求につき一度生成し、同一バッチの全statementへ同じISO 8601文字列をバインドする。
- 列挙コードは大文字スネークケースとし、意味は機能要件・共通定義を正本とする。

## 5.16 D1 migration・検証引継ぎ

DDL・固定データ・トリガーはD1 migrationとして版管理し、Workers起動時やAPI・JOBから実行しない。migrationの適用は承認済みデプロイ工程だけが行い、アプリケーション実行時のDBアクセスは引き続きM-006だけに限定する。

| Migration ID | 適用順 | 変更内容 | 適用後検証・期待値 | 復旧・前進修正方針 |
|---|---:|---|---|---|
| MIG-001 | 1 | EMPLOYEE〜AUDIT_LOGおよびTX_MUTATION_GUARDを外部キー・CHECK・UNIQUE付き`STRICT` tableとして作成する | `pragma_table_list`で対象10表の`strict = 1`、`PRAGMA foreign_key_check`が0行 | 未公開環境はmigration取消し。本番適用後は破壊的rollbackを行わず前進修正 |
| MIG-002 | 2 | IDX-001〜IDX-022、TRG-001〜TRG-005の物理トリガー、更新集合ガードトリガーを作成し、ROLE固定4件を冪等投入する | index/triggerの件数と名称が本章に一致、ROLEが4件、全negative testが固定エラーで拒否 | 欠落定義を追加migrationで補完し、適用済みDDLを手作業変更しない |
| MIG-003 | 任意 | 性能移行基準を満たした場合だけEMPLOYEE_SEARCH_FTSと同期トリガーを作成して再構築する | 有効EMPLOYEE件数とFTS件数が一致し、代表検索結果がSQL-001/032の初期正本と一致 | 読取経路を初期`instr()`へ戻した後、追加migrationでFTS定義を修正 |

- ローカルD1互換SQLiteとpreview D1の両方で、全migrationを空DBへ順適用し、通常・技術テーブルが`STRICT`であることを検証する。
- 型不一致、日付形式、列挙外コード、期間重複、組織循環、参照中マスター無効化、退職予定済み上長の新規指定をnegative testに含める。
- destructiveなSQLite schema変更は、新`STRICT` table作成、データ変換、件数・制約検証、切替、旧table廃止を別migrationとして順序化する。
- 他DBMSのdumpやDDLをそのまま投入せず、SQLite物理型・CHECK・triggerへ変換してから検証する。
