## 13.2 データアクセス境界

- オンライン経路は「SCR-006 → M-001 → Cloudflare Workers `fetch` → API-006 → M-002/IF-06」、定期経路は「Cloudflare Cron Trigger → Worker `scheduled`ハンドラー → Cloudflare Queues → Worker `queue`ハンドラー → JOB-001 → M-002/IF-07」とする。`scheduled`は初回投入だけ、`queue`は1メッセージ・最大40社員の1チャンクだけを担当する。
- API-006とJOB-001はD1 Binding `env.DB`、Prepared Statement、`batch()`、TBL、SQL-ID、トランザクションを扱わない。両者からD1・SQLへの直接アクセスは禁止する。
- M-002、M-003、M-004、M-007は論理的な公開IFで連携し、DBデータが必要な場合はM-006の公開IFへ委譲する。
- composition rootだけがWorker環境からD1 Bindingを取り出してM-006実装を構築し、M-006だけが`env.DB.prepare(...).bind(...)`と`env.DB.batch(...)`によりSQL-003、SQL-007、SQL-009〜SQL-013、SQL-031を実行する。
- 更新処理はM-002が論理境界を宣言し、M-006が社員versionガードと業務条件を含むD1 mutation batchを全成功または全ロールバックで実行する。即時再試行はCloudflareが再試行推奨とする明示的エラーだけを対象にfull jitter付きで初回+最大2回とし、書込み結果不明時はM-006経由で状態・versionを再確認する。過負荷・タイムアウト・CPU・メモリ・実行文予算超過は即時再試行せず、定期経路ではQueue再配信へ委ねる。
- `queue` invocationはD1実行文数900以下（Paid上限1000）、対象40社員以下、保守的最悪841文とする。後続ありではM-002がJOB-001経由で返す退職日・社員IDカーソルをqueueハンドラーが次メッセージへ引き継ぎ、1件・並列1・再試行3回、失敗後はDLQとする。
- BR-001〜BR-008の全経路は10章、実装基盤・運用環境・容量・試験実装の引継ぎは12章を正本とする。
- NFR-001のログアウト経路は「共通ヘッダー／認証失効 → UC-013(F-013) → §3.15 → M-001/IF-04 → SCR-010」で完結し、API・JOB・M-006・SQL・D1を使用しない。

## 13.3 認可済み参照条件の伝播

- 社員検索は「SCR-001 → M-001 → API-001 → M-002/IF-01 → M-003/IF-03 → M-006/IF-26・IF-16 → M-003/IF-03 → M-002/IF-01 → M-006/IF-01・IF-02」とする。社員詳細は同じ認可往復後にM-002/IF-02がM-006/IF-03を、存在確認用`ALL`と実スコープ用の順で呼ぶ。
- M-003/IF-03は§2.4の固定ロールと§2.2の各UC認可定義から`ALL` / `ORGANIZATION` / `SELF`、操作者社員ID、許可組織ID集合、認可基準日、返却可能項目を確定する。社員検索・詳細・更新・出力の認可基準日は呼出時点のAsia/Tokyo業務日に固定し、部門管理者の許可組織集合はその日の主所属組織と全子孫である。組織・役職候補取得の`effectiveOn`を過去・未来の認可範囲へ流用しない。
- M-002は認可結果を縮小してよいが拡張せず、`scopeType`、操作者社員ID、許可組織ID集合、認可基準日をM-006へすべて渡す。M-006は9章の固定マッピングで論理項目をD1対応の順序付きパラメータ`?NNN`へ変換して`.bind(...)`し、独自に認可範囲を追加しない。
- APIやJOBは認可用データを取得するためにもM-006を直接呼ばない。M-006/IF-26・IF-16はM-003からの認可材料取得、M-006/IF-01〜IF-03はM-002からの業務データ取得であり、呼出主体と目的を分離する。
