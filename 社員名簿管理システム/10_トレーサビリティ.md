# 10. 要求・設計トレーサビリティ

業務要求BR-001〜BR-008とログイン・ログアウトの根拠となるNFR-001を起点に、機能、想定される全ユースケース、シーケンス、画面、API、モジュール、JOB、SQL、DBまでの実現経路を示す。APIはCloudflare Workersの`fetch`ハンドラー、JOBはCron Triggersを受ける`scheduled`ハンドラーからCloudflare Queuesへ投入し、`queue`ハンドラーで継続する境界とする。APIはログイン以外をM-002、ログインをM-003、JOBは1メッセージにつきM-002の公開処理だけを1回呼び出し、D1 Binding `env.DB`、Prepared Statement、`batch()`、SQLはすべてM-006 D1データアクセスだけが扱う。API・JOBからM-006、D1、TBL、SQLへの直接アクセスは禁止する。

## 10.1 追跡ルール

- 正引きは「BR/NFR → F → UC → §3.x → SCR/API/JOB → M → SQL → DB」、逆引きはこの逆順で確認する。
- F-001〜F-014とUC-001〜UC-014を本表へ1回以上記載し、クライアント内完結はAPI/JOB/SQL/DBを「なし」、システム起動は画面/APIを「なし」と明示する。
- 画面初期表示に必要なマスター取得API・SQLは、その画面を利用する要求行にも併記する。
- SQL-031 監査ログ登録は監査対象の更新・出力・権限変更要求へ対応付ける。
- 「なし」は設計漏れではなく、その要求に画面、APIまたはJOBの起動境界が存在しないことを示す。
