# 1. 基本情報

| 項目 | 内容 |
|---|---|
| シーケンスID | SEQ-001 |
| シーケンス名 | 会議室予約登録シーケンス |
| 概要 | 利用者が予約登録画面から会議室を予約する際の連携。重複予約チェックと、有料会議室の場合の支払い方法(従量契約)確認を経て予約を確定する。 |
| 契機 | 利用者操作(予約登録画面での登録) |
| 関連要素 | SCR-003, API-003, MOD-002, MOD-003, TBL-001, TBL-002, TBL-003 |

# 2. 登場要素

| 要素 | 種別 | ID/参照 | 役割 |
|---|---|---|---|
| 利用者 | アクター | - | 予約を登録する一般ユーザー／管理者 |
| 予約登録画面 | 画面 | SCR-003 | 予約内容の入力・登録操作・結果表示 |
| 予約登録API | API | API-003 | 予約登録リクエストの受付・応答 |
| 会議室検索サービス | モジュール | MOD-002 | 会議室の存在確認・詳細取得(getRoomById)。存在・利用停止・過去日時の前提判定に用いる |
| 予約サービス | モジュール | MOD-003 | 会議室行ロック・有料契約確認・重複判定・予約確定 |
| ユーザーマスタ | テーブル | TBL-001 | 会議室が有料の場合の課金契約状態(BILLING_STATUS)参照 |
| 会議室マスタ | テーブル | TBL-002 | 会議室の利用単価(HOURLY_RATE)参照(有料判定) |
| 予約トランザクション | テーブル | TBL-003 | 重複予約の判定・予約レコードの確定 |

# 3. シーケンス図

```mermaid
sequenceDiagram
    actor U as 利用者
    participant SCR as SCR-003 予約登録画面
    participant API as API-003 予約登録
    participant MOD2 as MOD-002 会議室検索サービス
    participant MOD as MOD-003 予約サービス
    participant R as TBL-002 M_ROOMS
    participant USR as TBL-001 M_USERS
    participant RES as TBL-003 T_RESERVATIONS

    U->>SCR: 会議室・日時・タイトルを入力し登録
    SCR->>API: POST /api/reservations
    Note over API: API-COM §7 共通前処理(認証・認可・入力検証)
    API->>MOD2: 会議室取得(getRoomById)
    MOD2->>R: 会議室を取得(存在・ステータス・利用単価)
    alt 会議室なし / 利用停止 / 過去日時(API-003 (2) 予約前提判定)
        API-->>SCR: エラー応答(ERR-007 / ERR-010 / ERR-004)
        SCR-->>U: MSG-016 等を表示(会議室に留まり再入力)
    else 前提OK
        API->>MOD: 予約登録(会議室ID・開始・終了・タイトル)
        MOD->>R: 会議室行ロック取得・利用単価(HOURLY_RATE)取得
        opt 有料会議室(HOURLY_RATE ＞ 0)
            MOD->>USR: 課金契約状態(BILLING_STATUS)を取得
        end
        alt 有料 かつ 未契約(BILLING_STATUS ≠ 2)
            MOD-->>API: ERR-008
            API-->>SCR: エラー応答
            SCR-->>U: MSG-014(支払い方法登録が必要)
        else 前提OK(無料 または 契約有効)
            MOD->>RES: 重複予約検索(同一ROOM_ID・STATUS=1・時間帯重複)
            alt 重複あり
                MOD-->>API: ERR-003
                API-->>SCR: エラー応答
                SCR-->>U: MSG-003(重複エラー表示)
            else 重複なし
                MOD->>RES: 予約を確定(STATUS=1 予約済)
                RES-->>MOD: 登録完了(予約ID)
                MOD-->>API: 予約ID
                API-->>SCR: 201 Created
                SCR-->>U: MSG-001(予約完了)
            end
        end
    end
```

# 4. ステップ説明

| No | 送信元 → 送信先 | 内容 |
|---|---|---|
| 1 | 利用者 → SCR-003 | 会議室・利用日時・タイトルを入力し登録操作を行う |
| 2 | SCR-003 → API-003 | 予約登録リクエストを送信する |
| 3 | API-003 | API-COM §7 の共通前処理(認証・認可・入力検証)を実行する |
| 4 | API-003 → MOD-002 | 会議室を取得(getRoomById)し、存在・利用停止・過去日時の予約前提を判定する(会議室なし=ERR-007 / 利用停止=ERR-010 / 過去日時=ERR-004) |
| 5 | API-003 → MOD-003 | 前提を満たす場合、予約登録処理を呼び出す |
| 6 | MOD-003 → TBL-002 | 会議室行ロックを取得し、利用単価(HOURLY_RATE)で有料/無料を判定する。利用停止(STATUS=2)はここでも再判定し ERR-010 |
| 7 | MOD-003 → TBL-001 | 有料の場合、利用者の課金契約状態(BILLING_STATUS)を確認する(未契約=ERR-008) |
| 8 | MOD-003 → TBL-003 | 同一会議室・予約済(STATUS=1)・時間帯重複の予約有無を判定する(重複=ERR-003) |
| 9 | MOD-003 → TBL-003 | 判定を通過した予約を STATUS=1(予約済)で確定する |
| 10 | API-003 → SCR-003 → 利用者 | 予約完了(MSG-001)を表示する。予約は即確定し決済でブロックしない |

# 5. 例外・代替

| 分岐 | 分岐後の流れ |
|---|---|
| 会議室が存在しない | API-003 (2) 予約前提判定が ERR-007 を返し、SCR-003 はエラーを表示する |
| 会議室が利用停止(STATUS=2) | API-003 (2) 予約前提判定が ERR-010 を返し(MOD-003 でも行ロック下で再判定)、SCR-003 は MSG-016 を表示する |
| 開始日時が過去 | API-003 (2) 予約前提判定が ERR-004 を返し、SCR-003 は MSG-006 を表示する |
| 有料会議室かつ支払い方法未登録(BILLING_STATUS ≠ 2) | MOD-003 が ERR-008 を返し、SCR-003 は MSG-014 を表示し支払い方法登録(SCR-007)へ導線を示す |
| 時間帯が重複する予約が存在 | MOD-003 が ERR-003 を返し、SCR-003 は MSG-003 を表示する |
