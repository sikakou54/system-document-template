AI Agent および開発者が docs/ 配下のドキュメントを作成・更新・レビューする際のルールを定義する。
**作業開始時に本ファイルと対象ディレクトリのindex.mdまたはプロット.mdを必ず読むこと。**

# 1. 役割

docs/ 配下の全ドキュメントを対象に、AI Agent と開発者が企画・要求定義・要件定義・機能設計ドキュメントの作成・更新・レビュー・横断整合チェックを行う。基本原則: 各仕様は正本ファイル1箇所にのみ記載し、他文書は ID で参照する。

# 2. 開発プロセス

企画・構想 ＞ 要求定義 ＞ 要件定義 ＞ 機能設計 ＞ コーディング ＞ テスト ＞ リリース

| 工程 | 対応ディレクトリ | 定義するもの |
|---|---|---|
| 企画・構想 | 00_企画/ | 対象システムが解決する課題、価値、対象範囲、期待する状態 |
| 要求定義 | 01_要求定義/ | 利用者・管理者・組織が対象システムに求めること(RQ) |
| 要件定義 | 02_要件定義/ | 機能要件(FR)、共通機能要件(CFR)、非機能要件(NFR)、システムユースケース(UC。各FR/CFR内に定義)、データモデル(MDL)、要件定義に関わるトレーサビリティ(04_トレーサビリティ: 要求RQ→要件・FR→CFR/NFR・機能×データCRUD相関) |
| 機能設計 | 03_機能設計/ | 概要設計の後、機能要件・ユースケースからシーケンスを作成し、その連携契約を共通入力として画面、API、JOB、モジュール、データベース、クエリ、共通定義へ具体化する。対応関係は 11_トレーサビリティで一元管理する |

# 4. ID規則

ID採番・命名の**正本は 99_その他/ID採番ルール.md**。本節はその要点と、各IDの一覧(index.md／インライン定義)の所在を示す。

## 4.1 グローバルID

- 種別・ID形式・採番例・命名の詳細は正本 **99_その他/ID採番ルール.md** に定義する。対象: RQ / CFR / FR / NFR / MDL※ / RULE / RULE-COM / SCR / API / MOD※ / JOB※ / TBL / SQL / SEQ / ERR / MSG（※=本プロジェクト拡張ID）。
- 各IDの一覧(採番基準=最大値+1)の所在: 要件・設計IDは §3.1 の対応ディレクトリ配下 index.md。例外 — RQ=01_要求定義/要求一覧.md、CFR・RULE-COM=CFR-000_システム共通要件.md、RULE=該当FR、ERR=03_機能設計/04_API設計/エラーメッセージ一覧.md、MSG=各SCR文書「メッセージ一覧」節(インライン)。
- 採番は一覧の最大値+1（ERR/MSG は全文書横断）、欠番の再利用は禁止。廃止はファイル・行を削除せず状態を「廃止」で残す(§7)。RQ は 要求一覧.md を正本とし補足要求のみ個別ファイル(RQ-XXX_要求名.md)。要件ファイル名は ID_名称.md、CFR-XXX・RULE-COM-XXX は CFR-000 に集約、NFR-XXX は 02_要件定義/02_非機能要件/ に個別定義。
- シーケンス作成時に下位設計が未作成の場合、SEQで参照するSCR/API/JOB/MOD/TBL/SQL IDを各index.mdへ状態「未着手」で先に予約する。予約IDは名称・責務・関連SEQを対応表へ記録し、文書作成時も同じIDを使用する。

# 5. 参照方向ルール

参照を3種類に分類する。**根拠参照**=設計が直接使用した上位IDを示す(例 SEQ-001→FR-002/UC-01)=許可、**依存参照**=呼ぶ側が呼ばれる側IDを記載(例 API-003→MOD-003、MOD-003→SQL-025)=許可、**列挙型逆参照**=自分を実現・利用する下流／呼び出し元を本文へ網羅列挙する(例 FR-001に全SCR/API、TBL-003に全利用元MOD)=禁止。

設計間の対応関係は 11_トレーサビリティの専用対応表を正本とする。個別文書に書くIDは、その文書を理解・実装するための直接根拠または呼び出し先に限定し、対応表と同じ網羅一覧を再掲しない。SEQの「関連設計ID」は後続設計への引き渡し契約として例外的に記載できるが、詳細仕様は記載しない。

## 参照方向マトリクス

| 参照元 | 参照してよい先 |
|---|---|
| RQ | なし(後続工程IDを参照しない) |
| FR | なし(用語・業務ルールの言及のみ可) |
| NFR | なし |

要求定義(RQ)・要件定義(FR/NFR/UC/MDL)は機能設計IDを参照・言及しない。UCは各FR/CFR文書内のローカル連番とし、他文書ではFR-XXX/UC-01の完全修飾形を使う。概念MDLはSEQより前に作成してSEQの入力とし、SEQで不足が判明した場合は要件変更としてMDLを更新してからSEQを再確認する。API・JOBはSQLを直接参照せず、必ずMODへ委譲する。

# 6. トレーサビリティ

- 対応関係の正本は専用トレーサビリティ表とする。個別文書の根拠・依存欄は設計の理解に必要な直接参照だけを持ち、下位成果物から上位成果物を逆引きする網羅一覧を重複記載しない。
- トレースは2層で管理する。要件定義層(RQ→FR/CFR/NFR、FR→CFR/NFR、機能×MDL CRUD)=02_要件定義/04_トレーサビリティ/、機能設計層(FR/UC→SEQ、SEQ→SCR/JOB/API/MOD/SQL/MDL/TBL/ERR/MSG)=03_機能設計/11_トレーサビリティ/。
- 全体経路は `RQ → FR/CFR → UC → SEQ → SCR/JOB → API → MOD → SQL → MDL/TBL` を基本とする。画面を持たないジョブ・Webhook・外部連携はSEQから該当する入口へ直接分岐する。
- 最低限、FR/UCとSEQ、SEQとSCR/API/MOD/MDL/ERR/MSGの対応を確認できるようにする。SEQを省略する場合もFR/UC→SEQ表とSEQ一覧に省略理由を残す。
- FR→UCは各FR/CFR文書の「システムユースケース」節を参照する。RQ→FRは要件定義側のtraceability_rq_fr.mdを参照する。
- **文書の作成・更新時は、対応する一覧の行更新を同一作業内で必ず行う(§10-7)。**

# 7. 状態定義

状態の定義・遷移条件は本ファイルにのみ記載する。各文書・一覧は状態値のみ持つ。

| 状態 | 意味 | 次状態への遷移条件 |
|---|---|---|
| 未着手 | 一覧に行のみ存在、またはテンプレート未記入 | 着手時に「作成中」へ |
| 作成中 | 記入中。「未定」「要確認」が残っていてよい | 必須セクションがすべて埋まったら「レビュー中」へ |
| レビュー中 | 作成者が完了宣言し、レビュー依頼済み | レビュー観点(§9)全パス＋「未定」ゼロで「確定」へ |
| 確定 | 実装の根拠にしてよい | 変更時は「作成中」に戻す |
| 廃止 | 使用しない。削除はしない | — |

ディレクトリ再編は廃止文書の削除ではない。内容とIDを新しい正規パスへ移行し、全参照を更新できた場合に限り旧ディレクトリを削除できる。今回の旧08→新02移行はこの構造移行例外を適用し、SEQ-IDは維持する。

# 9. レビュー・整合チェックルール

品質は個別文書レビュー(§9.1)と横断整合チェック(§9.2)の2段で担保する。結果は 99_その他/テンプレート_レビューチェックリスト.md に記録する。

## 9.1 個別文書レビュー

作成・更新した文書を次の観点でレビューする。

| No | 観点 |
|---|---|
| 1 | ID が重複していない・参照先 ID(上流・依存先)がすべて実在する |
| 2 | 列挙型逆参照がない(§5) |
| 3 | 一覧(index)と本文の基本情報が一致している |
| 4 | 上流文書(トレース元)と内容が矛盾していない |
| 5 | 共通仕様(API-COM・共通カラム・共通定義設計)との重複記載がない(§3.2) |
| 6 | 要件定義の内容と機能設計の内容が混在していない |
| 7 | 個別設計は実装判断できる粒度、SEQは後続設計へ連携契約を引き渡せる粒度で書かれている |
| 8 | 「未定」「要確認」の残数と確認先が明確 |
| 9 | SEQはFR/UCを根拠とし、基本・代替・例外フロー、条件分岐、CRUD、トランザクション境界を過不足なく表現している(対象UCの状態パターンSP-xを図の分岐で網羅している) |
| 10 | SEQ上のUI/API/JOB/MOD/MDL/ERR/MSGが予約済みまたは実在し、後続設計と責務・順序が一致している |
| 11 | UIに業務ロジック、APIに過剰な業務ロジック、DBにアプリケーション処理責務が混入していない |
| 12 | SEQに変数操作、具体的SQL、ライブラリ固有処理、具体的なERR-ID・MSG-ID、個別設計と同じ詳細仕様が混入していない |
| 13 | システムユースケースの状態パターンが基本・代替・例外フローと双方向に網羅されている(全パターンに対応フロー、全ALT/EXCがいずれかのパターンから参照。状態は要件レベルの日本語名で設計層IDを参照していない) |

- 確定条件=全観点パス＋「未定」ゼロ。出力は 指摘ID・重要度・対象箇所・指摘内容・修正案 の表(なければ「レビュー指摘なし」)。

## 9.2 横断整合チェック(節目ごと)

フェーズ完了・リリース前・大きな変更後に docs/ 全体を走査し次を機械的に検証する(全 index.md・エラーメッセージ一覧.md(ERR)・各SCRインライン(MSG)・共通定義設計を突合)。問題ゼロなら各項目の確認件数を記録、検出時は 対象ファイル・内容・修正案 を挙げて是正する。

| No | チェック | 検出方法 |
|---|---|---|
| 1 | ID の一意性 | 全一覧、エラーメッセージ一覧.md(ERR)、各SCR文書のインライン定義(MSG)で同一 ID が2回定義されていないか |
| 2 | 参照先の実在 | 本文中の全 ID(FR/CFR/NFR/MDL/UC/SCR/API/MOD/JOB/TBL/SQL/SEQ/ERR/MSG)が一覧・エラーメッセージ一覧.md・各SCRのインライン定義に存在するか |
| 3 | 一覧と文書の同期 | 一覧の行と文書の基本情報欄が一致するか。02_要件定義/01_機能要件/index.md は掲載列のみ確認する。文書のない ID 行・一覧にない文書がないか |
| 4 | 逆参照の混入 | 上流文書に下流 ID の一覧が書かれていないか |
| 5 | トレース鎖の断絶 | RQ→FR/CFR→UC→SEQ→SCR/JOB→API→MOD→SQL→MDL/TBLを対応表で辿れるか。SEQ省略対象は理由があるか(共通基盤は除外を明記) |
| 6 | 定義箇所の重複 | ERR の文言・HTTPステータス、MSG 文言、区分値(共通コード定義)、共通カラム、API 共通仕様が、定義箇所以外に再記載されていないか |
| 7 | ローカルIDの参照形式 | 他文書からの EVT/ALT/EXC/SP/IDX 参照が完全修飾(SCR-XXX/EVT-01 形式)か |
| 8 | 状態の妥当性 | 状態欄を持つ文書で、「確定」文書に「未定」「要確認」が残っていないか |
| 9 | 外部ライブラリ直接利用禁止 | 外部ライブラリ利用専用モジュール以外の API・JOB・モジュールに外部ライブラリ・標準実行基盤APIの直接呼び出しが記載されていないか(必要な場合は MOD-ID 参照か。§8-14) |
| 10 | 処理結果定義 | 戻り値・レスポンスを返す処理に「処理結果」ブロックがあり返却項目表が定義されているか。例外でない無効・不一致が処理結果に定義されているか(§8-15) |
| 11 | フローチャートの簡潔性 | API・JOB・モジュールのフローのノード/エッジに呼び出し先・外部サービス名・ステータス値等の詳細が書かれていないか(詳細は処理詳細か。§8-16) |
| 12 | シーケンス構成 | 02_シーケンス設計が存在し、旧08パスが存在せず、03〜08の番号・README・目次・プロンプトが一致するか |
| 13 | シーケンス根拠 | 各SEQの関連FR/UCが実在し、要件にない処理が混入せず、基本・代替・例外フローと対象UCの状態パターン(SP-x)を分岐で網羅するか |
| 14 | 下位設計への反映 | SEQ上のSCR/API/JOB/MOD/MDL/TBL/SQL/ERR/MSGが対応表と各設計に存在し、責務・順序・CRUDが一致するか |
| 15 | シーケンス粒度 | SEQに具体的SQL、変数操作、物理メソッド名、ライブラリ固有処理、具体的なERR-ID・MSG-ID、個別設計の詳細が混入していないか |
| 16 | 共通入力の確認 | 画面/API/JOB/MOD/TBL/SQL/共通定義の作成・更新時に関連SEQを確認し、適用結果をレビューしたか |
| 17 | 対応表の一元性 | 個別文書へ下位／利用元の網羅一覧を重複記載せず、FR/UC→SEQとSEQ→各設計の対応を専用表で一元管理しているか |
