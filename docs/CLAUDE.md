AI Agent および開発者が docs/ 配下のドキュメントを作成・更新・レビューする際のルールを定義する。
**作業開始時に本ファイルと対象ディレクトリのindex.mdまたはプロット.mdを必ず読むこと。**

# 1. 役割

| 項目 | 内容 |
|---|---|
| 対象 | docs/ 配下の全ドキュメント |
| AI Agent の役割 | 企画・要求定義・要件定義・機能設計ドキュメントの作成、更新、レビュー、横断整合チェック |
| 基本原則 | 各仕様は正本ファイル1箇所にのみ記載し、他文書は ID で参照する |

# 2. 開発プロセス

```text
企画・構想 ＞ 要求定義 ＞ 要件定義 ＞ 機能設計 ＞ コーディング ＞ テスト ＞ リリース
```

| 工程 | 対応ディレクトリ | 定義するもの |
|---|---|---|
| 企画・構想 | 00_企画/ | 対象システムが解決する課題、価値、対象範囲、期待する状態 |
| 要求定義 | 01_要求定義/ | 利用者・管理者・組織が対象システムに求めること(RQ) |
| 要件定義 | 02_要件定義/ | 業務要件(BR)、機能要件(FR)、共通機能要件(CFR)、非機能要件(NFR)、業務ユースケース(UC) |
| 機能設計 | 03_機能設計/ | 実装に必要な設計情報(概要、画面、API、JOB、モジュール、データベース、クエリ、シーケンス、権限、共通定義) |
| トレーサビリティ | 04_トレーサビリティ/ | 要件〜設計の対応表(各文書基本情報の転記) |

「基本設計」「詳細設計」という分類は使用しない(禁止事項 §10-1)。

## 3.2 カタログ正本

以下の内容は正本ファイルにのみ定義する。他文書での再記載は禁止(ID参照のみ)。

| 定義対象 | 正本ファイル | 他文書での扱い |
|---|---|---|
| 共通設計(認証・エラー封筒・ページネーション・共通エラー) | 03_機能設計/03_API設計/API-COM_共通設計.md | 差分のみ記載 |
| 区分値(共通定義) | 03_機能設計/10_共通定義設計/DEF-XXX_名称.md | DEF-XXX/CODE-XXX で参照 |
| 共通カラム | 03_機能設計/06_データベース設計/共通カラム.md | 各 TBL に再記載しない |
| 外部ライブラリ・標準実行基盤APIの利用 | 03_機能設計/05_モジュール設計/MOD-XXX_名称.md | API・JOB・モジュールは MOD-ID と処理名のみ参照 |
| 用語 | 99_その他/用語集.md | — |

※ ERR(エラーコード)・MSG(画面表示文言)は中央カタログを**廃止**し、インライン定義とする。ERR は共通エラー(認証/認可/バリデーション)を API-COM、各 API の固有エラーを当該 API 文書に、MSG は各 SCR 文書の「メッセージ一覧」節に定義する。

# 4. ID規則

ID採番・命名の**正本は 99_その他/ID採番ルール.md**。本節はその要点と、各IDの一覧(index.md／インライン定義)の所在を示す。

| 画面 | SCR-XXX | SCR-001 | 03_機能設計/02_画面設計/index.md |
| API | API-XXX | API-001 | 03_機能設計/03_API設計/index.md |
| モジュール ※拡張 | MOD-XXX | MOD-001 | 03_機能設計/05_モジュール設計/index.md |

| テーブル | TBL-XXX | TBL-001 | 03_機能設計/06_データベース設計/index.md |

| エラー | ERR-XXX | ERR-001 | 各API文書＋API-COM_共通設計.md にインライン定義 |

※「拡張」は本プロジェクト固有のプロジェクト拡張ID。採番ルール上の汎用ID種別には含まれない。

- 採番は一覧の最大値 +1(ERR/MSG はインライン定義のため全文書横断で最大値 +1)。欠番の再利用は禁止。
- 廃止する場合はファイル・行を削除せず、状態を「廃止」にして残す。
- 要求(RQ)の正本一覧は 01_要求定義/index.md に置く。補足説明が必要な要求のみ、個別ファイル「ID_要求名.md」(例: 01_要求定義/RQ-XXX_要求名.md)を作成する。
- 要件ファイル名は「ID_要件名.md」とする(例: 02_要件定義/02_機能要件/FR-XXX_機能名.md)。システム全体の要件・共通機能要件(CFR-XXX)・共通業務ルール(RULE-COM-XXX)は CFR-000_システム共通要件.md に集約する。非機能要件(NFR-XXX)は 02_要件定義/03_非機能要件/ に個別文書として定義する。

# 5. 参照方向ルール

参照を3種類に分類する。

| 種類 | 内容 | 可否 |
|---|---|---|
| トレース参照 | 下流文書が、実現元である上流文書の ID を基本情報欄に記載する(例: UC-002 の「トレース元: FR-002」) | ✅ 許可 |
| 依存参照 | 呼び出す側が、呼び出される側の ID を記載する(例: API-003 → MOD-003、MOD-003 → TBL-003) | ✅ 許可 |
| 列挙型逆参照 | 自分を実現・利用する下流／呼び出し元文書の一覧を記載する(例: FR-001 に関連UC一覧、TBL-003 に参照元MOD一覧) | ❌ 禁止 |

**判定基準: 後から新しい文書が追加されたとき、既存文書の編集が必要になる参照は逆参照であり禁止。**
逆方向の追跡は一覧で行う(§6)。

## 参照方向マトリクス

| 参照元 | 参照してよい先 |
|---|---|
| RQ | なし(後続工程IDを参照しない) |
| FR | なし(用語・業務ルールの言及のみ可) |
| NFR | なし |
| UC | FR(トレース元) |
| SCR | UC(トレース元)／API(呼び出し)／MSG／DEF(共通定義) |
| API | UC(トレース元)／MOD(実装先)／ERR／DEF(共通定義)／API-COM_共通設計.md |
| MOD | TBL(CRUD対象)/DEF(共通定義)／他MOD(呼び出し) |
| JOB | FR(トレース元)／MOD・TBL(利用)／ERR |
| TBL | 他TBL(外部キー)/DEF(共通定義) |
| MSG | ERR(対応付け) |
| ERR | なし(末端) |

要求定義(RQ)は後続工程のIDを参照・言及しない。
要件定義(FR / NFR / UC)は機能設計(SCR / API / MOD / TBL / JOB)を参照・言及しない。要件→設計のトレースは、各対応表と機能設計側のトレース元で辿る。

# 6. トレーサビリティ

- リンクの正本は各文書のトレース欄・依存欄。**一覧は各文書の基本情報の転記のみで構成する**(一覧に独自情報を書かない)。
- 各一覧は文書種別ごとの一覧テンプレートに従う(99_その他/テンプレート_一覧.md)。
- 企画・構想は 00_企画/プロット.md で管理する。要求(RQ)の一覧は 01_要求定義/index.md で管理する。機能要件(FR)の一覧は 02_要件定義/02_機能要件/index.md で管理する。非機能要件(NFR)の一覧は 02_要件定義/03_非機能要件/index.md で管理する。システム全体の要件・共通機能要件・共通業務ルールは CFR-000_システム共通要件.md で管理する。
- 上流→下流の追跡は一覧のフィルタで行う。

| 追跡したい経路 | 手順 |
|---|---|
| FR-002 を実現する UC | 02_要件定義/04_業務ユースケース/index.md で「トレース元 = FR-002」の行を探す |
| UC-002 を実現する SCR / API | 03_機能設計/02_画面設計/index.md と 03_機能設計/03_API設計/index.md で「トレース元 = UC-002」の行を探す |
| API-003 が使う MOD / TBL | 03_機能設計/03_API設計/index.md の主要依存列 → 03_機能設計/05_モジュール設計/index.md の主要依存列を辿る |

- **文書の作成・更新時は、対応する一覧の行更新を同一作業内で必ず行う(§10-7)。**

# 7. 状態定義

状態の定義・遷移条件は本ファイルにのみ記載する。各文書・一覧は状態値のみ持つ。

| 状態 | 意味 | 次状態への遷移条件 |
|---|---|---|
| 未着手 | 一覧に行のみ存在、またはテンプレート未記入 | 着手時に「作成中」へ |
| 作成中 | 記入中。「未定」「要確認」が残っていてよい | 必須セクションがすべて埋まったら「レビュー中」へ |
| レビュー中 | 作成者が完了宣言し、レビュー依頼済み | レビュー観点(§9)全パス＋「未定」ゼロで「確定」へ |
| 確定 | 実装の根拠にしてよい | 変更時は「作成中」に戻す |
| 廃止 | 使用しない。削除はしない | — |

# 8. 作成ルール

| No | ルール |
|---|---|
| 1 | 各文書ディレクトリに置かれた テンプレート_XXX.md をコピーして使う(共通の一覧雛形・レビュー観点は 99_その他/テンプレート_一覧.md / 99_その他/テンプレート_レビューチェックリスト.md)。セクションの独断追加・削除は禁止。構成を変えたい場合はテンプレート自体の変更を先に提案する |
| 2 | 概要・説明文は1〜3行。一覧・項目・条件・分岐・処理はテーブルで書く |
| 3 | 不明点は推測で埋めない。要件定義では確認できた内容だけを記載し、未確認の内容は本文に入れない |
| 4 | 詳細仕様は正本ファイル1箇所にのみ書き、他文書は ID 参照とする |
| 5 | 新規文書は「一覧に行追加 → 文書作成」の順で行う |
| 6 | コーディング時に迷わない粒度まで書く(型・必須・条件・エラー・境界値まで) |
| 7 | 確定文書に触る場合は、変更内容と影響 ID 一覧を先に提示してから編集する |
| 8 | 要件定義では「制約・補足」などの自由記述欄を使わず、前提条件・制限事項に分けて記載する |
| 9 | 非機能要件(NFR)は 02_要件定義/03_非機能要件/ に個別文書(NFR-XXX)として定義する(機能横断・特定機能固有のいずれも同ディレクトリ)。FR との適用関係は 04_トレーサビリティ/traceability_fr_nfr.md で管理する。システム全体の要件・共通機能要件(CFR)・共通業務ルール(RULE-COM)は CFR-000_システム共通要件.md に定義する |
| 10 | ユーザーが設計書のフォーマットの定義ルール(記法・セクション構成・項目・入力形式・値・条件など)について指摘・発言した場合は、その内容を必ず該当テンプレート(テンプレート_XXX.md / index.md / プロット.md)の当該セクション直上の定義ルールコメント(定義内容 / 定義する条件 / 項目説明 / 定義ルール)へ反映する。会話内での回答やその場限りの修正で終わらせない。セクション構成自体の変更を伴う場合は No.1 に従いテンプレート変更を先に提案する |

| 12 | 処理（メソッド内ステップ含む）の命名は「XXXX取得処理」「XXXX登録処理」「XXXX更新処理」「XXXX削除処理」「XXXX判定処理」のいずれかで終わる形に統一する |
| 13 | データベース設計(06)とクエリ設計(07)以外のすべてのドキュメントにおいて、物理名（英語の変数名や関数・メソッド名等）を記載しないこと。ただし、外部サービス連携で外部サービス側のAPI項目名・イベント名・ヘッダ名など、物理名以外に定義する術がない場合は例外として最小限の記載を許可する |

| 15 | API・モジュール・JOBの取得処理を含め、戻り値やレスポンスを返す処理では、フロー末尾に「処理結果」ブロックを配置し、返却する項目を \| 項目名 \| データ型 \| 値 \| 説明 \| 表で定義する。例外ではない無効・不一致などの結果は例外表ではなく処理結果に定義する |
| 16 | フローチャートのノードとエッジラベルには処理名・判定結果だけを短く記載し、呼び出し先、外部サービス名、外部ライブラリ名、リトライ回数、ステータス値、通知内容などの詳細を書かない。詳細は処理詳細に定義する |
| 17 | 企画・構想を変更する場合は 00_企画/プロット.md を先に更新し、要求定義へ反映が必要な変更を確認する。プロットには要求ID・要件ID・設計IDなどの後続工程IDを記載しない |

# 9. レビュールール

レビューは 99_その他/プロンプト/02_文書レビュー.md を使用し、99_その他/テンプレート_レビューチェックリスト.md の観点で行う。

| No | 観点 |
|---|---|
| 1 | ID が重複していない・参照先 ID がすべて実在する |
| 2 | 列挙型逆参照がない |
| 3 | 共通仕様(API-COM・共通カラム・共通定義設計)との重複記載がない |
| 4 | この記述だけで実装判断できる(要件が機能設計へ落ちている) |
| 5 | 要件定義の内容と機能設計の内容が混在していない |
| 6 | 「未定」「要確認」の残数と確認先が明確 |

- 確定条件: 全観点パス ＋「未定」ゼロ。
- 節目ごとに 99_その他/プロンプト/03_横断整合チェック.md で横断整合チェック(全ID実在・一覧⇔文書同期・トレース鎖の断絶検出)を行う。
