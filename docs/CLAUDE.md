AI Agent および開発者が docs/ 配下のドキュメントを作成・更新・レビューする際のルールを定義する。
**作業開始時に本ファイルと対象ディレクトリのindex.mdまたはプロット.mdを必ず読むこと。**

# 1. 役割

docs/ 配下の全ドキュメントを対象に、AI Agent と開発者が企画・要求定義・要件定義・機能設計ドキュメントの作成・更新・レビュー・横断整合チェックを行う。基本原則: 各仕様は正本ファイル1箇所にのみ記載し、他文書は ID で参照する。

# 2. 開発プロセス

企画・構想 ＞ 要求定義 ＞ 要件定義 ＞ 機能設計 ＞ コーディング ＞ テスト ＞ リリース

| 工程 | 対応ディレクトリ | 定義するもの |
|---|---|---|
| 企画・構想 | 00_企画/ | 対象システムが解決する課題、価値、対象範囲、期待する状態 |
| 要求定義 | 01_要求定義/ | 利用者・管理者・組織が対象システムに求めること(RQ) |
| 要件定義 | 02_要件定義/ | 機能要件(FR)、共通機能要件(CFR)、非機能要件(NFR)、システムユースケース(UC。各FR/CFR内に定義)、データモデル(MDL)、要件定義に関わるトレーサビリティ(04_トレーサビリティ: 要求RQ→要件・FR→CFR/NFR・機能×データCRUD相関) |
| 機能設計 | 03_機能設計/ | 実装に必要な設計情報(概要、画面、API、JOB、モジュール、データベース、クエリ、シーケンス、共通定義)、機能設計に関わるトレーサビリティ(11_トレーサビリティ: 要件UC→SCR/API・SCR→API→MOD→TBL) |

# 4. ID規則

ID採番・命名の**正本は 99_その他/ID採番ルール.md**。本節はその要点と、各IDの一覧(index.md／インライン定義)の所在を示す。

## 4.1 グローバルID

- 種別・ID形式・採番例・命名の詳細は正本 **99_その他/ID採番ルール.md** に定義する。対象: RQ / CFR / FR / NFR / MDL※ / RULE / RULE-COM / SCR / API / MOD※ / JOB※ / DEF※ / TBL / SQL / SEQ / ERR / MSG（※=本プロジェクト拡張ID）。
- 各IDの一覧(採番基準=最大値+1)の所在: 要件・設計IDは §3.1 の対応ディレクトリ配下 index.md。例外 — RQ=01_要求定義/要求一覧.md、CFR・RULE-COM=CFR-000_システム共通要件.md、RULE=該当FR、ERR=03_機能設計/03_API設計/エラーメッセージ一覧.md、MSG=各SCR文書「メッセージ一覧」節(インライン)。
- 採番は一覧の最大値+1（ERR/MSG は全文書横断）、欠番の再利用は禁止。廃止はファイル・行を削除せず状態を「廃止」で残す(§7)。RQ は 要求一覧.md を正本とし補足要求のみ個別ファイル(RQ-XXX_要求名.md)。要件ファイル名は ID_名称.md、CFR-XXX・RULE-COM-XXX は CFR-000 に集約、NFR-XXX は 02_要件定義/02_非機能要件/ に個別定義。

## 4.2 ローカルID

文書内でのみ一意なID。他文書から参照する場合は必ず完全修飾する。

# 5. 参照方向ルール

参照を3種類に分類する。**トレース参照**=下流が実現元の上流IDを基本情報欄に記載(例 SCR-003「トレース元: FR-002/UC-01」)=許可、**依存参照**=呼ぶ側が呼ばれる側IDを記載(例 API-003→MOD-003、MOD-003→TBL-003)=許可、**列挙型逆参照**=自分を実現・利用する下流／呼び出し元の一覧を記載(例 FR-001 にそれを実現する SCR/API 一覧、TBL-003 に参照元MOD一覧)=**禁止**。

**判定基準: 後から新しい文書が追加されたとき、既存文書の編集が必要になる参照は逆参照であり禁止。**
逆方向の追跡は一覧で行う(§6)。

## 参照方向マトリクス

| 参照元 | 参照してよい先 |
|---|---|
| RQ | なし(後続工程IDを参照しない) |
| FR | なし(用語・業務ルールの言及のみ可) |
| NFR | なし |
| MDL | FR/CFR/NFR(関連要件)／他MDL(エンティティ間の関連) |
| SCR | UC(トレース元)／API(呼び出し)／MSG／DEF(共通定義) |
| API | UC(トレース元)／MOD(実装先)／ERR／DEF(共通定義)／API-COM_共通設計.md |
| MOD | TBL(CRUD対象)/DEF(共通定義)／他MOD(呼び出し) |
| JOB | FR(トレース元)／MOD・TBL(利用)／ERR |
| TBL | 他TBL(外部キー)/DEF(共通定義) |
| MSG | ERR(対応付け) |
| ERR | なし(末端) |

要求定義(RQ)・要件定義(FR/NFR/UC/MDL)は機能設計(SCR/API/MOD/TBL/JOB)を参照・言及しない(要件→設計のトレースは各対応表と機能設計側のトレース元で辿る)。UC は独立文書でなく各FR/CFR文書内「システムユースケース」節で定義し、UC ID はローカル連番 UC-01、SCR/API は完全修飾形 FR-XXX/UC-01 でトレース元参照する。MDL は同じ要件定義層の FR/CFR/NFR(関連要件)と他MDLのみ参照し設計層ID(DEF/TBL)は参照しない(逆参照禁止・区分値/権限は要件レベルで自己完結)。物理テーブル(TBL)は MDL をトレース元にできる(本サンプル未連携)。

# 6. トレーサビリティ

- リンクの正本は各文書のトレース欄・依存欄。一覧は各文書の基本情報の転記のみで構成し独自情報を書かない(一覧雛形は 99_その他/テンプレート_一覧.md)。
- トレースは2層で管理する。要件定義層(要求RQ→要件・FR→CFR/NFR・機能×データCRUD)= 02_要件定義/04_トレーサビリティ/、機能設計層(要件UC→SCR/API・SCR→API→MOD→TBL)= 03_機能設計/11_トレーサビリティ/。各経路でどの対応表を辿るかは両ディレクトリの index.md の一覧を参照する。工程をまたぐ対応は下流側の層が担う。
- FR→UC は対応表を持たず各FR/CFR文書の「システムユースケース」節を参照。上流→下流の追跡は一覧のフィルタで行う。
- **文書の作成・更新時は、対応する一覧の行更新を同一作業内で必ず行う(§10-7)。**

# 7. 状態定義

状態の定義・遷移条件は本ファイルにのみ記載する。各文書・一覧は状態値のみ持つ。

| 状態 | 意味 | 次状態への遷移条件 |
|---|---|---|
| 未着手 | 一覧に行のみ存在、またはテンプレート未記入 | 着手時に「作成中」へ |
| 作成中 | 記入中。「未定」「要確認」が残っていてよい | 必須セクションがすべて埋まったら「レビュー中」へ |
| レビュー中 | 作成者が完了宣言し、レビュー依頼済み | レビュー観点(§9)全パス＋「未定」ゼロで「確定」へ |
| 確定 | 実装の根拠にしてよい | 変更時は「作成中」に戻す |
| 廃止 | 使用しない。削除はしない | — |

| 17 | 企画・構想を変更する場合は 00_企画/プロット.md を先に更新し、要求定義へ反映が必要な変更を確認する。プロットには要求ID・要件ID・設計IDなどの後続工程IDを記載しない |
| 18 | フローチャート(API §4・JOB §4・MOD 各インターフェースの処理フロー)の処理ノードは、処理詳細で他の定義単位を呼び出す(委譲する)処理 —— \| MOD-ID \| 処理名 \|・\| SQL-ID \| クエリ名 \|・\| 外部サービス \| 処理名 \| のいずれかの表を持つ処理 —— を二重角括弧 `[["(n) 処理名"]]`(関数呼び出しブロック=サブルーチン形状)で表し、委譲を伴わない内部処理・「処理結果」/「レスポンス」ブロックは矩形 `["(n) 処理名"]`、判定は `{"(n) 判定名"}`、エラー返却/送出ノードは矩形で表す。システム構成図など処理フロー以外の図(概要設計)は対象外 |
| 19 | 何らかの処理(取得・検証・整形・登録・更新・削除など)を行い、その結果を見て分岐する場合は、処理と判定を必ず別ステップに分けて定義する。処理ノードから直接分岐エッジを出さず、処理の直後に判定ステップ(判定型 `{"(n) XXX判定"}`)を置き、判定はその処理結果(「(x) 処理名の結果」)を参照する。判定ノードの中で取得・整形・登録・更新・削除等の処理を暗黙に行わない。個別処理フローだけでなく共通処理フロー(認証の JWT 検証など)にも適用する |
| 20 | データベースのクエリ(SQL-XXX)を直接実行してよいのはモジュール(MOD-XXX)のみとする。API・JOB からクエリ(SQL)を直接呼び出すことは禁止し、データの参照・登録・更新・削除は必ずモジュール(MOD-XXX)を介して `\| MOD-ID \| 処理名 \|` で呼び出す(呼び出し元は MOD-ID と処理名のみを参照)。該当する処理がモジュールに無ければ、モジュール設計(05)に定義してから呼び出す。テーブルのデータを参照・更新する処理は、モジュール側で必ず対応するクエリ設計(SQL-XXX)を伴う(クエリを伴わないテーブル参照は禁止) |
| 21 | 入出力・引数・戻り値などデータ項目を定義する箇所では、物理名や型名による配列表記(例: 会議室一覧(Room[]))を禁止し、Object または Object[] 等の型を明記した上で、内訳の項目をインデントで展開して定義する(§8-13 の物理名禁止と合わせて適用する) |

# 9. レビュー・整合チェックルール

品質は個別文書レビュー(§9.1)と横断整合チェック(§9.2)の2段で担保する。結果は 99_その他/テンプレート_レビューチェックリスト.md に記録する。

## 9.1 個別文書レビュー

作成・更新した文書を次の観点でレビューする。

| No | 観点 |
|---|---|
| 1 | ID が重複していない・参照先 ID(上流・依存先)がすべて実在する |
| 2 | 列挙型逆参照がない(§5) |
| 3 | 一覧(index)と本文の基本情報が一致している |
| 4 | 上流文書(トレース元)と内容が矛盾していない |
| 5 | 共通仕様(API-COM・共通カラム・共通定義設計)との重複記載がない(§3.2) |
| 6 | 要件定義の内容と機能設計の内容が混在していない |
| 7 | この記述だけで実装判断できる粒度で書かれている |
| 8 | 「未定」「要確認」の残数と確認先が明確 |

- 確定条件=全観点パス＋「未定」ゼロ。出力は 指摘ID・重要度・対象箇所・指摘内容・修正案 の表(なければ「レビュー指摘なし」)。

## 9.2 横断整合チェック(節目ごと)

フェーズ完了・リリース前・大きな変更後に docs/ 全体を走査し次を機械的に検証する(全 index.md・エラーメッセージ一覧.md(ERR)・各SCRインライン(MSG)・共通定義設計(DEF)を突合)。問題ゼロなら各項目の確認件数を記録、検出時は 対象ファイル・内容・修正案 を挙げて是正する。

| No | チェック | 検出方法 |
|---|---|---|
| 1 | ID の一意性 | 全一覧、エラーメッセージ一覧.md(ERR)、各SCR文書のインライン定義(MSG)で同一 ID が2回定義されていないか |
| 2 | 参照先の実在 | 本文中の全 ID(FR/CFR/NFR/MDL/UC/SCR/API/MOD/JOB/TBL/SQL/SEQ/DEF/ERR/MSG)が一覧・エラーメッセージ一覧.md・各SCRのインライン定義に存在するか |
| 3 | 一覧と文書の同期 | 一覧の行と文書の基本情報欄が一致するか。02_要件定義/01_機能要件/index.md は掲載列のみ確認する。文書のない ID 行・一覧にない文書がないか |
| 4 | 逆参照の混入 | 上流文書に下流 ID の一覧が書かれていないか |
| 5 | トレース鎖の断絶 | どの FR からも一覧経由で辿れない SCR/API/MOD/TBL/JOB がないか(共通基盤は除外を明記) |
| 6 | 定義箇所の重複 | ERR の文言・HTTPステータス、MSG 文言、区分値(DEF)、共通カラム、API 共通仕様が、定義箇所以外に再記載されていないか |
| 7 | ローカルIDの参照形式 | 他文書からの EVT/ALT/EXC/IDX 参照が完全修飾(SCR-XXX/EVT-01 形式)か |
| 8 | 状態の妥当性 | 状態欄を持つ文書で、「確定」文書に「未定」「要確認」が残っていないか |
| 9 | 外部ライブラリ直接利用禁止 | 外部ライブラリ利用専用モジュール以外の API・JOB・モジュールに外部ライブラリ・標準実行基盤APIの直接呼び出しが記載されていないか(必要な場合は MOD-ID 参照か。§8-14) |
| 10 | 処理結果定義 | 戻り値・レスポンスを返す処理に「処理結果」ブロックがあり返却項目表が定義されているか。例外でない無効・不一致が処理結果に定義されているか(§8-15) |
| 11 | フローチャートの簡潔性 | API・JOB・モジュールのフローのノード/エッジに呼び出し先・外部サービス名・ステータス値等の詳細が書かれていないか(詳細は処理詳細か。§8-16) |

# 10. 禁止事項

| No | 禁止事項 |
|---|---|
| 1 | 「基本設計」「詳細設計」という分類への逆戻り |
| 2 | 列挙型逆参照(自分を利用する側の一覧記載) |
| 3 | 共通設計(API-COM)・共通カラム・共通定義設計 の内容の再記載 |
| 4 | テンプレートにないセクションの独断追加・削除 |
| 5 | 確定文書を「作成中」に戻さずに変更すること |
| 6 | 「未定」「要確認」を根拠なく確定させること |
| 7 | 一覧を更新せずに文書だけを作成・変更すること |
| 8 | ユーザープロンプト・仕様の原文を要約で置き換えて記録すること |
