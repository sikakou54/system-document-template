# エンティティ定義：認証セッション

## 1. 基本情報
* **エンティティID:** MDL-009
* **概要:** 利用者のログイン認証に伴う一時的な認証状態。認証トークン(JWT)はクライアントが保持し、サーバーは失効させたトークン識別子だけを一時ストア(KV)で保持する
* **データ管理者:** システム管理部門

## 2. データ項目定義
当エンティティが保持する具体的なデータ項目（属性）の詳細を定義する。

| No | 属性名 (論理名) | データ型 | 長さ | 主キー | 必須 | 説明 (初期値など) |
|:---|:---|:---|:---:|:---:|:---|:---|
| 1 | 認証トークン識別子 | 文字列 | - | 〇 | 〇 | JWTごとに一意な識別子。JWTのクレームとしてクライアントが保持し、失効時は同じ値をKVへ登録する |
| 2 | 利用者ID | 数値 | - | - | 〇 | 認証対象の利用者（利用者(MDL-001)を参照）。JWTのクレームとして保持する |
| 3 | 発行日時 | 日時 | - | - | 〇 | JWTを発行した日時。JWTのクレームとして保持する |
| 4 | 有効期限 | 日時 | - | - | 〇 | JWTの有効期限。失効情報のKV保持期限にも用いる |
| 5 | 認証状態 | 数値 | 1 | - | 〇 | `1`:有効, `2`:失効, `3`:期限切れ。有効はJWT検証結果、失効はKV登録、期限切れは現在日時との比較で判定する |
| 6 | 失効日時 | 日時 | - | - | - | ログアウトまたは強制失効時に、トークン識別子とともにKVへ保持する |

## 3. データライフサイクル・制約事項（CRUD）
本データの発生から廃棄までのルールやデータ操作における制約を定義する。

| 操作 | ルール・制約 |
|:---|:---|
| 作成 (Create) | ログイン認証（CFR-001）の成功時にJWTを発行してクライアントへ返す。サーバーには有効JWTのセッションレコードを作成しない。ログアウト・強制失効時だけトークン識別子・失効日時・有効期限をKVへ登録する |
| 参照 (Read) | 各操作の認証・認可（CFR-001／CFR-002）でJWTの署名・有効期限と、KVに同一トークン識別子の失効情報がないことを確認する |
| 更新 (Update) | JWT自体は更新しない。失効後の状態変更も行わず、新たなログインでは別のトークン識別子を持つJWTを発行する |
| 削除 (Delete) | クライアントはログアウト時にJWTを破棄する。KVの失効情報はJWT有効期限の経過後にTTLで自動消失する |

## 4. 関連機能・業務ルール
本データが影響を与える機能や、依存する業務ルールを定義する。

**関連機能（機能要件）**

| 関連要件ID | 要件名 |
|:---|:---|
| CFR-001 | 認証 |
| CFR-002 | 認可・権限制御 |

**業務ルールへの影響**

| No | 内容 |
|:---|:---|
| 1 | 署名が正当で有効期限内であり、トークン識別子がKVの失効情報に存在しないJWTだけを認証に使用できる |
| 2 | 有効JWTのサーバー側セッションレコードは作成せず、失効情報だけを有効期限まで保持する |

## 5. セキュリティ・アクセス権限
本データに対する参照・更新権限や機密性を定義する。

| 項目 | 内容 |
|:---|:---|
| 参照権限 | JWTは保持する利用者のクライアントと認証処理のみが扱い、KV失効情報は認証基盤だけが参照する |
| 更新権限 | システムだけがJWTを発行し、KVへ失効情報を登録する。利用者・管理者が値を直接更新することはできない |
| 機密性レベル | 高（JWTの漏えいはなりすましにつながるため、暗号化通信、有効期限、失効確認で保護する。KVにはJWT本文を保存しない） |
