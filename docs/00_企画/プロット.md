# 1. 概要

MeetRoom の企画・構想を定義する。
本書は要求定義より上位の正本であり、プロダクトとして何を解決し、誰にどのような価値を届けるかを明確にする。

# 2. 企画の骨子

| 項目 | 内容 |
|---|---|
| 企画名 | MeetRoom |
| 種別 | 社内向け会議室予約プロダクト |
| 一言説明 | 社内会議室の空き状況、予約、利用状況を分かりやすく管理するプロダクト |
| 主な利用者 | 会議室を利用する社員、会議室を管理する総務・管理部門、組織の管理者 |
| 企画目的 | 社内会議室の利用・管理にかかる確認、調整、問い合わせの負担を減らし、会議室を必要な人が使いやすい状態にする |

# 3. 背景

| No | 背景 |
|---|---|
| 1 | 会議室ごとの空き状況、収容人数、設備を確認する手間が大きい |
| 2 | 同じ会議室に同じ時間帯の予約が入り、会議直前の調整が発生する |
| 3 | 会議室や設備の情報が担当者の把握に依存し、問い合わせが増える |
| 4 | 会議室の利用頻度や偏りが分からず、改善判断がしづらい |
| 5 | 予約した会議室を使い忘れることで、会議室が使われないまま占有されることがある |
| 6 | 有料会議室の費用を利用量に応じて公平に扱う必要がある |

# 4. 解決したい課題

| 課題 | 課題の内容 | 影響を受ける人 |
|---|---|---|
| 空き状況が分かりにくい | 会議室ごとの空き時間、収容人数、設備を確認する手間が大きい | 社員 |
| 予約が重なる | 同じ会議室に同じ時間帯の予約が入り、会議直前の調整が発生する | 社員、総務・管理部門 |
| 管理が人に依存する | 会議室や設備の情報が担当者の把握に依存し、問い合わせが増える | 総務・管理部門 |
| 利用実態が見えない | 会議室の利用頻度や偏りが分からず、改善判断がしづらい | 総務・管理部門、組織の管理者 |
| 予約忘れが発生する | 予約した会議室が使われないまま占有される | 社員、組織 |
| 費用負担が不透明になる | 有料会議室の利用量と費用の関係が利用者に分かりにくい | 社員、組織 |

# 5. 解決仮説

| No | 解決仮説 | 対応する課題 |
|---|---|---|
| 1 | 日時、人数、設備などの条件から、利用できる会議室を探せるようにする | 空き状況が分かりにくい |
| 2 | 同じ会議室に同じ時間帯の予約が入らないようにする | 予約が重なる |
| 3 | 会議室、収容人数、設備、利用単価、利用状態を一元管理する | 管理が人に依存する |
| 4 | 会議室ごとの利用実績を確認できるようにする | 利用実態が見えない |
| 5 | 予約開始前に必要な通知を行う | 予約忘れが発生する |
| 6 | 有料会議室の利用量と請求状況を利用者が確認できるようにする | 費用負担が不透明になる |

# 6. 対象範囲

| 区分 | 内容 |
|---|---|
| 対象 | 会議室検索、会議室予約、予約変更・キャンセル、予約リマインド、会議室・設備管理、利用実績確認、有料会議室の利用量・請求確認 |
| 対象外 | 社外向け会議室予約、会議そのものの運営、会議資料管理、参加者の日程調整、会議室以外の設備貸出 |

# 7. 期待する状態

| 対象 | 期待する状態 |
|---|---|
| 社員 | 条件に合う会議室を短時間で見つけ、安心して予約できる |
| 総務・管理部門 | 会議室や設備の情報を管理しやすくなり、問い合わせや個別調整が減る |
| 組織の管理者 | 利用状況をもとに、会議室数や設備の見直しを判断できる |
| 組織全体 | 会議室に関する確認・調整の時間が減り、会議準備に集中できる |

# 8. 要求定義への接続

| 項目 | 内容 |
|---|---|
| 接続方針 | 本書の背景、課題、解決仮説、期待する状態から、要求定義で利用者・管理者・組織が求めることを抽出する |
| 書かない内容 | 要求ID、要件ID、画面、API、DB、モジュール、外部サービス、物理名、処理方式などの後続工程の情報 |
| 更新ルール | 企画・構想が変わった場合は本書を先に更新し、要求定義に影響する変更を確認する |
