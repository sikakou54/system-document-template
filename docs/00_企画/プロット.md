# 1. 概要

MeetRoom の企画・構想を定義する。
本書は要求定義より上位の正本であり、ターゲットと課題(Who/Why)・提供する価値とゴール(What)・体験プロセス(UX)・手段と優先順位(How)の4観点で「何を・誰に・なぜ作るのか」を明確にする。

# 2. 企画の骨子

| 項目 | 内容 |
|---|---|
| 企画名 | MeetRoom |
| 種別 | 社内向け会議室予約プロダクト |
| 一言説明 | 社内会議室の空き状況、予約、利用状況を分かりやすく管理するプロダクト |
| 主な利用者 | 会議室を利用する社員、会議室を管理する総務・管理部門、組織の管理者 |
| 企画目的 | 社内会議室の利用・管理にかかる確認、調整、問い合わせの負担を減らし、会議室を必要な人が使いやすい状態にする |

# 3. ターゲットと課題（Who / Why）

## 3.1 背景

| No | 背景 |
|---|---|
| 1 | 会議室ごとの空き状況、収容人数、設備を確認する手間が大きい |
| 2 | 同じ会議室に同じ時間帯の予約が入り、会議直前の調整が発生する |
| 3 | 会議室や設備の情報が担当者の把握に依存し、問い合わせが増える |
| 4 | 会議室の利用頻度や偏りが分からず、改善判断がしづらい |
| 5 | 予約した会議室を使い忘れることで、会議室が使われないまま占有されることがある |
| 6 | 有料会議室の費用を利用量に応じて公平に扱う必要がある |

## 3.2 ペルソナ

| ペルソナ | 状況 | 抱えるペイン |
|---|---|---|
| 会議室を予約する社員 | 会議のたびに空いている会議室を探して予約する | 空き状況・収容人数・設備の確認に手間がかかり、予約が重なって直前調整が発生する |
| 会議室を管理する総務・管理部門 | 会議室・設備・利用単価を管理し、利用者からの問い合わせに対応する | 情報が担当者依存で問い合わせが多く、重複予約の調整に追われる |
| 会議室を評価する組織の管理者 | 会議室の数・コスト・利用状況を見直したい | 利用実態が見えず、増設・削減・単価見直しの判断ができない |

## 3.3 解決したい課題

| 課題 | 課題の内容 | 影響を受ける人 |
|---|---|---|
| 空き状況が分かりにくい | 会議室ごとの空き時間、収容人数、設備を確認する手間が大きい | 社員 |
| 予約が重なる | 同じ会議室に同じ時間帯の予約が入り、会議直前の調整が発生する | 社員、総務・管理部門 |
| 管理が人に依存する | 会議室や設備の情報が担当者の把握に依存し、問い合わせが増える | 総務・管理部門 |
| 利用実態が見えない | 会議室の利用頻度や偏りが分からず、改善判断がしづらい | 総務・管理部門、組織の管理者 |
| 予約忘れが発生する | 予約した会議室が使われないまま占有される | 社員、組織 |
| 費用負担が不透明になる | 有料会議室の利用量と費用の関係が利用者に分かりにくい | 社員、組織 |

# 4. 提供する価値とゴール（What）

## 4.1 提供価値・解決仮説

| No | 提供価値（解決仮説） | 対応する課題 |
|---|---|---|
| 1 | 日時、人数、設備などの条件から、利用できる会議室を探せるようにする | 空き状況が分かりにくい |
| 2 | 同じ会議室に同じ時間帯の予約が入らないようにする | 予約が重なる |
| 3 | 会議室、収容人数、設備、利用単価、利用状態を一元管理する | 管理が人に依存する |
| 4 | 会議室ごとの利用実績を確認できるようにする | 利用実態が見えない |
| 5 | 予約開始前に必要な通知を行う | 予約忘れが発生する |
| 6 | 有料会議室の利用量と請求状況を利用者が確認できるようにする | 費用負担が不透明になる |

## 4.2 期待する状態

| 対象 | 期待する状態 |
|---|---|
| 社員 | 条件に合う会議室を短時間で見つけ、安心して予約できる |
| 総務・管理部門 | 会議室や設備の情報を管理しやすくなり、問い合わせや個別調整が減る |
| 組織の管理者 | 利用状況をもとに、会議室数や設備の見直しを判断できる |
| 組織全体 | 会議室に関する確認・調整の時間が減り、会議準備に集中できる |

## 4.3 開発者のゴール

| 区分 | ゴール |
|---|---|
| 収益 | 会議室運用にかかる工数削減を定量化し、有料会議室の従量課金（月次請求）で運用コストを賄える状態にする |
| 技術力向上 | Cloudflare(Workers / D1 / KV / Queues / Cron Triggers) ＋ Stripe(従量課金) ＋ Resend(メール送信) の構成を実装・運用して習得する |
| 実績づくり | 企画〜要求〜要件〜機能設計〜実装まで一貫したドキュメント駆動開発の成果物として公開する |

# 5. 体験プロセス（UX）

| ステップ | 利用者の行動 | 得られる状態 |
|---|---|---|
| 1 認知 | 会議室予約が必要になり MeetRoom にアクセスして本人確認する | 登録利用者として会議室を扱う入口に立てる |
| 2 検索 | 利用日時・人数・設備を条件に空き会議室を探す | 条件に合う会議室の候補が分かる |
| 3 予約 | 候補から会議室を選び、日時とタイトルを指定して予約する | 会議室が自分の会議のために確保される |
| 4 リマインド | 予約開始前にリマインド通知を受け取る | 予約を忘れず会議室を無駄にしない |
| 5 見直し | 予定変更に応じて自分の予約を変更・キャンセルする | 予約を最新に保ち、空いた時間帯を他の人が使える |
| 6 管理・振り返り | （総務・管理部門／管理者）会議室・設備を管理し、利用実績・請求を確認する | 情報を最新に保ち、利用状況をもとに見直しを判断できる |

# 6. 手段と優先順位（How）

## 6.1 対象範囲

| 区分 | 内容 |
|---|---|
| 対象 | 会議室検索、会議室予約、予約変更・キャンセル、予約リマインド、会議室・設備管理、利用実績確認、有料会議室の利用量・請求確認 |
| 対象外 | 社外向け会議室予約、会議そのものの運営、会議資料管理、参加者の日程調整、会議室以外の設備貸出 |

## 6.2 コア機能（MVP）

MVP は「登録済み利用者が、無料会議室を安全に検索・予約できる」ことの最小成立とする。

| No | コア機能 | なぜ必須か |
|---|---|---|
| 1 | 空き会議室検索 | 中核体験の入口であり、会議室を確保する前提になる |
| 2 | 会議室予約（重複防止） | 中核体験そのもの。同一会議室・同一時間帯の二重予約を防いで確実に確保する |
| 3 | 認証・認可 | 登録利用者だけが権限内で安全に利用できる前提を担保する |
| 4 | 入力検証・データ保護 | 不正・誤入力の混入を防ぎ、業務データと認証情報を守る |
| 5 | 会議室・設備管理 | 検索・予約の対象となる会議室・設備・利用状態を正しく整える前提になる |

## 6.3 見送り要件

| No | 見送る要件 | 理由 |
|---|---|---|
| 1 | 有料会議室の従量課金・月次請求 | 無料会議室で中核体験（検索・予約）を先に検証すればよく、MVP検証には不要 |
| 2 | 利用実績レポート | 管理判断を高度化する機能で、中核体験の成立後でよい |
| 3 | 予約リマインド通知 | あると良いが、まず検索・予約の成立を優先する |
| 4 | 予約変更・キャンセルの拡張 | 初回は最小の予約成立を優先し、見直し機能は後続で拡充する |
