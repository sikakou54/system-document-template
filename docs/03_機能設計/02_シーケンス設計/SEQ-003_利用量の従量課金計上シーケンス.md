# 3. シーケンス

本シーケンスは、利用終了した有料会議室予約を完了とし、予約単位の利用量を重複なく記録・送信したうえで、月次請求の支払結果を請求状態へ冪等に反映する連携を扱う。根拠は FR-008/UC-03(完了予約の利用量を自動計上する)と FR-008/UC-04(月次請求を確定し請求結果を反映する)であり、両UCの状態パターン(UC-03/SP-1〜SP-8、UC-04/SP-1〜SP-7)を下表で本シーケンスの分岐へ対応づける。なお FR-008/UC-01(支払い方法登録)は支払い方法登録シーケンス(SEQ-004)、FR-008/UC-02(利用量・請求確認)は利用量・請求確認シーケンス(SEQ-011)の対象範囲のため、本シーケンスの対象外とする。

| パターンID | 状態パターン(条件) | 本シーケンスでの表現 |
|---|---|---|
| FR-008/UC-03/SP-1 | 有料・予約済/終了済・利用量なし→送信成功 | 3.1 正常系(予約完了・利用量記録・Meter Event送信) |
| FR-008/UC-03/SP-2 | 無料(予約確定時単価0)→利用量を作成しない | 3.2 代替系(無料会議室) |
| FR-008/UC-03/SP-3 | 完了・計上済→追加なしで正常終了 | 3.2 代替系(有料会議室かつ利用量記録あり) |
| FR-008/UC-03/SP-4 | 完了・失敗記録→再送成功で計上済へ | 3.3 例外系(Meter Event送信失敗)の保持記録が次回実行で 3.1 正常系として計上済になる |
| FR-008/UC-03/SP-5 | 送信失敗→記録を失敗状態で保持 | 3.3 例外系(Meter Event送信失敗) |
| FR-008/UC-03/SP-6 | 予約完了の保存失敗→未完了のまま | 4.3 TX-01(予約完了更新失敗時は未完了で次回再抽出) |
| FR-008/UC-03/SP-7 | 外部受付後の状態保存失敗→同一計上要求で再試行 | 4.3 TX-03(送信状態更新失敗時は同じ固定キーで再送) |
| FR-008/UC-03/SP-8 | 予約完了後に利用量記録保存失敗→次回再抽出 | 3.3 例外系(予約完了後に利用量保存失敗) |
| FR-008/UC-04/SP-1 | 対象月利用量あり・支払成功→支払済で反映 | 3.1 正常系(支払成功Webhookの反映) |
| FR-008/UC-04/SP-2 | 対象月利用量0件→請求を作成しない | Stripe側で請求が確定せず本SEQへ反映イベントが到達しない(対象外) |
| FR-008/UC-04/SP-3 | 同一イベント再受信→変更せず正常終了 | 3.1 正常系(Webhook Inbox の Event ID一意受付による重複排除)／4.3 TX-04 |
| FR-008/UC-04/SP-4 | 支払失敗→請求を失敗で反映 | 3.3 例外系(請求支払失敗のWebhook) |
| FR-008/UC-04/SP-5 | 検証不能・対象外→既存請求を変更しない | 3.3 例外系(Webhookの検証または状態反映失敗)／COND-05不成立時は正常終了 |
| FR-008/UC-04/SP-6 | 妥当だが保存失敗→部分更新せず再処理可能 | 3.3 例外系(状態反映失敗→Inboxを失敗として再処理可能にする) |
| FR-008/UC-04/SP-7 | 支払済への遅延イベント→巻き戻さず変更なし | 3.1 正常系(新しいイベントの場合だけ請求へ反映)／4.3 TX-04 |

## 3.1 正常系シーケンス

有料会議室の利用量送信と、後続の月次請求成功通知を示す。

```mermaid
sequenceDiagram
    participant CRON as Cloudflare Cron Trigger
    participant JOB as JOB-002 従量課金計上ジョブ
    participant RES as MOD-003 予約サービス
    participant BILL as MOD-007 課金サービス
    participant DB as データベース
    participant STR as Stripe
    participant API as API-011 Stripe Webhook API

    CRON->>JOB: 定期実行を起動する
    JOB->>RES: 完了・再処理対象取得を依頼する
    RES->>DB: 終了予約と完了済み利用量未作成を参照する（予約, 利用量記録）
    DB-->>RES: 予約確定時単価を含む対象予約を返す
    RES-->>JOB: 対象予約を返す
    loop 対象予約ごと
        opt 予約済み
            JOB->>RES: 予約完了条件付き更新を依頼する
            RES->>DB: 予約済み・終了済みなら完了へ更新する（予約）
            DB-->>RES: 完了後予約を返してTXを確定する
        end
        JOB->>BILL: 予約IDと予約確定時単価で利用量計上を依頼する
        BILL->>DB: 固定Stripe冪等キー付き利用量を条件付き登録する（利用量記録）
        DB-->>BILL: 利用量記録を返してTXを確定する
        BILL->>DB: Stripe送信に必要な利用者情報を参照する（利用者）
        DB-->>BILL: 利用者情報を返す
        BILL->>STR: 固定冪等キーでMeter Eventを送信する
        STR-->>BILL: 送信成功を返す
        BILL->>DB: 送信済み状態と外部イベントIDを更新する（利用量記録）
    end
    BILL-->>JOB: 完了件数・成功件数・失敗件数を返す
    Note over STR: 月末に利用量を集計して請求を確定する
    STR-)API: 支払成功のWebhookを通知する
    API->>BILL: Webhookの検証と反映を依頼する
    BILL->>DB: Event IDを処理中として一意受付する（Webhook Inbox）
    BILL->>DB: 新しいイベントの場合だけ請求を支払済みで登録または更新する（請求）
    BILL->>DB: Webhookを処理済みに更新する（Webhook Inbox）
    BILL-->>API: 反映成功を返す
    API-->>STR: 受信成功を返す
```

## 3.2 代替系シーケンス

無料会議室、またはすでに利用量記録が存在する予約は重複計上しない。

```mermaid
sequenceDiagram
    participant JOB as JOB-002 従量課金計上ジョブ
    participant RES as MOD-003 予約サービス
    participant BILL as MOD-007 課金サービス
    participant DB as データベース

    JOB->>RES: 完了・再処理対象取得を依頼する
    RES->>DB: 対象予約を参照する（予約, 利用量記録）
    alt 予約確定時単価が0
        DB-->>RES: 無料予約を返す
        Note over JOB: 利用量記録を作成せず計上対象外とする
    else 有料予約かつ利用量記録あり
        JOB->>BILL: 利用量計上を依頼する
        BILL->>DB: 予約IDで利用量を条件付き登録する（利用量記録）
        DB-->>BILL: 既存記録を返す
        Note over BILL: 新たな利用量記録・送信を作成しない
    end
    BILL-->>JOB: 重複計上なしの処理結果を返す
```

## 3.3 例外系シーケンス

```mermaid
sequenceDiagram
    participant JOB as JOB-002 従量課金計上ジョブ
    participant RES as MOD-003 予約サービス
    participant BILL as MOD-007 課金サービス
    participant DB as データベース
    participant STR as Stripe
    participant API as API-011 Stripe Webhook API

    alt Meter Event送信失敗
        JOB->>BILL: 利用量計上を依頼する
        BILL->>STR: 固定冪等キーでMeter Eventを送信する
        STR-->>BILL: 送信失敗を返す
        BILL->>DB: 送信失敗状態へ更新する（利用量記録）
        BILL-->>JOB: 計上失敗を返す
        Note over JOB,BILL: 当該記録と固定キーを保持し、後続実行で再送する
    else 予約完了後に利用量保存失敗
        JOB->>RES: 予約完了条件付き更新を依頼する
        RES->>DB: 予約を完了へ更新してTXを確定する（予約）
        JOB->>BILL: 利用量計上を依頼する
        BILL->>DB: 利用量記録を登録する（利用量記録）
        DB-->>BILL: DB失敗を返す
        Note over JOB,RES: 予約を予約済みに戻さず、次回「完了済み・利用量なし」で再抽出する
    else 請求支払失敗のWebhook
        STR-)API: 支払失敗のWebhookを通知する
        API->>BILL: Webhookの検証と反映を依頼する
        BILL->>DB: Event IDを一意受付し請求を支払失敗で登録または更新する（Webhook Inbox, 請求）
        BILL-->>API: 反映成功を返す
        API-->>STR: 受信成功を返す
    else Webhookの検証または状態反映失敗
        STR-)API: Webhookを通知する
        API->>BILL: Webhookの検証と反映を依頼する
        BILL->>DB: Webhookを処理失敗に更新する（Webhook Inbox）
        BILL-->>API: 反映エラーを返す
        API-->>STR: エラー結果を返す
    end
```

# 4. 連携定義

## 4.1 条件分岐

| 条件ID | 判定箇所 | 条件 | 成立時 | 不成立時 | 根拠 |
|---|---|---|---|---|---|
| COND-01 | MOD-003 | 予約済みで終了日時経過、または完了済み・利用量記録なしである | 完了更新または利用量再作成を行う | 対象外 | FR-008 業務ルール3, 8 |
| COND-02 | JOB-002 | 予約確定時単価が0より大きい | 利用量を計上 | 無料として計上しない | FR-008 業務ルール1, 2 |
| COND-03 | MOD-007 | 同一予約の利用量記録が存在しない | 利用量記録を作成 | 重複計上せず既存状態に従う | FR-008 業務ルール3 |
| COND-04 | MOD-007 | Meter Event送信に成功 | 送信済みに更新 | 送信失敗に更新し計上失敗 | FR-008 業務ルール6 |
| COND-05 | MOD-007 | Webhookが正当で対象イベントである | イベント種別に応じ状態反映 | 不正時は反映エラー、対象外は正常終了 | FR-008 業務ルール6 |

## 4.2 データ参照・更新

| データモデル | CRUD | 目的 | 実行主体 |
|---|---|---|---|
| MDL-003 予約 | R / U | 終了・中間失敗予約の抽出、予約確定時単価参照、完了更新 | MOD-003 |
| MDL-001 利用者 | R | Stripe送信に必要な利用者情報取得 | MOD-007 |
| MDL-006 利用量記録 | R / C / U | 二重計上確認、利用量登録、送信状態更新 | MOD-007 |
| MDL-007 請求 | C / U | Stripe請求結果の冪等な反映 | MOD-007 |
| MDL-011 Webhook受信イベント | R / C / U | Event ID一意受付、重複・失敗再処理・処理結果記録 | MOD-007 |

## 4.3 トランザクション境界

| 境界ID | 開始 | 終了 | 対象更新 | ロールバック条件 | 管理主体 |
|---|---|---|---|---|---|
| TX-01 | 予約完了条件付き更新 | 完了状態確定 | MDL-003 | DB更新失敗。完了済みは正常継続 | MOD-003 |
| TX-02 | 利用量記録条件付き登録 | 固定冪等キーを含む記録確定 | MDL-006 | DB更新失敗。TX-01は戻さず次回再抽出 | MOD-007 |
| TX-03 | Meter Eventの送信結果受領 | 送信済みまたは送信失敗状態の確定 | MDL-006 | 送信状態更新失敗。同じ固定キーで再送 | MOD-007 |
| TX-04 | Webhook Event ID受付 | 状態反映・Inbox処理済みの確定 | MDL-007, MDL-011 | 状態反映失敗時はInboxを失敗として再処理可能にする | MOD-007 |

- 外部サービスの応答待ちをDBトランザクションに含めず、利用量記録と送信結果を分けて短く確定する。

## 4.4 補足事項

| 観点 | 内容 |
|---|---|
| 同期/非同期 | Cron起動の非対話処理。StripeからAPI-011へのWebhookは非同期。 |
| 冪等性・再試行 | 予約完了の期待状態、予約ID一意の利用量記録、送信前固定Stripeキー、Webhook Event ID一意制約を併用する。 |
| 排他制御 | 予約完了、利用量記録、外部送信結果を別TXにし、完了済み・利用量なしを再抽出して中間失敗を回復する。 |
| 外部連携 | Cloudflare Cron Triggerで起動し、StripeへMeter Eventを送信する。Stripeは月次請求後にAPI-011へ結果を通知する。 |
