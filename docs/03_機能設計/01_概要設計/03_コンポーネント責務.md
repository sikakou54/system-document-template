# 1. 目的

MeetRoom の業務能力を技術レイヤーから独立して分割し、各コンポーネントの責務・非責務、所有データ、書込主体、依存方向、トランザクション管理主体を定義する。

# 5. 依存規則

1. UI→API→MOD→SQL→TBL の方向だけを許可する。API/JOBからSQL/TBLへ直接アクセスしない。
2. APIは原則1つのユースケースMODを呼ぶ。複数業務能力の調整はTX主体となるMODが行う。
3. MOD間呼出は公開インターフェースだけを利用し、他MODの所有テーブルを直接更新しない。
4. SQLは期待所有者・期待状態・現在時刻・重複なし等をWHERE/SELECT条件に含め、判定と更新の間の競合を防ぐ。
5. D1(SQLite)に行ロック構文は用いない。原子的な条件付きINSERT/UPDATEと更新件数で競合を判定する。
6. 外部サービス呼出をD1トランザクションに含めない。送信前記録、外部送信、結果更新を分け、固定冪等キーで再実行可能にする。
7. Cronは処理対象メッセージをQueueへ投入し、Queue ConsumerがMODを呼ぶ。再試行・DLQはQueue境界が管理する。

# 6. 境界変更時の確認

- Writerを追加する場合は本書、対象SEQ、MOD/TBL/SQL、トレーサビリティを同時更新する。
- TXをまたぐ外部連携には、冪等キー、再試行、応答消失、重複配送、補償の設計を必須とする。
- シーケンスで新たな責務が必要になった場合、SEQへ直接埋め込まず本書を先に変更する。
