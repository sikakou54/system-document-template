# 1. 目的

Step 7のシーケンス検証より前に、MeetRoom のAPI、JOB/Queue、外部サービス連携について実装方式に依存しない契約ベースラインを確定する。HTTPの共通物理規約は API-COM、個別項目は各API設計を正本とする。

# 2. 共通契約

| 観点 | 契約 |
|---|---|
| 相関 | 入口で相関IDを受理または生成し、API、Queue、MOD、外部連携、運用ログへ伝播する |
| API応答期限 | UIのクライアントタイムアウトは10秒。タイムアウト後の副作用有無は冪等キーで再確認できること |
| 外部呼出期限 | Stripe/Resendの1回の呼出タイムアウトは5秒。外部呼出をDBトランザクションに含めない |
| 冪等受付 | 副作用を持つPOSTはIdempotency-Key必須。ログインは対象外、Stripe WebhookはEvent IDを使用 |
| 再試行 | 同期APIはサーバ側で自動再試行しない。Queue Consumerと外部送信だけが、タイムアウト・429・一時的5xxを指数バックオフ付きで最大3回再試行する |
| 互換性 | 現行内部APIをv1とする。v1内は後方互換な追加のみ。削除・型変更・意味変更は `/api/v2` で提供する |
| 一般失敗 | 認証、認可、入力、競合、冪等キー衝突、レート制限、タイムアウト、依存先停止、予期しない内部障害を共通エラーへ写像する |

# 6. シーケンス検証への引渡し条件

各SEQは、正常・代替・業務例外に加え、次を最低1つずつ検証する。

- 同一Idempotency-KeyまたはEvent IDの重複
- DB更新成功後のレスポンス消失と再送
- 外部送信成功後の結果更新失敗
- Queue再配送とDLQ到達
- 同じ予約に対する同時登録・変更・キャンセル・完了
- 外部イベントの順不同
- DBロールバックまたは外部副作用の補償不能時の再実行
