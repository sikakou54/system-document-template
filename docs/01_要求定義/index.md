# 1. 概要

MeetRoom に対して利用者・管理者・組織が求めることを、要求(RQ)として定義する。
本ディレクトリは 00_企画/プロット.md で定義した企画・構想を受け、要求定義として利用者・管理者・組織が求めることを整理する。

# 2. 定義ルール

| 項目 | 内容 |
|---|---|
| 目的 | 利用者・管理者・組織が何を求めているかを、実現手段に寄せずに定義する |
| 粒度 | 受入条件・業務ルール・ユースケース上の期待結果を、単独で充足可否を判断できる詳細要求に分解する |
| 網羅範囲 | 業務課題、利用者の期待結果、利用シナリオ、品質・運用上の期待から読み取れる要求を対象とする |
| 上流文書 | 00_企画/プロット.md |
| 書かない内容 | 画面、API、DB、モジュール、外部サービス、物理名、処理方式などの設計詳細 |
| 正本 | 要求一覧は本ファイルを正本とする。個別RQファイルは、補足説明が必要な要求に限り作成する |

# 3. 要求領域

| 領域 | 内容 |
|---|---|
| 会議室検索 | 条件に合う空き会議室を探し、候補の有無を理解できること |
| 会議室予約 | 空き会議室を確実に予約し、重複・不正な予約を防ぐこと |
| 予約変更・キャンセル | 予定変更に応じて自分の予約を変更・キャンセルできること |
| リマインド通知 | 予約忘れによる空押さえを減らすこと |
| 会議室・設備管理 | 会議室、設備、単価、利用状態を正しく保つこと |
| 利用実績 | 会議室の利用状況を管理判断に使えること |
| 認証・認可・共通制御 | 登録利用者だけが権限内で安全に利用できること |
| 従量課金 | 有料会議室の費用を利用量に応じて公平に請求すること |
| 品質・保護・監査 | 業務時間中の安定利用、データ保護、重要操作の追跡を実現すること |

# 5. 要求間の関係

| 関係元 | 関係先 | 関係 |
|---|---|---|
| RQ-001〜RQ-008 | RQ-009〜RQ-014 | 会議室を探せることが予約成立の前提になる |
| RQ-009〜RQ-014 | RQ-015〜RQ-023 | 予約成立後に変更・キャンセルできることが会議室の有効活用につながる |
| RQ-009〜RQ-014 | RQ-024〜RQ-028 | 予約成立後にリマインドすることで空押さえを減らす |
| RQ-029〜RQ-038 | RQ-001〜RQ-014, RQ-057〜RQ-066 | 会議室・設備・単価・利用状態の正確性が検索・予約・課金の前提になる |
| RQ-039〜RQ-044 | RQ-029〜RQ-038 | 利用状況をもとに会議室・設備・運用の見直し判断を行う |
| RQ-045〜RQ-056 | RQ-001〜RQ-067 | 認証・認可・入力検証・エラー表示・一覧制御が各機能の共通前提になる |
| RQ-057〜RQ-067 | RQ-009〜RQ-014, RQ-029〜RQ-038 | 有料会議室の予約・単価設定を前提に利用量と請求を管理する |
| RQ-068〜RQ-073 | RQ-001〜RQ-067 | 品質・保護・監査の要求としてシステム全体を支える |
