# 要求ID・要件ID・設計ID 採番ルール

本ファイルは MeetRoom における要求ID・要件ID・設計IDの採番とファイル命名の**正本**である。
ID採番・命名に関する判断は本ファイルを参照する（作業全体のルール正本は `docs/CLAUDE.md`）。

---

# 0. 本プロジェクトでの適用（採用時の調整）

汎用の採番ルールを本プロジェクトへ適用するにあたり、以下の点を調整して採用する。

| No | 調整 | 内容 |
| -- | --- | --- |
| 1 | プロジェクト拡張ID | 汎用ルールに無い `MOD-XXX`(モジュール)・`JOB-XXX`(ジョブ)・`DEF-XXX`(共通定義)・`MDL-XXX`(データモデル)を**プロジェクト拡張ID**として正式採用する（§3の表に記載）。 |
| 2 | ディレクトリ構成 | §14 の推奨構成（英語・フラット）は**不採用**。本プロジェクトは現行の日本語・工程別ディレクトリ（`00_企画/`・`01_要求定義/`・`02_要件定義/`・`03_機能設計/`）を正とする（§14参照）。 |
| 3 | CFR-000 の配置 | `CFR-000_システム共通要件.md` は `02_要件定義/01_機能要件/` 直下に配置する（現行ディレクトリ維持のため）。 |
| 4 | CFR の個別文書化 | 共通機能要件(CFR-XXX)は `CFR-000` 内のエントリではなく、FR と同様に **`CFR-XXX_名称.md` の個別文書**とする。`CFR-000_システム共通要件.md` は、システム全体の要件・CFR一覧・共通業務ルール(RULE-COM)を統括する親ファイルとする（§3・§5・§6）。 |
| 5 | NFR の個別文書化 | 非機能要件(NFR-XXX)は `CFR-000` 内ではなく、`02_要件定義/02_非機能要件/` に FR と同様の個別文書 `NFR-XXX_名称.md` として定義する（一覧は同ディレクトリ index.md）。機能横断・特定機能固有のいずれの NFR も同ディレクトリで管理する（§8）。 |

---

# 1. 目的

本ルールは、要求定義・要件定義・システムユースケース・機能設計におけるID採番とファイル命名を統一し、以下を実現することを目的とする。

| 目的         | 内容                         |
| ---------- | -------------------------- |
| 一意性の確保     | 各要求・要件・設計成果物をIDで一意に識別できるようにする |
| 変更耐性の確保    | 名称変更・並び替え・追加・削除に強いID体系にする  |
| 粒度分離       | 要求、要件、ルール、設計成果物の粒度をIDで混在させない  |
| 重複防止       | 共通要件を個別要件に重複記載しない          |
| トレーサビリティ確保 | 企画から要求、要件、設計、実装、テストへ流れを追えるようにする   |

---

# 2. 基本方針

| No | 方針                 | 内容                                       |
| -- | ------------------ | ---------------------------------------- |
| 1  | IDは一意にする           | 同じIDを複数の要件・設計に使用しない                      |
| 2  | IDは再利用しない          | 削除・廃止したIDは欠番として扱い、再利用しない                 |
| 3  | IDは意味で維持する         | 名称が変わっても、要件の意味が同じであればIDは変更しない            |
| 4  | 並び替えで振り直さない        | 表示順やファイル順を変えるためにIDを振り直さない                |
| 5  | 上位要件に下位IDを埋め込みすぎない | 画面ID、API ID、DB IDなどはトレーサビリティ表で管理する       |
| 6  | 共通要件は一元管理する        | 複数機能に共通する要件は `CFR-000_システム共通要件.md` に集約する |
| 7  | 個別要件は差分だけ書く        | 個別FRには、その機能に固有の要件のみ記載する                  |

| 種別           | ID形式           | 用途                   | 主な記載先                 |
| ------------ | -------------- | -------------------- | --------------------- |
| 要求           | `RQ-XXX`       | 利用者・管理者・組織が求めること | `01_要求定義/要求一覧.md` |
| システム共通要件ファイル | `CFR-000`      | 共通要件を統括する親ファイル（全体要件・CFR一覧・RULE-COM） | `CFR-000_システム共通要件.md` |
| 共通機能要件       | `CFR-XXX`      | 認証、認可、入力検証などの共通機能   | `CFR-XXX_名称.md`（一覧は `CFR-000`） |
| 個別機能要件       | `FR-XXX`       | 個別機能が実現すべき内容         | `FR-XXX_機能名.md`       |
| 非機能要件        | `NFR-XXX`      | 性能、可用性、セキュリティ、監査など   | `02_要件定義/02_非機能要件/NFR-XXX_名称.md` |
| データモデル ※拡張   | `MDL-XXX`      | エンティティごとのデータ定義書(データ項目・CRUD・関連機能・セキュリティ) | `02_要件定義/03_データモデル/MDL-XXX_○○情報.md` |
| 業務ルール        | `RULE-XXX`     | 業務上の判断基準・制約          | 業務ルール定義／該当FR          |
| 共通業務ルール      | `RULE-COM-XXX` | 複数機能に共通する業務ルール       | `CFR-000_システム共通要件.md` |
| システムユースケース   | `UC-01`（機能要件ごと連番・完全修飾 `FR-XXX/UC-01`） | 利用者・システムの振る舞い(シナリオ) | 各FR/CFR文書のシステムユースケース節 |
| 画面           | `SCR-XXX`      | 画面単位の設計              | 画面設計                  |
| 画面イベント       | `EVT-XX`       | ボタン押下などの画面操作（SCR文書内ローカル） | 画面イベント設計          |
| API          | `API-XXX`      | API単位の設計             | API設計                 |
| モジュール ※拡張    | `MOD-XXX`      | サービス・ドメインロジック・内部処理   | モジュール設計               |
| ジョブ ※拡張      | `JOB-XXX`      | バッチ・定期実行・非同期処理       | JOB設計                 |
| 共通定義 ※拡張    | `DEF-XXX`      | 共通コード・区分値・定数           | 共通定義設計             |
| テーブル         | `TBL-XXX`      | テーブル単位の設計            | DB設計                  |
| クエリ          | `SQL-XXX`      | クエリ単位の設計             | クエリ設計                 |
| シーケンス        | `SEQ-XXX`      | 複数要素の時系列連携           | シーケンス設計               |
| エラー          | `ERR-XXX`      | エラー定義                | `03_機能設計/03_API設計/エラーメッセージ一覧.md` |
| メッセージ        | `MSG-XXX`      | 表示・通知・エラーメッセージ       | 各SCR文書の「メッセージ一覧」節にインライン定義 |

※「拡張」は本プロジェクト固有のプロジェクト拡張ID（§0-1）。汎用ルールには含まれない。

# 4. 採番形式

IDは原則として以下の形式とする。

```text
<接頭辞>-<3桁連番>
```

例：

```text
RQ-001
CFR-001
FR-001
NFR-001
MDL-001
RULE-001
SCR-001
API-001
MOD-001
JOB-001
DEF-001
TBL-001
```

システムユースケース(UC)は機能要件ごとのローカル連番とし、`UC-01` 形式で採番する（各FR/CFR内で番号をリセット。他文書からは完全修飾 `FR-XXX/UC-01` で参照する）。3桁連番の対象外。

ただし、共通業務ルールは以下の形式とする。

```text
RULE-COM-001
RULE-COM-002
RULE-COM-003
```

# 5. システム共通要件のルール

## 5.1 ファイル名・配置

システム全体または複数機能に共通する要件は、`CFR-000_システム共通要件.md` を親ファイルとして統括する。共通機能要件(CFR-XXX)の詳細は、FR と同様に個別文書 `CFR-XXX_名称.md` に記載する。

```text
CFR-000_システム共通要件.md   … 親/一覧（システム全体の要件・CFR一覧・RULE-COM・共通制限事項）
CFR-001_認証.md               … 共通機能要件の個別文書（詳細）
CFR-002_認可・権限制御.md
...
```

配置は `02_要件定義/01_機能要件/` 直下とする（§0-3）。
`CFR-000` は個別要件IDではなく、**システム共通要件を統括するコンテナID** として扱う（§0-4）。

---

## 5.2 CFR-000 に含める内容

`CFR-000` は以下を統括する。共通機能要件(CFR)は一覧のみを持ち、詳細は個別文書 `CFR-XXX_名称.md` に記載する。

| 種別               | 内容                         | 記載形式                 |
| ---------------- | -------------------------- | -------------------- |
| システム全体の要件        | 全機能に共通する前提・方針              | 本文（表）                |
| 共通機能要件(CFR)一覧    | `CFR-XXX` の一覧（詳細は各個別文書）    | 一覧（表・詳細は `CFR-XXX_名称.md`） |
| 共通業務ルール(RULE-COM) | 権限外操作禁止、権限外データ参照禁止など       | 本文（表）                |
| 共通制限事項           | 共通の対象外、制限、前提、記載ルール         | 本文（表）                |

---

## 5.3 CFR-000 に書かない内容

| 書かない内容            | 記載先                          |
| ----------------- | ---------------------------- |
| 個別機能に固有の機能要件      | `FR-XXX_機能名.md`              |
| 個別機能に固有の入力条件・出力内容 | `FR-XXX_機能名.md`              |
| 個別機能に固有の業務ルール     | `FR-XXX` または `RULE-XXX`      |
| 非機能要件(NFR)     | `02_要件定義/02_非機能要件/NFR-XXX_名称.md`       |
| 画面レイアウト           | 画面設計                         |
| API仕様             | API設計                        |
| DBテーブル・カラム        | DB設計                         |
| 処理順序・分岐詳細         | シーケンス設計 / モジュール設計            |
| 実装方式              | 機能設計 / API設計 / DB設計 / インフラ設計 |

---

# 6. 共通機能要件の採番ルール

共通機能要件は、`CFR-001` から採番し、FR と同様に個別文書 `CFR-XXX_名称.md` として作成する（一覧は `CFR-000`）。

例：

| ID      | 要件名        | 内容                          |
| ------- | ---------- | --------------------------- |
| CFR-001 | 認証         | 利用者が本人確認を行い、システムを利用可能な状態にする |
| CFR-002 | 認可・権限制御    | 利用者のロール・権限に応じて利用可能な機能を制御する  |
| CFR-003 | 入力検証       | 入力内容の不足、不正形式、範囲外の値を検知する     |
| CFR-004 | エラー表示      | エラー内容を利用者が理解できる形で通知する       |
| CFR-005 | 一覧検索・ページング | 一覧表示において検索、絞り込み、ページングを可能にする |
| CFR-006 | 通知         | 業務上必要なタイミングで利用者に通知する        |
| CFR-007 | 監査ログ記録     | 重要操作を監査ログとして記録する            |

（上表は採番例。各 CFR の詳細は個別文書 `CFR-XXX_名称.md` に記載し、`CFR-000_システム共通要件.md` はその一覧を持つ。要件定義では確認できた内容のみ記載し、未確認の共通機能要件は採番しない。）

---

# 7. 個別機能要件の採番ルール

個別機能要件は、`FR-001` から単純連番で採番する。

## 7.1 ルール

| No | ルール                               |
| -- | --------------------------------- |
| 1  | 個別機能要件は `FR-001` から採番する           |
| 2  | 機能カテゴリや画面IDをFR IDに含めない            |
| 3  | システム共通要件を `FR-000` として扱わない        |
| 4  | 共通要件は `CFR-000_システム共通要件.md` に記載する |
| 5  | 個別FRには、共通要件と同じ内容を重複記載しない          |
| 6  | 個別FRには、その機能に固有の要件だけを記載する          |

---

# 8. 非機能要件の採番ルール

非機能要件(NFR)は、`02_要件定義/02_非機能要件/` に個別文書 `NFR-XXX_名称.md` として定義する。機能横断の品質・特定機能固有の品質のいずれも同ディレクトリで管理する。

| 種別    | 記載先                              | ID        |
| ----- | -------------------------------- | --------- |
| 非機能要件 | `02_要件定義/02_非機能要件/NFR-XXX_名称.md` | `NFR-XXX` |

## 8.1 採番

非機能要件は、`NFR-001` から単純連番で採番する（一覧は `02_要件定義/02_非機能要件/index.md`）。採番は一覧の最大値 +1。欠番の再利用は禁止。

## 8.2 対象範囲とトレース

各 NFR は「対象範囲」で適用範囲（全機能 / システム全体 / 特定機能名）を示す。FR と NFR の適用関係は `02_要件定義/04_トレーサビリティ/traceability_fr_nfr.md` で管理し、NFR 文書・FR 文書の双方に品質要求を重複記載しない。

---

# 9. 業務ルールの採番ルール

業務ルールは、共通業務ルールと個別業務ルールに分ける。

| 種別      | ID形式           | 記載先                   |
| ------- | -------------- | --------------------- |
| 共通業務ルール | `RULE-COM-XXX` | `CFR-000_システム共通要件.md` |
| 個別業務ルール | `RULE-XXX`     | 業務ルール定義、または該当FR       |

## 9.1 共通業務ルールの例

| ID           | ルール名       | 内容                          |
| ------------ | ---------- | --------------------------- |
| RULE-COM-001 | 権限外操作禁止    | 利用者は自身に許可されていない機能を利用できない    |
| RULE-COM-002 | 権限外データ参照禁止 | 利用者は自身に許可されていないデータを参照できない   |
| RULE-COM-003 | 削除済みデータ非表示 | 削除済みとして扱うデータは通常の検索・一覧に表示しない |

# 10. ファイル命名ルール

ファイル名は以下の形式に統一する。

```text
<ID>_<名称>.md
```

例：

```text
CFR-000_システム共通要件.md
FR-001_会議室検索.md
FR-002_会議室予約.md
MDL-003_予約情報.md
SCR-002_会議室検索.md
API-002_会議室検索.md
MOD-003_予約サービス.md
JOB-001_予約リマインド通知.md
TBL-001_ユーザーマスタ.md
```

## 10.1 ファイル命名のルール

| No | ルール                                  |
| -- | ------------------------------------ |
| 1  | ファイル名の先頭に必ずIDを付与する                   |
| 2  | IDと名称は `_` で区切る                      |
| 3  | 名称は日本語でよい                            |
| 4  | ID部分は一度確定したら変更しない                    |
| 5  | 名称変更が必要な場合は、IDを維持したまま名称のみ変更する        |
| 6  | ファイル名には対象の機能・画面・API・テーブルなどが分かる名称を付ける |

---

# 11. ID追加・変更・削除ルール

| 操作   | ルール                      |
| ---- | ------------------------ |
| 追加   | 未使用の次番号を採番する             |
| 名称変更 | IDは変更せず、名称のみ変更する         |
| 内容修正 | 要件の意味が同じであればIDは維持する      |
| 要件分割 | 既存IDは代表要件として残し、新規IDを追加する |
| 要件統合 | 統合元IDは廃止扱いにし、再利用しない      |
| 削除   | IDは欠番として扱い、再利用しない        |
| 並び替え | IDは変更しない                 |

---

# 12. 禁止事項

| No | 禁止事項                               |
| -- | ---------------------------------- |
| 1  | `SYS-XXX` を使用すること                  |
| 2  | `FR-000` をシステム共通要件として使用すること        |
| 3  | `CFR-000` を個別要件IDとして扱うこと           |
| 4  | 削除済みIDを別要件に再利用すること                 |
| 5  | 並び順を揃えるためにIDを振り直すこと                |
| 6  | 画面IDやAPI IDを機能要件IDに埋め込むこと          |
| 7  | 共通要件を各FRに重複記載すること                  |
| 8  | 個別機能にしか適用されない要件を `CFR-000` に記載すること |
| 9  | 非機能要件(NFR)を `CFR-000`・`CFR-XXX`・`FR-XXX` 内に定義すること（NFR は `02_要件定義/02_非機能要件/` に定義する） |
| 10 | 要件IDに実装方式や技術要素を含めること               |
| 11 | 上位要件本文に下位設計IDを大量に記載すること            |
| 12 | 画面仕様、API仕様、DB仕様、実装方式を要件定義に記載すること   |

---

# 13. トレーサビリティ管理

要件・ユースケース・設計成果物の対応関係は、各ドキュメント本文ではなく、各一覧（index.md）で一元管理する（詳細は `docs/CLAUDE.md` §5 参照方向ルール・§6 トレーサビリティ）。

※ ERR(エラーコード)は 03_API設計/エラーメッセージ一覧.md で一元管理し(システム全体を通した連番)、各API文書・API-COM はエラーコードで参照する。MSG(画面表示文言)は中央カタログを設けず各SCR文書にインライン定義する。

# 15. 最終方針

本プロジェクトでは、以下の採番方針を採用する。

```text
CFR-000_システム共通要件.md   … システム全体/複数機能に共通する要件をまとめる親ファイル
FR-001_個別機能名.md          … 個別機能要件は FR-001 から単純連番
```

`CFR-000_システム共通要件.md` の中には、以下を定義する。

```text
CFR-XXX       共通機能要件
RULE-COM-XXX  共通業務ルール
```

非機能要件(NFR-XXX)は `02_要件定義/02_非機能要件/` に個別文書として定義する。

個別FRには、共通要件を重複記載せず、その機能に固有の要件・業務ルール・入力条件・出力内容・受入条件のみを記載する。非機能要件(NFR)は 02_要件定義/02_非機能要件/ に定義する。
